### WR-01: "never raises" guarantee is unverified for non-owning / non-dispatched controllers

**File:** `cement/ext/ext_argparse.py:382-388`
**Issue:** The docstring promises the accessor returns `CommandMeta | None`
and never raises. The property is robust only for the call site exercised by
the tests: an exposed command on the **base** controller reading its own meta
during dispatch. Two reachable-but-untested call sites threaten the guarantee:

1. `_post_argument_parsing()` runs on **every** controller during dispatch
   (lines 853-854), not just the dispatched one. If user code reads
   `self._command_meta` there, `self.__class__` may not own `func_name`. The
   code does guard this with `hasattr(self.__class__, func_name)` →
   `return None` (line 385-386) — but that line is marked
   `# pragma: nocover  # defensive: unreachable` and is never executed by any
   test, so the "returns None instead of raising" behavior for a foreign
   controller is asserted nowhere.
2. The shared `CommandMeta` is resolved via `getattr(self.__class__,
   func_name).__cement_meta__`. The `hasattr` guard checks that the attribute
   exists, but not that it carries `__cement_meta__`. If `func_name` resolves
   to a class attribute that is *not* a decorated command (only possible on a
   foreign controller whose attribute name collides), line 387 raises
   `AttributeError`, contradicting the docstring.

**Fix:** Guard on the decorator marker, not just attribute existence, so the
foreign-controller path provably returns `None`:
```python
member = getattr(self.__class__, func_name, None)
meta = getattr(member, '__cement_meta__', None)
return meta
```
This collapses both branches, removes the untestable `pragma: nocover` line,
and makes the never-raises guarantee hold for every reachable caller. Add a
test that accesses `_command_meta` from a controller that does not own the
dispatched command and asserts `None`.

### WR-02: `_command_meta` behavior on embedded/nested controllers is untested

**File:** `tests/ext/test_ext_argparse.py:39-44, 289-303`
**Issue:** The only test that exercises the dispatched-command branch of
`_command_meta` (`test_command_meta_dispatched`) uses `command_meta_cmd` on
the **base** controller (`Base`). For a base command, `self.__class__` is the
base controller and `__dispatch__ == 'base.<func>'`, which is the simplest
possible resolution. The phase's value proposition is that *any* exposed
command can read its own meta — including commands on embedded controllers
(e.g. `Second.cmd2`, `__dispatch__ == 'second.cmd2'`) and nested controllers
(`Third.cmd3`). None of these assert that `_command_meta.label` /
`parser_options` resolve correctly when `self` is a non-base controller.
Line coverage is 100% only because the same lines execute regardless of which
controller dispatches — behavioral coverage of the cross-controller case is
absent.

**Fix:** Add an exposed command on `Second` (embedded) and/or `Third` (nested)
whose body returns `self._command_meta.label` / `parser_options['help']`, and
assert the resolved values, mirroring `test_command_meta_dispatched`. This
locks in the #670 ergonomic contract for the controller topologies it is
actually meant to serve.

### WR-03: `__dispatch__.split('.')[1]` is fragile to controller labels containing a dot

**File:** `cement/ext/ext_argparse.py:384`
**Issue:** `func_name = self.app.pargs.__dispatch__.split('.')[1]` assumes the
dispatch token is exactly `controller_label.func_name` with a single dot.
`__dispatch__` is built as `f"{controller._meta.label}.{func_name}"`
(line 707). A controller `Meta.label` containing a `.` (e.g. `'a.b'`) yields
`'a.b.cmd'`; `split('.')[1]` then returns `'b'`, not the real `func_name`, so
the lookup silently mis-resolves or (with the WR-01 fix applied) returns
`None`. This mirrors a pre-existing assumption already present in `_dispatch`
(line 865), so it is not newly introduced — but the new property advertises a
robust, never-raises read accessor, which raises the bar for handling odd-but-
legal labels.

**Fix:** Split on the first separator only and keep the remainder as the func
name, so multi-dot labels degrade predictably:
```python
_, _, func_name = self.app.pargs.__dispatch__.partition('.')
```
At minimum, document the "labels must not contain `.`" constraint on the
property. (Surfacing the latent `_dispatch` assumption as a separate issue is
appropriate per the project's "don't fix unrelated code, but flag it" rule.)

## Info

### IN-01: Accessing `_command_meta` before argument parsing raises rather than returning `None`

**File:** `cement/ext/ext_argparse.py:382`
**Issue:** The first statement dereferences `self.app.pargs`. During
controller construction / setup, `self.app` is initialized to `None`
(line 336) and `pargs` does not yet exist, so an early read of
`self._command_meta` raises `AttributeError` instead of returning `None`. The
documented usage is strictly post-parse (inside a command body), so this is
not a defect on the intended path — but it is a sharp edge for a property
whose docstring leads with "never raises."

**Fix:** Optionally narrow the docstring to state the accessor is only valid
once the app has parsed arguments, or guard with
`getattr(self, 'app', None)` / `getattr(self.app, 'pargs', None)` before the
`hasattr` check.

### IN-02: `# pragma: nocover  # defensive: unreachable` on a reachable line

**File:** `cement/ext/ext_argparse.py:386`
**Issue:** Line 386 (`return None` when `func_name` is absent on
`self.__class__`) is labeled `defensive: unreachable`, but it is reachable
whenever `_command_meta` is read from a controller that does not own the
dispatched command (see WR-01, e.g. via `_post_argument_parsing`). Marking a
reachable branch as unreachable obscures the missing test and weakens the
never-raises guarantee.

**Fix:** If WR-01 is adopted, this line disappears. Otherwise, replace the
`unreachable` justification with an honest one (e.g. "covered by foreign-
controller access; see test_...") and add the corresponding test rather than
suppressing it from coverage.
