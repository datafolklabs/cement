# Phase 05.2: ext.argparse command self-meta accessor (#670) - Context

**Gathered:** 2026-06-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Give an exposed `@ex`/`@expose` controller command a backward-compatible, DRY
way to read its **own** `CommandMeta` (label, func_name, hide, arguments, and
the `parser_options` dict holding `help`/`aliases`/etc.) from inside the
function body — resolving Tom Freudenberg's request in
[#670](https://github.com/datafolklabs/cement/issues/670).

**The problem:** today a command can only reach its own decorator meta via the
brittle, repeat-yourself dance
`getattr(self, self.app.pargs.__dispatch__.split('.')[1]).__cement_meta__`.
Tom wants `@ex(help=...)` help text (and other decorator data) reusable inside
the command (e.g. for log/usage output) without that boilerplate.

**Why the literal proposal is rejected:** the issue proposes changing dispatch
from `return func()` to `return func(func_name, func.__cement_meta__)`
(`ext_argparse.py` `_dispatch`, ~L845). That is **backward-incompatible** —
every existing `def cmd(self)` command would suddenly receive two unexpected
positional args. This violates the hard 3.0.x constraint: **zero public-API
breakage, including subclass-exposed internals** (see PROJECT.md). The
`_dispatch()` → `func()` call surface is fully *released*, so the change must be
purely **additive**: a new read-only accessor that changes nothing for code
that ignores it.

**Stale-workaround note:** the workaround in the issue
(`__cement_meta__['parser_options']['help']`) no longer works as written —
`CommandMeta` was refactored from a plain dict to a `@dataclass`
(`ext_argparse.py:108-116`). The current manual form is
`...__cement_meta__.parser_options['help']`. This makes the ergonomic accessor
more valuable, and fixes what is now an actively-broken snippet in the wild.

**In scope:** one additive, read-only accessor on `ArgparseController` returning
the currently-dispatched command's `CommandMeta`; docstring/API docs; CHANGELOG
entry; tests holding the 100% coverage / ruff / mypy gates.

**Out of scope:** any change to the `func()` dispatch signature; signature-based
injection of meta (Tom's literal ask — deferred to Cement 4, see Deferred); a
general command-introspection map; convenience shortcuts over `parser_options`.

</domain>

<decisions>
## Implementation Decisions

### Access mechanism
- **D-01:** Expose the command's own meta via a **read-only property on
  `ArgparseController`**, proposed name **`_command_meta`** — parallels the
  existing `self._meta` convention that downstream code already reads. Usage:
  `self._command_meta.parser_options['help']`. Chosen over the
  `app.pargs.__command_meta__` dunder (less discoverable, more verbose) and over
  signature injection (dispatch magic, hard to type-check). The leading
  underscore matches cement's framework-provided-but-user-read attributes
  (`_meta`, `_parser`).
- **D-02:** The property **resolves the running command itself** — it derives
  `func_name` from `self.app.pargs.__dispatch__` and returns the matching
  `CommandMeta`. No new state is threaded through `_dispatch()`; the property is
  a centralized read of data that already exists.
- **D-03:** When accessed **outside a dispatched command** (no `__dispatch__`
  on `pargs`, e.g. the default-func / no-sub-command branch, or during
  `_post_argument_parsing`), the property returns **`None`** rather than raising.

### Exposed shape
- **D-04:** Hand back the **raw `CommandMeta` dataclass instance as-is** — no
  wrapper, no convenience shortcuts. It is already typed and used internally.
  The `help` line lives at `.parser_options['help']`; `parser_options` is the
  freeform `**kwargs` from `@ex(...)`, so no single key is privileged with its
  own field. Smallest new public surface to lock under 3.0.x.

### Introspection scope
- **D-05:** Expose **only the currently-running command's** meta. A broader
  read-only map for looking up *any* command's meta is explicitly **not** added
  — it is a larger permanently-locked 3.0.x surface and exceeds the issue's
  need. Can be added additively later if demand appears (see Deferred).

### Forward-compat path
- **D-06:** Ship the **accessor only** for 3.0.16. Do **not** seed the
  signature-injection mechanism (Tom's original `def cmd(self, func_name,
  func_meta)` form) now — that would lock two mechanisms and start a deprecation
  clock in a maintenance release.
- **D-07:** **Add a code comment/note** near the accessor (and/or `_dispatch`)
  recording that a future major release (Cement 4) may introduce the opt-in
  func-signature injection form as the ergonomic default, so the intent is not
  lost. This is a comment, not shipped behavior. *(User-requested.)*

### Claude's Discretion (flagged for planner — change if wrong)
- **Naming:** `_command_meta` is proposed, not hard-locked. If the planner finds
  a collision on `ArgparseController` (existing attr/method) or a more idiomatic
  name, adjust. Verify no clash with `_meta`, `_parser`, `_controllers`,
  `_command` before finalizing.
- **Cross-controller resolution:** the common path is the user reading
  `self._command_meta` from *within* the running command method (so `self` owns
  the command — resolve via `getattr(self.__class__, func_name)`). Decide and
  document behavior when accessed from a *different* controller instance than
  the one running the command — recommended: resolve the running command's meta
  and return it, else `None`. This is a HOW detail for research/planning.
- **Class-level sharing caveat:** `__cement_meta__` is attached to the class
  function object and its `.controller` field is mutated to `self` during
  `_collect_commands()` (`ext_argparse.py:707-709`). The accessor therefore
  returns a **shared, class-level** object, not a per-invocation copy. Do not
  stash per-call state on it. Tests should not assume per-instance isolation of
  the meta object.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `self.app.pargs.__dispatch__` — already carries `"<contr_label>.<func_name>"`
  for the running command; the accessor reuses it (no new dispatch state).
- `CommandMeta` (`@dataclass`) — already the typed object to return; no new type
  needed.
- `func.__cement_meta__` — already attached to every exposed method by the
  decorator; the accessor just locates the right one.

### Established Patterns
- `self._meta` / `self._parser` — framework-provided, user-read attributes with
  a leading underscore. `_command_meta` follows this exact convention.
- `_dispatch()` already does the `__dispatch__.split('.')` resolution — the
  accessor centralizes that one-liner so users stop hand-rolling it.

### Integration Points
- New property lives on `ArgparseController` in `cement/ext/ext_argparse.py`.
- Reads from `self.app.pargs`; returns `None` when no command is dispatched.

</code_context>

<specifics>
## Specific Ideas

- Target ergonomic call site (from the issue): replace
  `getattr(self, self.app.pargs.__dispatch__.split('.')[1]).__cement_meta__`
  with `self._command_meta`, e.g. `self._command_meta.parser_options['help']`.
- The accessor must make Tom's "reuse the `@ex(help=...)` line inside the
  command" use case a one-liner.

</specifics>

<deferred>
## Deferred Ideas

- **Signature-injection form** (`def cmd(self, func_name, func_meta)` /
  `def cmd(self, command=None)`) — Tom's literal request. Deferred to **Cement 4**
  as a potential ergonomic default; recorded as a code comment now (D-07), not
  shipped. Would be opt-in/BC-managed via deprecation when introduced.
- **General command-introspection map** (`self.commands['my-command']` to read
  *any* command's meta) — additive future enhancement if demand appears; out of
  scope to avoid over-locking the 3.0.x surface.
- **Convenience shortcuts** over `parser_options` (e.g. a `.help` property) —
  rejected for now (privileges one arbitrary key); revisit only with real
  demand.
