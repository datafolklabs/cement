# Backlog Note — Pydantic Settings config handler (#674)

Captured from issue [#674](https://github.com/datafolklabs/cement/issues/674)
(open; author derks). This is a parking-lot note; promote to a real phase via
`/gsd-review-backlog` → `/gsd-discuss-phase 999.1` when the next milestone opens.

## Why deferred (scope)

New, additive feature. The current v1.0 "Clean & Green" milestone is
maintenance-only (PROJECT.md → Requirements → Out of Scope: *"New features
beyond maintenance … defers to later milestones"*), and Phase 6 is the release
cut for 3.0.16. So this lands in the **next** milestone. It is backward-compatible
(purely additive extension), so it fits a future 3.0.x or 3.2.x line.

**Prerequisite met:** the issue says it should follow *"some effort … toward
adding typing across Cement."* That typing effort landed across Phases 05.x
(all `cement generate` templates typed; broader mypy-strict work), so the stated
gate is satisfied.

## Technical shape

- New module `cement/ext/ext_pydantic.py` with a `PydanticConfigHandler`
  subclassing `cement.core.config.ConfigHandler` and a `load(app)` registrar
  (mirror `ext_yaml` / `ext_json` structure).
- Implement the full `ConfigInterface` contract (see `cement/core/config.py`):
  `parse_file` / `_parse_file`, `get`, `set`, `keys`, `get_sections`,
  `get_dict`, `get_section_dict`, `add_section`, `merge`, `has_section`.
- Backed by `pydantic-settings` (Pydantic v2 `BaseSettings`).
- New optional extra in `pyproject.toml` `[project.optional-dependencies]`:
  `pydantic = ["pydantic-settings"]` (follows the existing `yaml = ["pyYaml"]`
  pattern). Core framework stays zero-runtime-dependency.
- Quality gates as standard: mypy strict, 100% coverage, ruff-clean.
- Docs: Sphinx API reference entry + a GitBook / builtoncement.com narrative
  note. CHANGELOG bucket `[ext.pydantic]`.

## Reference: existing config handlers

- `cement/ext/ext_configparser.py` — `ConfigParserConfigHandler` (default).
- `cement/ext/ext_yaml.py`, `cement/ext/ext_json.py` — subclass the configparser
  handler and override `_parse_file`. Decide whether the pydantic handler
  subclasses similarly or implements the interface directly (it is paradigm-
  different — typed models vs. section/key dicts — so a direct implementation is
  likely cleaner).

## Open design questions (resolve at planning/discuss time)

1. **Model mapping.** Cement's config model is section/key (`get(section, key)`).
   Pydantic Settings is a single typed model. How to map: sections → nested
   `BaseModel` fields? A flat model with dotted keys? A dict-backed
   `BaseSettings` that preserves Cement's dynamic section semantics?
2. **Env-var / precedence.** Mirror ConfigParser's `_get_env_var`
   (`<SECTION>_<KEY>` lookup) and Cement's existing precedence, or adopt
   pydantic-settings' native env/`.env`/secrets sourcing? Reconcile the two so
   behavior is predictable and documented.
3. **Dynamic `set()` / `merge()` vs. validation.** Cement expects to add
   sections and set arbitrary keys at runtime (`add_section`, `set`, `merge`);
   Pydantic models are typically fixed-schema and may be frozen. Decide how a
   dynamic, validation-optional handler coexists with Pydantic's strictness
   (e.g. `model_config` with `extra="allow"`, a dict-backed settings source).
4. **Version pin.** Minimum `pydantic` / `pydantic-settings` (v2) versions, and
   Python-matrix compatibility (3.10–3.14).

## Acceptance sketch (for the future phase)

- A `TestApp` can be configured with `config_handler = 'pydantic'`, parse a
  config file, and round-trip `get`/`set`/`merge`/`has_section` with parity to
  the configparser handler for the documented behaviors.
- Env-var override behavior documented and tested.
- Extra installs cleanly; core install unaffected; smoke-test/matrix green.
