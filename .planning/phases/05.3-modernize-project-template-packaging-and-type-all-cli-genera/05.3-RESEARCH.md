# Phase 05.3: Modernize project template packaging and type all CLI-generated templates - Research

**Researched:** 2026-07-11
**Domain:** CLI code-generation templates (packaging + typing + idiom modernization) — internal, precedent-driven
**Confidence:** HIGH

## Summary

This is a well-scoped, precedent-driven phase. Every fact needed to plan it lives in the
repository — no external/web research was required. Phase 01.1 already migrated the
`project` template's build to `pdm-backend` and established the canonical shape; this phase
extends that shape to `todo-tutorial`, adds a "green out of the box" gate config (ruff +
mypy + pytest) to the `project` template, fully types all generated Python across the 5
templates, converts `%`-formatting to f-strings, and hardens the CI `cli-smoke-test` to
enforce the green bar as a regression guard.

The single highest-leverage design decision is **mypy scope in the generated `project`**
(D-04/D-05): cement's own mypy config *excludes* `tests/`, but the generated test files
(`tests/test_{{ label }}.py`, `tests/conftest.py`) are currently untyped. The planner must
choose between (a) annotating the generated tests so mypy can run over the whole project,
or (b) scoping the shipped `[tool.mypy]` to the package dir only (mirroring cement). This
choice cascades into how much typing work each template needs.

Two latent bugs surfaced that this phase should fix in passing: the `todo-tutorial`
`setup.py` build-time self-import (D-03, the exact PEP 517 isolation bug 01.1 fixed for
`project`), and a genuine `NameError` in `todo-tutorial/todo/main.py:99`
(`except TodoError:` with no `as e` but the body references `e.args[0]`). The
`todo-tutorial/Dockerfile` is also badly stale (references `python3.6`, `setup.py install`,
and a `src/cement` git checkout).

**Primary recommendation:** Mirror the 01.1 pdm-backend shape verbatim for `todo-tutorial`;
ship cement's own gate config as a trimmed-but-strict subset in the generated `project`;
scope generated mypy to the package dir AND annotate the tests (satisfies both D-04 and
D-05 without a config carve-out); add a generated-project `make comply` + `make test` step
to the smoke-test but DO NOT gate `todo-tutorial` on `make test` (its tutorial-stub tests
intentionally `raise Exception`).

## Architecture Patterns

### The canonical pdm-backend shape (mirror this for `todo-tutorial`)

From `project/pyproject.toml` [VERIFIED: cement/cli/templates/generate/project/pyproject.toml]:

```toml
[build-system]
requires = ["pdm-backend"]
build-backend = "pdm.backend"

[project]
name = "{{ label }}"                      # todo-tutorial: hardcoded "todo"
description = "{{ description }}"
authors = [{name = "{{ creator }}", email = "{{ creator_email }}"}]
license = {text = "{{ license }}"}
readme = "README.md"
requires-python = ">=3.10"
dynamic = ["version"]
dependencies = [
    "cement[jinja2,yaml,colorlog]=={{ cement.version }}",  # todo adds: "tinydb"
]

[project.urls]
Homepage = "{{ url }}"

[project.scripts]
{{ label }} = "{{ label }}.main:main"     # todo: todo = "todo.main:main"

[tool.pdm.version]
source = "file"
path = "{{ label }}/core/version.py"      # todo: path = "todo/core/version.py"

[dependency-groups]
dev = ["pytest", "pytest-cov", "coverage", "twine>=1.11.0"]
```

**Why `source = "file"` matters (D-03):** pdm-backend reads the `__version__` literal via
regex without executing/importing the module. This is what closes the PEP 517 build-isolation
bug. The current `todo-tutorial/setup.py` does `from todo.core.version import get_version` at
build time [VERIFIED: todo-tutorial/setup.py:3], which fails under default build isolation —
the exact bug 01.1 diagnosed and fixed for `project`. The migration MUST also change
`todo/core/version.py` to the "Shape B" form (literal `__version__` + trivial wrapper), not
just swap file formats.

**version.py Shape A → Shape B conversion (todo-tutorial, D-03):**
```python
# CURRENT (Shape A — imports cement, tuple source) [VERIFIED: todo-tutorial/todo/core/version.py]
from cement.utils.version import get_version as cement_get_version
VERSION = (0, 0, 1, 'alpha', 0)
def get_version(version=VERSION):
    return cement_get_version(version)

# TARGET (Shape B — matches project template) [VERIFIED: project/{{ label }}/core/version.py]
__version__ = "0.0.1a0"
def get_version() -> str:
    return __version__
```

### `.generate.yml` interaction with file deletion (no impact)

Each template dir carries a `.generate.yml`. The `exclude`/`ignore` keys are **regex filters
that control which files get Jinja substitution**, not a manifest of which files exist.
`todo-tutorial/.generate.yml` is `exclude: ['(.*)']` [VERIFIED: todo-tutorial/.generate.yml] —
i.e. the tutorial is rendered *verbatim* with no variable substitution (it's always "todo").
**Deleting `setup.py`/`setup.cfg`/`requirements*.txt`/`MANIFEST.in` requires no `.generate.yml`
change** — the generator copies whatever files are present. This mirrors 01.1, which deleted
5 files from `project/` without touching its `.generate.yml`.

### Jinja substitution points that MUST be preserved
[VERIFIED: across template files + .generate.yml variable lists]
- `project`: `{{ label }}`, `{{ name }}`, `{{ class_name }}`, `{{ description }}`, `{{ creator }}`, `{{ creator_email }}`, `{{ url }}`, `{{ license }}`, `{{ cement.version }}`
- `script`: `{{ label }}` only
- `extension`: `{{ label }}` only
- `plugin`: `{{ label }}`, `{{ class_name }}`
- `todo-tutorial`: **none** (verbatim, all literal "todo")

### Anti-Patterns to Avoid
- **Adding `[tool.pdm.build]` to generated pyproject.toml** — 01.1 explicitly did NOT add it, because the default pdm-backend packaging auto-includes `templates/*.jinja2` data. A stray `[tool.pdm.build]` block silently strips template files. [VERIFIED: 01.1-01-SUMMARY.md key-decisions]
- **Running `make test` on `todo-tutorial` in CI** — its `tests/test_main.py::test_todo` does `raise Exception` as a tutorial placeholder [VERIFIED: todo-tutorial/tests/test_main.py:8]. Gating the smoke-test on todo's `make test` would always fail.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Version single-sourcing at build | Custom setup.py import | `[tool.pdm.version] source="file"` | Regex read, no build-time import — the D-03 fix |
| Generated dev deps | `requirements-dev.txt` | PEP 735 `[dependency-groups].dev` | 01.1 precedent; single source of truth |
| Gate config authoring | Novel ruff/mypy rules | Trimmed subset of cement's own `pyproject.toml` sections | D-04 says "same gate suite cement enforces" |

## Common Pitfalls

### Pitfall 1: mypy scope determines how much test-typing is required
**What goes wrong:** cement's own `[tool.mypy]` sets `disallow_untyped_defs = true` and
`exclude` includes `^tests` [VERIFIED: pyproject.toml:186,205-209]. If the generated
`project` ships mypy configured to run over `tests/`, every generated test function
(`def test_x():`) and the `tmp` fixture become mypy errors → `make comply` red out of the box.
**How to avoid:** Decide up front. Two consistent options:
- **Option A (recommended):** annotate generated tests (`def test_myapp() -> None:`,
  `def tmp(request: pytest.FixtureRequest) -> Generator[fs.Tmp, None, None]:`) AND let mypy
  see them — most faithful to D-05.
- **Option B (mirror cement):** ship `[tool.mypy]` with `files = ["{{ label }}/"]` /
  `exclude ^tests`, and only type the package. Less literal to D-05 but matches cement exactly.
**Warning sign:** `make comply` fails on `tests/` after generation.

### Pitfall 2: ruff runs over everything regardless of mypy scope
**What goes wrong:** `ruff check .` in the generated `make comply` lints `tests/` too.
Generated tests use `data,output = app.last_rendered` (missing space → E231) and
`from pytest import raises` (PT013 if that rule is selected) [VERIFIED: project/tests/test_{{ label }}.py:25]. Selecting cement's full rule
family (which includes `PT`, `N`, `T20`) will flag generated test/idiom patterns.
**How to avoid:** Ship a **trimmed** rule set. cement itself carries per-file-ignores and
framework-specific ignores (`A001/A002/C901`, `T20` print in tests) that DON'T apply to a
fresh app. The generated `print(...)` statements in `script`/`plugin` command1 bodies would
trip `T20` (flake8-print) — either omit `T20` from the generated config or keep the prints as
intended tutorial output. Pick a subset that is green against the *generated* code as-shipped.

### Pitfall 3: todo-tutorial green-gate would fail on stub tests
**What goes wrong:** Adding `todo-tutorial` to the smoke-test's `make test` gate fails because
`test_todo` deliberately `raise Exception` [VERIFIED].
**How to avoid:** If adding `todo-tutorial` coverage (D-07 planner call), gate only on
**build + install** (`pdm build` / `pip install .`) — proving the D-02/D-03 packaging fix —
not on `make test`. Alternatively, fix the tutorial stub tests to be real, but that desyncs
from the GitBook walkthrough (a deferred concern).

### Pitfall 4: generated Makefile has no `comply` target today
**What goes wrong:** D-04/D-07 say run `make comply`, but the generated `project/Makefile`
only defines `clean/setup/test/docker/dist/dist-upload` [VERIFIED: project/Makefile] — `make comply`
would error "No rule to make target". The `todo-tutorial/Makefile` similarly has no comply.
**How to avoid:** Add `comply`, `comply-ruff` (`pdm run ruff check .`), `comply-mypy`
(`pdm run mypy`) targets to the generated `project/Makefile`, mirroring cement's Makefile
[VERIFIED: cement Makefile:48-57].
