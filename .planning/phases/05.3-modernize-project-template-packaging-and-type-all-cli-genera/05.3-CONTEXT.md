# Phase 05.3: Modernize project template packaging and type all CLI-generated templates - Context

**Gathered:** 2026-07-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Modernize the packaging/setup of the `cement generate` templates and make **all**
generated code fully typed and idiomatically cleaned up. Continues the work started
in Phase 01.1 (which migrated only the `project` template's build to pdm-backend) and
expands it to every CLI-generated template.

These are **generated outputs**, not cement's public API — so the strict
no-public-API-breakage rule of the 3.0.x track does **not** bind template content the
same way. The real risks here are (a) surprising users who regenerate/diff, and (b)
desyncing the `todo-tutorial` template from the external GitBook tutorial narrative.

**In scope:** all 5 templates under `cement/cli/templates/generate/` — `project`,
`script`, `extension`, `plugin`, `todo-tutorial`.

**Out of scope:** cement's own framework/CLI source (already handled in Phases 1–3);
the Cement-4 rewrite; any public-API change to `ext.generate` itself.
</domain>

<decisions>
## Implementation Decisions

### Template scope
- **D-01:** All 5 CLI-generated templates are in scope: `project`, `script`,
  `extension`, `plugin`, and `todo-tutorial`. None excluded.

### Packaging modernization
- **D-02:** Migrate `todo-tutorial` from setup.py-era packaging
  (`setup.py` + `setup.cfg` + `requirements.txt` + `requirements-dev.txt` +
  `MANIFEST.in`) to the same `pdm-backend` + `[project]`-table `pyproject.toml`
  shape already used by the `project` template (per Phase 01.1). Delete the legacy
  packaging files as part of the migration.
- **D-03:** The `todo-tutorial/setup.py` currently does
  `from todo.core.version import get_version` at build time — the same PEP 517
  build-isolation failure Phase 01.1 fixed for `project`. The pdm-backend migration
  MUST also close this latent bug (version sourced via `[tool.pdm.version]`,
  `source = "file"`), not just swap file formats.

### Generated-code quality bar
- **D-04:** "Green out of the box." A freshly generated `project` must pass the same
  gate suite cement itself enforces: ship `[tool.ruff]`, `[tool.mypy]`, and
  `[tool.pytest]` config in the generated `pyproject.toml` so `make comply`
  (ruff + mypy) and `make test` are green immediately after generation.
- **D-05:** All generated Python across every template is fully type-annotated —
  `def main() -> None:`, controller actions, `get_version() -> str`, hook signatures
  (`def load(app: App) -> None:`), etc. — clean under cement's strict mypy config.

### Idiom modernization
- **D-06:** Modernize generated-code idioms to match cement's own post-Phase-3 style:
  replace `%`-string formatting (e.g. `'AssertionError > %s' % e.args[0]`,
  `VERSION_BANNER = """...""" % (...)`) with f-strings, tidy imports, modern stdlib
  idioms. Generated code is a teaching surface, so readable/modern idioms matter.

### Enforcement / verification
- **D-07:** Extend the existing CI `cli-smoke-test` (scripts/cli-smoke-test.sh,
  Python 3.10–3.14 matrix) so that after generating a project it runs the generated
  project's own `make comply` + `make test` and fails the build if they are not green.
  This is the regression guard for D-04/D-05 — the "green out of the box" bar is
  enforced in CI, not just verified once by hand. Note: the smoke-test currently
  generates `project` and `script` but does NOT install/build `todo-tutorial`;
  the planner should decide whether to add coverage for the newly-migrated
  `todo-tutorial` package build.

### Claude's Discretion
- Exact division into plans/waves, per-template ordering, and how the shipped
  `[tool.ruff]`/`[tool.mypy]`/`[tool.pytest]` config is templated (inline vs. minimal
  vs. mirroring cement's own config) are left to research + planning — subject to the
  D-04 "green out of the box" acceptance test.
- Whether `script`'s `VERSION = (0, 0, 1, 'alpha', 0)` tuple convention is unified
  with `project`'s `__version__` file-source convention, or kept as-is, is a planner
  call (script is intentionally single-file/lean).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- Phase 01.1's `project/pyproject.toml` is the canonical pdm-backend template shape —
  reuse its structure (`[build-system]`, `[project]` with `dynamic=["version"]`,
  `[tool.pdm.version] source="file"`, `[project.scripts]`, `[dependency-groups]`) for
  the `todo-tutorial` migration.
- cement's own `pyproject.toml` `[tool.ruff]`/`[tool.mypy]`/`[tool.pytest.ini_options]`
  sections are the reference for what "strict but generated-project-appropriate" gate
  config looks like (D-04).

### Established Patterns
- Templates are Jinja2-rendered with vars like `{{ label }}`, `{{ class_name }}`,
  `{{ description }}`, `{{ cement.version }}` (see `project/pyproject.toml`). New/typed
  code must keep these substitution points intact.
- Each template dir carries a `.generate.yml` config driving prompts/defaults — changes
  to file sets (e.g. deleting todo-tutorial's setup.py) may need `.generate.yml` review.

### Integration Points
- CI: `scripts/cli-smoke-test.sh` generates `project`/`script` in a Docker matrix and
  `pip install .`s them — the natural place to add the comply+test gate (D-07).

</code_context>

  `cement generate project -D <dest>` with no manual fixups — this is the concrete

  explaining the packaging change (folded from the tutorial-docs decision).
