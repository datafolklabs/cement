<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cement dev → generated todo project build | The generated todo project is built/installed (pip install .) and dockerized by the user |
| Docker image build → base image + install steps | The old Dockerfile cloned+built cement from source over the network (git) |

No untrusted runtime input is introduced. The packaging migration is a positive-security change:
`source="file"` removes the build-time import, and the Dockerfile rewrite removes `git` + a
network source checkout.

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05.3-04 | Tampering | todo build-time version import (old setup.py) | medium | mitigate | Replace with `[tool.pdm.version] source="file"` regex read — no code execution at build (D-03) |
| T-05.3-05 | Tampering | todo Dockerfile git clone + build of cement from source | medium | mitigate | Collapse to `pip install --no-cache-dir .` — drops `git`, removes network source checkout (smaller attack surface) |
| T-05.3-SC | Tampering | generated todo deps (cement, tinydb, ruff, mypy) | low | accept | No packages installed INTO cement; all are established (cement toolchain / pre-existing tutorial dep). Legitimacy audit clean (RESEARCH Package Legitimacy Audit — all OK) |
</threat_model>
