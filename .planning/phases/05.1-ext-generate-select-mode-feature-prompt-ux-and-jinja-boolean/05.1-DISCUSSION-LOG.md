## Scope/compat posture

Key finding surfaced during discussion: the entire feature mechanism is
**unreleased** (no tag contains `dc7bd473` or `0a8ffcb0`), so there is no BC
constraint on the feature schema — only the released `variables:` schema is
locked.

**User's choice:** Full redesign for consistency (features aligned with
variables), reframed via clarification.
**Notes:** User: "Tom is a user — we are not taking his request as concrete
absolute requirements, but we want to offer a solid solution for his needs."
User: features should align with variables; a feature is fundamentally boolean
with a default-but-overridable prompt; "features are just a variable whose input
triggers a nested action."

## Redesign shape (authoring surface)

| Option | Description | Selected |
|--------|-------------|----------|
| A: Full merge (implicit) | One variables list; effect-blocks make a var a feature | |
| B: Two keys, aligned UX | Keep `features:` separate but share prompt engine | |
| C: Merge with explicit type | One variables list, explicit `type: string\|boolean\|choice` | ✓ |

**User's choice:** C — explicit `type:` + merge into variables.
**Notes:** "wins on display ordering as well as symmetry with variables." Then
steered toward replacing the boolean `enabled/disabled` sub-keys and choice
`options` nesting with a unified "on input, trigger/extend" mechanism.

## Trigger vocabulary

| Option | Description | Selected |
|--------|-------------|----------|
| Rule-list (`extend:`) | List of `{when: <match>, +effects}`; multi-value/regex; composes | ✓ |
| Value-keyed map (`when:`) | Map keyed by resolved value → effects; compact switch | |

**User's choice:** Rule-list (`extend:`).
**Notes:** Matches the "trigger" mental model; nested vars prompt depth-first
only when their trigger fires; nesting naturally replaces the flat
`requires:` DAG for the common case.

## Dependencies

| Option | Description | Selected |
|--------|-------------|----------|
| Nesting + requires (both) | Structural nesting + referential `requires:` (same match vocab) | ✓ |
| Nesting only | Drop `requires:`; structural only | |
| Requires only (flat) | Flat list, all top-level w/ `requires:` | |

**User's choice:** Nesting + requires (both).
**Notes:** Asked to see concrete `requires:` syntax first; converged on

AND-ed across keys; single-parent truthy nest ≡ `requires: {parent: true}`.

## Boolean prompt rendering

| Option | Description | Selected |
|--------|-------------|----------|
| Framework auto-renders Y/N | Author writes label; framework appends hint + parses bool | ✓ |
| Author controls full text | Framework appends only `[default]:` | |

**User's choice:** Framework auto-renders, "but supports prompt override when
developer wants it — maybe?"
**Notes:** Insight during discussion: `type: string` is NOT a valid escape
hatch for "custom prompt + real bool" (a string is truthy for any non-empty
answer). So a boolean-specific override is genuinely warranted.

---

## Boolean truth mapping (junk-input behavior)

| Option | Description | Selected |
|--------|-------------|----------|
| token lists + abort-on-no-match | Explicit value lists; non-match aborts like `validate:` | ✓ |
| regex (validate-style) | Regex classifier; non-match silently false | |

**User's choice:** Token lists, with abort-on-no-match folded in.
**Notes:** User asked what `validate:` looks like today (regex via
`re.match`, asserts/aborts on failure — no re-prompt). User: "fold the
consistency in" — object-form boolean input matching neither set aborts with an
`Invalid Response`-style assertion, mirroring `validate:`. Polymorphic
`prompt:` confirmed: string | false | object `{text, accept, reject}`.

## Token-key naming

| Option | Description | Selected |
|--------|-------------|----------|
| true_values / false_values | Ties to stored bool; dodges YAML coercion | |
| yes_values / no_values | Ties to Yes/No prompt language | |
| accept / reject | Verb-framed toggle | ✓ |

**User's choice:** accept / reject.
**Notes:** User rejected `truthy`/`falsy` (Python jargon). Bare
`yes`/`no`/`true`/`false` keys ruled out — YAML 1.1 `full_load` coerces them to
boolean keys.
