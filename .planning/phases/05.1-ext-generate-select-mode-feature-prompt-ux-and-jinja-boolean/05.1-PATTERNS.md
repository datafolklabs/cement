## Pattern Assignments

### Unified resolver — `cement/ext/ext_generate.py` (engine, transform)

**Analog A — lazy recursive memoized `_resolve` (the `requires` DAG to PRESERVE):**

Source `cement/ext/ext_generate.py:118-128`:
```python
feature_states: dict[str, bool | str] = {}

def _resolve(name: str) -> bool | str:
    if name in feature_states:          # memoize → order-independent (D-11)
        return feature_states[name]
    feature = feature_by_name[name]
    for req in feature.get('requires', []):
        if not _resolve(req):           # recurse before resolving self
            self.app.log.warning(
                f"Feature '{name}' disabled (requires: {req})")
            feature_states[name] = False
            return False
    ...
```
**Replicate:** This memoization-keyed recursion is the model for the new `requires` resolution
(D-09/D-10/D-11). Re-key from a separate `feature_states` dict onto the `data` dict (or a
parallel `resolved` set) so "already resolved" == "already in `data`". The `requires` vocab
expands per D-09: bare `[name]`→truthy (current behavior), `{name: value}`→equality,
`{name: [v1,v2]}`→in-list — AND-ed across keys. Reuse the `not _resolve(req)` short-circuit but
generalize the truthiness test to the match vocabulary (see `extend.when` matching below).

**Analog B — top-level `data[name]` emit sink (the D-05 routing target):**

Source `cement/ext/ext_generate.py:325` (the released variable sink — already top-level):
```python
data[var['name']] = val
```
Source `cement/ext/ext_generate.py:199-204` (the namespaced features sink — to DELETE):
```python
if 'features' not in data:
    data['features'] = {}                 # ← REMOVE entirely (root of bug #4)
...
data['features'][name] = state            # ← REMOVE — route every var through L325 instead
```
**Replicate:** Every variable (incl. former features) emits at `data[name]` top-level. Booleans
emit a real Python `bool` (Jinja/pystache truthy); choices emit the chosen `str`; strings emit
`str` (unchanged). NEVER write `data['features']`.

**Analog C — `or []` coalescing idiom (PRESERVE for extend blocks):**

Source `cement/ext/ext_generate.py:227-229`:
```python
vars.extend(block.get('variables') or [])
exclude_list.extend(block.get('exclude') or [])
ignore_list.extend(block.get('ignore') or [])
```
Source `cement/ext/ext_generate.py:252-254` (top-level keys — same idiom):
```python
vars = g_config.get('variables') or []
exclude_list = g_config.get('exclude') or []
ignore_list = g_config.get('ignore') or []
```
**Replicate:** When an `extend` rule fires, accumulate its `variables`/`exclude`/`ignore` with the
exact same `... or []` pattern (handles `key: null` — locked by test12/test23/test24).

**Analog D — `var_defaults` merge (the COMPAT ANCHOR — keep verbatim, extend additively):**

Source `cement/ext/ext_generate.py:266-276`:
```python
var_defaults: dict = {
    'name': None,
    'prompt': None,
    'validate': None,
    'case': None,
    'default': None,
}
for defined_var in vars:
    var = var_defaults.copy()
    var.update(defined_var)
    for key in ['name', 'prompt']:
        assert var[key] is not None, \
            f"Required generate config key missing: {key}"
```
**Replicate:** Extend this dict ADDITIVELY (`'type': 'string'`, `'extend': None`,
`'requires': None`). A `type: string` variable with no new keys must take the identity path and
behave byte-identically. NOTE: per D-04 `type` defaults to `string`. CAUTION on the
`for key in ['name', 'prompt']` assert — a `type: boolean` with `prompt: false` (silent) or a
choice may legitimately lack a freeform `prompt`; the planner must rework which keys are required
per-type without breaking the released string path (test1/test22/test27).

**Control-flow shape to build** (from RESEARCH §Recommended control-flow, lines 175-189) — a single
recursive `resolve_and_emit(var)` walking `variables` in declaration order (D-06), recursing
depth-first into fired `extend.variables` (D-08), resolving `requires` on demand (D-11).

### Boolean prompt — `cement/ext/ext_generate.py` (prompt, request-response)

All three existing prompt forms are `shell.Prompt` subclasses defined inline with a nested `Meta`.
The new polymorphic boolean prompt (D-12) replicates this construction shape.

**Analog — `BoolFeaturePrompt` (the UX being REPLACED; the construction pattern to KEEP):**

Source `cement/ext/ext_generate.py:182-192`:
```python
default_hint = 'Y/n' if bool_default else 'y/N'  # pragma: nocover
default_val = 'y' if bool_default else 'n'  # pragma: nocover

class BoolFeaturePrompt(shell.Prompt):  # pragma: nocover
    class Meta:
        text = f"Enable Feature: {name} [{default_hint}]:"
        default = default_val

bp = BoolFeaturePrompt(auto=False)  # pragma: nocover
bval: str = bp.prompt() or default_val  # pragma: nocover
feature_states[name] = bval.lower() == 'y'  # pragma: nocover
```
**Replicate (D-12 string-form):** Build text as
`f"{var['prompt']} [(Y)es/(N)o] [{'Y' if default else 'N'}]:"` (NOT "Enable Feature:" — that is
exactly Tom's complaint). Then parse the returned token EXTERNALLY: `y`/`yes`→True, `n`/`no`→False,
empty→`default`, anything else → assert-abort (see accept/reject analog). The y/n→bool mapping
(`bval.lower() == 'y'`) is the coverable logic — unit-test it via patched `prompt()` rather than
pragma it (see test pattern below). Keep `auto=False` + explicit `.prompt()` OR construct with
default `auto=True` and read return — both work; current code uses `auto=False`.

**Analog — variable `MyPrompt` (the released text+default format to MIRROR):**

Source `cement/ext/ext_generate.py:304-310`:
```python
class MyPrompt(shell.Prompt):
    class Meta:
        text = f"{var['prompt']}{default_text}:"   # default_text == f" [{var['default']}]"
        default = var.get('default', None)
p = MyPrompt()
val = p.prompt()    # pragma: nocover  # defensive: unreachable
```
**Replicate (D-12 object-form):** For `prompt: {text, accept, reject}`, the author owns the full
`text` — set `Meta.text = var['prompt']['text']` and append NOTHING. `Meta.default = ...` still
seeds empty→default. The trailing-`:` and `[default]` framework decoration applies ONLY to the
string-form and the released string variable, NOT the object-form.

**Analog — `shell.Prompt.Meta` capabilities** (`cement/utils/shell.py:392-435`): `text`,
`default` (empty input → default, else `None`; see `_prompt` L490-493), `options`+`numbered=True`
(numbered picker), `case_insensitive` (default True), `auto` (default True). Built-in retry loop
(`prompt()` L500-534) only re-prompts on `options` membership/numbered failure — a bare prompt
accepts any non-empty input, which is WHY the boolean parse must validate-and-abort externally.

### Accept/reject junk-input abort — `cement/ext/ext_generate.py` (validation, D-14)

**Analog — `validate:` assert-and-abort (the EXACT model for D-14):**

Source `cement/ext/ext_generate.py:321-323`:
```python
if var['validate'] is not None:
    assert re.match(var['validate'], val), \
        f"Invalid Response (must match: '{var['validate']}')"
```
**Replicate:** For the boolean object-form, an answer matching NEITHER `accept:` nor `reject:`
(after lowercasing) is invalid → `assert <matched>, "Invalid Response (must be one of accept/reject)"`.
Same `assert`+abort UX as `validate:` — NO re-prompt, NO silent coercion to False (D-14). Empty
input still falls through to `default` BEFORE this check. This is the runtime-response validation
path → keep `assert` (test_invalid_variable_value at `test_ext_generate.py:80-86` locks the
`AssertionError`/"Invalid Response.*" behavior). CONTRAST: config-schema errors use `raise
ValueError` (L40, L59) and survive `python -O` — do NOT convert those to assert.

### Choice numbered picker — `cement/ext/ext_generate.py` (prompt, request-response)

**Analog — `SelectPrompt` (the numbered-list flow to KEEP, re-keyed to `type: choice`):**

Source `cement/ext/ext_generate.py:165-176`:
```python
class SelectPrompt(shell.Prompt):  # pragma: nocover
    class Meta:
        text = prompt_text
        options = labels
        numbered = True
        case_insensitive = True
        default = default_label

chosen_label = SelectPrompt().prompt()  # pragma: nocover
feature_states[name] = (  # pragma: nocover
    values[labels.index(chosen_label)]
)
```
**Replicate:** `type: choice` reuses this numbered picker verbatim. The built-in retry/whitelist
(`shell.Prompt.prompt` L514-519) handles invalid numbers — let it. D-16: `options:` may be a scalar
list (`[none, flask, fastapi]`) OR per-option objects (`{value, prompt}`); the label/value split
(L142-148) already handles the object form. Keep `str()`-coercion of both `value` and the resolved
state (L142-143, L219) — YAML may decode `value: 1` as int. The interactive `.prompt()` line stays
`# pragma: nocover`; the value-mapping (`values[labels.index(...)]`) is coverable via patched prompt.

### `--defaults` dispatch — `cement/ext/ext_generate.py` (engine, D-18)

**Analog — the three existing `pargs.defaults` branches (consolidate per-type):**

Source `cement/ext/ext_generate.py:138-140` (select default):
```python

    feature_states[name] = default
```
Source `cement/ext/ext_generate.py:179-180` (bool default):
```python
if self.app.pargs.defaults:
    feature_states[name] = bool_default
```
Source `cement/ext/ext_generate.py:294-295` (string variable default):
```python
elif var['default'] is not None and self.app.pargs.defaults:
    val = var['default']
```
**Replicate:** The unified resolver branches on `type` in the `--defaults` path: bool→Python bool
default, choice→option value default, string→string default. EVERY feature fixture is tested
`--defaults` (the only non-interactive path) — a missing type branch hangs on stdin
(`OSError: reading from stdin while output is captured`, see `test_prompt` at
`test_ext_generate.py:58-64`). Preserve the released silent-variable short-circuit (`prompt: false`)
VERBATIM:

Source `cement/ext/ext_generate.py:282-293` (silent sentinel — KEEP, generalize to bool):
```python
if var['prompt'] is False:
    assert var['default'] is not None, \
        f"Variable '{var['name']}' has prompt: false but no default"
    val = str(var['default'])
```
The `str(default)` coercion is locked by test29 (non-str default); the `assert default is not None`
is locked by test27. For `type: boolean` with `prompt: false` (D-12 false-form), the value must
land as a real `bool` at `data[name]`, NOT `str(default)` — so the bool path emits `bool(default)`
not `str(...)`. Decide this split carefully: released string silent keeps `str()`; bool silent emits bool.

## Shared Patterns

### Interactive-branch coverage (apply to ALL new prompt code)
**Source:** `cement/ext/ext_generate.py:301, 310` (variable prompt) and L141-192 (feature prompts)
**Apply to:** every new interactive `shell.Prompt().prompt()` line.
```python
val = p.prompt()    # pragma: nocover  # defensive: unreachable
```
Locked pragma vocabulary (RESEARCH §Project Constraints): `# pragma: nocover  # defensive: unreachable`
for interactive Prompt branches; `# pragma: nocover  # untestable: dynamic import` for import
failures. Do NOT invent new labels. `# noqa: E501` may sit AFTER a pragma label when the combined
comment exceeds 100 chars (established at `ext_generate.py:162`, also `shell.py:454-467`).
**Prefer a real patched-prompt test over a pragma** where the logic is coverable (the value-mapping,
not the input read).

### Patched-prompt unit test (cover the coverable mapping logic)
**Source:** `tests/ext/test_ext_generate.py:89-97` (`test_no_default`)
**Apply to:** y/n→bool mapping, accept/reject mapping, D-14 junk-abort, choice value-mapping.
```python
def test_no_default(tmp):
    with patch.object(shell.Prompt, 'prompt', return_value='Bogus'):
        argv = ['generate', 'test5', tmp.dir]
        with GenerateApp(argv=argv) as app:
            app.run()
            with open(os.path.join(tmp.dir, 'take-me')) as f:
                assert 'Bogus' in f.read()
```
**Replicate:** Patch `shell.Prompt.prompt` to return `"yes"` / `"n"` / junk and assert the resolved
`data[name]` (rendered output) is `True` / `False` / aborts. This puts the bug-prone mapping under
real coverage instead of a pragma. Imports already present at `test_ext_generate.py:1-7`
(`from unittest.mock import patch`, `from cement.utils import shell`,
`from cement.utils.test import raises`).

### Config-validation fail-fast (apply to new schema checks)
**Source:** `cement/ext/ext_generate.py:39-48` (name + requires-unknown validation), L59 (whitelist)
**Apply to:** `type:` whitelist, choice empty-options / default-not-in-options / duplicate-labels,
`requires` unknown-variable.
```python
if feature.get('name') is None:
    raise ValueError("Required feature config key missing: name")
...
if mode not in ('boolean', 'select'):
    raise ValueError(f"... invalid prompt_mode '{mode}' ...")
```
**Replicate:** `raise ValueError` (NOT assert) for schema misconfig — survives `python -O`, matches
the deliberate split documented at L35-36. New checks: `type: bogus` → ValueError (replaces the
removed `prompt_mode: bogus` check, test18); choice empty-options/missing-default/duplicate-labels
→ ValueError (preserve test26/test28/test30 spirit, Q3). The `requires unknown` message (L46-48)
adapts from "feature" → "variable" (test10).

### Fixture migration template (apply to ~25 testNN fixtures + demo)
**Source — released oracle (KEEP UNCHANGED):** `tests/data/templates/generate/test1/.generate.yml`
— plain `variables:` with `name/prompt/default/case/validate`. This is the byte-identical compat
oracle (test1-5, test22, test27, test29). Do not touch.

**Source — bool feature → migrate:** `tests/data/templates/generate/test6/.generate.yml`. The
`features: [{name, default, enabled:{variables,ignore}, disabled:{exclude,ignore}}]` shape becomes:
```yaml
variables:
  - name: feature1
    type: boolean
    default: true
    extend:
      - when: true
        variables:
          - name: feature1_var
            prompt: Feature1 Variable
            default: feature1_val
        ignore: ['.*no-feature1.*']
      - when: false
        exclude: ['.*feature1-file.*']
        ignore: ['.*feature1-only.*']
```
(`enabled:`→`extend: [{when: true, ...}]`; `disabled:`→`extend: [{when: false, ...}]`.)

**Source — select feature → migrate:** `tests/data/templates/generate/test16/.generate.yml` and
`demo/.../webapp/.generate.yml`. The `prompt_mode: select` + `options:[{value,prompt,ignore,variables}]`
shape becomes `type: choice` + `options:` (scalar list or `{value,prompt}` per D-16) + `extend:`
keyed by `when: <value>` carrying the `ignore`/`variables`. The demo's `web_framework` (lines 32-57)
and `docker`/`docker_compose` (lines 10-30, the latter with `requires: [docker]`) are the richest
migration exemplars.

**YAML 1.1 landmine (RESEARCH Pitfall 1):** in `accept:`/`reject:` token lists, QUOTE bool-like
tokens — `accept: [y, "yes"]` not `accept: [y, yes]` (bare `yes`→Python `True` under `yaml.full_load`
at L245). `when: true`/`default: true` are intentional Python bools (fine). Planner: consider a load
guard that `raise ValueError` if an accept/reject member loaded as a Python `bool`.

### ValueError-asserting test pattern
