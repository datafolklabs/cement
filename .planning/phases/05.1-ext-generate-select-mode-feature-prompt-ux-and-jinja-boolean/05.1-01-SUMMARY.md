**Collapsed the unreleased `features:` engine and the released `variables:` loop into one ordered recursive resolver; `type: boolean` now emits a real Python bool at top-level `data[name]` so `{% if feature_x %}` works (#782 bug-4 fix), with a vars-style boolean prompt and a legacy `features:` bridge keeping the un-migrated fixtures green.**

## Accomplishments
- Single recursive `resolve_and_emit` walks `variables:` in declaration order; every variable (string/boolean) emits at top-level `data[name]` — the `data['features']` namespace is gone (root of #782 bug #4).
- `type: boolean` resolves to a real Python `bool`, so `{% if feature_x %}` renders the enabled branch in jinja2/mustache.
- Vars-style boolean prompt `'<prompt> [(Y)es/(N)o] [default]:'` replaces the old "Enable Feature:" text (Tom's #782 pts 2/3); `y`/`yes`->True, `n`/`no`->False, empty->default — mapping covered by a real patched-prompt test, not a pragma.
- `type: string` (the released schema) stays on the identity path — test1-5/22/27/29 pass byte-identically.
- D-17 enforced: `type: boolean`/`choice` declaring `case:`/`validate:` raises `ValueError`; unknown `type:` raises `ValueError` (T-05.1-01 input-validation mitigation).
- `extend:` boolean `when: true/false` rules compose (accumulate variables/exclude/ignore, recurse nested vars depth-first in place) — enough for test6/test7.
- Legacy `features:` bridge keeps the 22 un-migrated fixtures (test8-30) green at 100% coverage; it is removed in Plan 03.

### Architectural (user-approved)

**1. [Rule 4 - Architectural, user-approved] Legacy `features:` compatibility bridge**
- **Found during:** Pre-execution structural sequencing analysis (resolved by the user before this session).
- **Issue:** Plan 01's must_have requires `features:`/`prompt_mode`/`enabled`/`disabled` to no longer exist in `ext_generate.py`. But ~22 fixtures (test8-30) exercise `requires`/`select`/`enabled`/`disabled`, whose new-schema equivalents (`type: choice`, `extend:`, top-level `requires:`) are not implemented until Plans 02/03 — so those fixtures cannot be migrated in Plan 01. Deleting the legacy reader outright would break them and the 100%-coverage-every-commit gate.
- **Fix:** Built the full new typed resolver exactly as specified (top-level typed emit, single declaration-order pass, `type: boolean`, boolean prompt UX, D-17 guard, `data['features']` namespace removed). KEPT a thin internal legacy `features:` bridge (`_process_features`) that folds each legacy feature's resolved enabled/disabled/options effects into the SAME unified vars/exclude/ignore accumulation — without writing any `data['features']` namespace. The bridge is documented in-code for removal in Plan 03.
- **Consequence:** The must_have "`features:` no longer exists in ext_generate.py" is RELAXED to become true at PHASE END (after Plan 03), NOT at the Plan 01 commit. The bug-4 fix (top-level typed emit, namespace removed, single declaration-order pass, boolean prompt UX) IS delivered now. `data['features']` namespace is verified gone; the word `features` still appears only in the bridge reader and its docstring.
- **Files modified:** cement/ext/ext_generate.py
