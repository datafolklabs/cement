# Phase 05.1 Plan 03: extend.when list/regex + nesting + requires; bridge removal Summary

**Completed the conditional-effects and dependency surface of the unified resolver — `extend.when` now composes scalar/in-list/string-regex matching with depth-first nested `extend.variables` and a full-vocabulary order-independent top-level `requires:` — and DELETED the legacy `features:` bridge so GEN-01 is finally, fully true.**

## Accomplishments
- `_when_matches(value, when, vtype)` composes all three D-07 forms: list-membership (`when: [a, b]`, str-coerced for choice/string, Python `==` for boolean), string-type `re.match` regex (string vars only), and type-dispatched scalar equality (boolean `==`, choice/string str-coerce). Shared by BOTH `extend.when` rule firing and `requires` gating.
- Multiple matching `extend` rules compose — each accumulates its `exclude`/`ignore` (`... or []` coalesced) and recurses depth-first into its nested `variables` in place, only when its rule fires (D-08).
- Top-level `requires:` (D-09/D-11): `[name]` truthy sugar, `{name: value}` equality, `{name: [v1,v2]}` in-list — AND-ed across keys. Each prerequisite resolves on demand via the lazy memoize-on-`data` recursion (generalized from the deleted feature DAG), so a prerequisite declared LATER still resolves first (order-independent, test15).
- Q1: a `requires`-gated-out variable is set to its typed `default` at `data[name]` (`bool(default)` / `str(default)`) so `{{ var }}`/`{% if var %}` never KeyError, and its `extend` rules do NOT fire (test40, test25, test11).
- `requires` against an unknown variable -> `ValueError` (T-05.1-07 mitigation, test10); a top-level variable missing `name` -> `ValueError` (test8, was an assert).
- DELETED `_process_features` and the `features:` read in `_generate` (~200 lines). GEN-01 fully true: grep confirms no `features:`/`prompt_mode`/`enabled`/`disabled` in the engine or any fixture.
- Migrated test8-12/14/15/23/24/25 to `type:`/`extend:`/`requires:`; removed obsolete test17/test18; added test38 (list + boolean list-form), test39 (string regex), test40 (Q1), test41 (dict-form requires equality).

## Decisions Made
- **Removed (not repurposed) test17/test18** — they asserted on `prompt_mode`/`enabled`/`disabled` schema that no longer exists; the `type:` whitelist test the plan offered as a repurpose target is already covered by test34 (`type: bogus` -> ValueError). The plan explicitly allowed remove OR repurpose.
- **Transitive cascade via Q1** — the legacy cascade relied on a requires-failed feature merging its `disabled` block; the new schema has no disabled block, so test14's cascade is expressed by giving each downstream var a falsy default and asserting on the rendered resolved values. A gated-out var's falsy default propagates the gate downstream.
- **`requires` is top-level-only** — `var_by_name` is built from the top-level `vars`; nested `extend.variables` are gated structurally by their parent rule (D-10 equivalence).
