## Gap-Closure (post-verification, 2026-05-29)

Live interactive verification surfaced one defect the automated suite missed:
the `type: boolean` string-form interactive prompt rendered the literal
`None [(Y)es/(N)o] [<hint>]:` when a boolean variable declared **no** `prompt:`
key (`var['prompt']` is `None`, interpolated straight into the text). The tests
missed it because they always supplied a `prompt:` or ran `--defaults` (which
returns before the prompt text is built). The demo's prompt-less `docker` /
`docker_compose` booleans exhibited it.

**Fix:** default the label to `f"Enable {var['name']}"` when `var['prompt']` is
falsy (CONTEXT D-12, mirroring the choice path's `or f"Select: {name}"`
fallback). `docker` (no prompt) now renders `Enable docker [(Y)es/(N)o] [Y]:`;
an explicit `prompt:` still wins; the object-form (`prompt: {text,...}`) is
untouched (author owns that text).

**Coverage:** new fallback branch driven under real coverage by test42 (boolean,
no `prompt:`) via a recorder patched onto `shell.Prompt.prompt` that captures the
bound `Meta.text` and asserts the `Enable enable_thing [(Y)es/(N)o]` label, plus
an explicit-override case. `make test` GREEN at 100% (370 passed,
`ext_generate.py` 270/0/100%); ruff + mypy clean.
