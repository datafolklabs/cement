# Phase 05.1: ext.generate select-mode feature prompt UX and Jinja boolean fixes - Context

**Gathered:** 2026-05-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign `ext.generate`'s optional-feature mechanism so it is unified with, and
behaves consistently with, the existing template `variables:` mechanism — driven
by Tom Freudenberg's feedback on closed PR #780 ([#782](https://github.com/datafolklabs/cement/issues/782)).

**Why this is free to redesign:** the entire feature mechanism is **unreleased**.
No git tag contains either the original boolean-features commit (`dc7bd473`) or
the `prompt_mode: select` commit (`0a8ffcb0`). Both first ship in 3.0.16. There
is therefore **no backward-compat constraint on the feature schema** — we are
reshaping it before it ever reaches users. The *only* compat-locked surface is
the pre-existing `variables:` schema (`name`/`prompt`/`validate`/`case`/`default`
+ `prompt: false`), which shipped in earlier 3.0.x. **All changes to `variables:`
in this phase MUST be purely additive** to that released surface — a variable
with no new keys must behave byte-identically to today.

**In scope:** merge the `features:` concept into `variables:` (new `type:`,
`extend:`, `requires:` keys + a polymorphic boolean `prompt:`); make features
prompt in declaration order; expose feature values at the top level of the
template context; migrate all in-repo templates and test fixtures to the new
schema; hold the 100% coverage / ruff / mypy gates.

**Out of scope:** any change to released `variables:` behavior; Cement-4
architectural seams; GitBook docs (tracked separately — see deferred). Tom's
report is treated as *needs to solve*, not literal requirements — the solution
is our design, not a transcription of his message.

</domain>

<decisions>
## Implementation Decisions

### Core model — merge features into variables
- **D-01:** Remove the `features:` top-level key entirely, along with
  `prompt_mode`, the boolean `enabled:`/`disabled:` blocks, and the select
  `options:`-with-effects schema. There is **no** deprecated alias — nothing
  shipped, so no alias debt. Everything an author used `features:` for is
  expressed under `variables:`.
- **D-02:** A "feature" is reframed as **a variable whose input triggers a
  nested action** (extends the generation). Conceptually: a feature = a
  `boolean` (or `choice`) variable that also carries conditional effect-blocks.
- **D-03:** This is a full redesign-for-consistency (not additive-only) of the
  *unreleased* feature surface — but additive-only against the *released*
  `variables:` surface (see Phase Boundary).

### Variable types
- **D-04:** Add an explicit `type:` key on variables: `string | boolean |
  choice`. Default is `string` (so existing typeless variables are unchanged).
- **D-05:** Template-context exposure is at the **top level** (`data[name]`),
  same as ordinary variables — NOT namespaced under `data['features']`. This is
  the direct fix for Tom's bug #4: `{% if feature_nicegui %}` now works.
  - `boolean` → stored as a real Python `bool` (Jinja-truthy)
  - `choice` → stored as the chosen value `str`
  - `string` → stored as `str` (unchanged)
- **D-06:** All variables (including the former features) are prompted in a
  single pass in **declaration order** — this is the fix for Tom's ordering
  point #1 (features no longer jump ahead of vars; author controls order).

### Conditional effects — `extend:`
- **D-07:** A variable may carry an `extend:` list of trigger rules. Each rule:
  `{ when: <match>, variables: [...], exclude: [...], ignore: [...] }`.
  - `when` matches the resolved value: a single value, a list of values
    (in-list), or — for `string` types — a regex.
  - Rules **compose**: multiple rules whose `when` matches all fire, and their
    `variables`/`exclude`/`ignore` accumulate.
- **D-08:** `extend.variables` are **nested**: prompted depth-first, in place,
  **only when their parent rule fires**. A nested variable is itself a full
  variable (may have its own `type`/`extend`/`requires`).

### Dependencies — nesting + requires (both)
- **D-09:** Support BOTH dependency surfaces:
  - **Nesting** (structural) — a dependent declared inside a parent's
    `extend.when.variables`. Sugar for the common single-parent gate.
  - **`requires:`** (referential, top-level) — a variable names other variables
    it depends on. Uses the **same match vocabulary as `extend.when`**, AND-ed
    across keys:
    - `requires: [name]` → sugar for "must be truthy"
    - `requires: {name: value}` → equality
    - `requires: {name: [v1, v2]}` → in-list
- **D-10:** Nesting and `requires` compile to the same gate: a single-parent
  truthy nest is equivalent to `requires: {parent: true}`. `requires` is the
  general referential form (handles many-to-one without duplicating a variable
  under two parents).
- **D-11:** Resolution is order-independent: a `requires`d variable is resolved
  (lazily, recursively) before its dependent, regardless of declaration order —
  preserve the existing lazy-`_resolve` recursion semantics.

### Boolean prompt — polymorphic `prompt:`
- **D-12:** For `type: boolean`, `prompt:` is polymorphic — `string | false |
  object`:
  - **string** → framework owns the rendering: appends `[(Y)es/(N)o] [default]:`,
    default token set `y`/`yes` ↔ `true`, `n`/`no` ↔ `false`, case-insensitive,
    empty input → `default`. (This is the 90% case; eliminates Tom's
    hand-wired `validate`/`case` boilerplate — points #2/#3.)
  - **false** → silent (no prompt), use `default` directly (the existing
    `prompt: false` sentinel, generalized to booleans).
  - **object** `{ text:, accept:, reject: }` → author owns the full prompt
    `text` (framework appends nothing); `accept:`/`reject:` are
    case-insensitive token **lists** mapping input → bool.
- **D-13:** Token-key naming is **`accept:` / `reject:`** (NOT `truthy`/`falsy`
  — Python jargon; and NOT bare `yes`/`no`/`true`/`false` — those coerce to
  booleans under PyYAML/YAML 1.1 `full_load`, a real landmine).
- **D-14:** Junk-input handling mirrors `validate:`: in the object form, an
  answer matching **neither** `accept:` nor `reject:` is an invalid response —
  it `assert`s and **aborts** with an `Invalid Response`-style message (same
  failure behavior as a failed `validate:` regex today). No silent coercion of
  unknown input to `false`. Empty input still falls through to `default`.
- **D-15:** `default:` for a boolean stays a real bool (`true`/`false`); it
  seeds empty input. `type: string` is explicitly NOT a valid "custom boolean"
  escape hatch (a string stores the token, so `{% if x %}` would be truthy for
  any non-empty answer incl. `"reject"`) — which is exactly why the boolean
  object override exists.

### Locked defaults (Claude's discretion — flagged for planner, change if wrong)
- **D-16:** `choice` `options:` accepts EITHER a scalar list
  (`options: [none, flask, fastapi]`, common case) OR a per-option object
  (`{ value: flask, prompt: "Flask (lightweight)" }`) when a custom numbered-
  picker label is wanted. Effects live in `extend:` keyed by value, not inline
  in `options`.
- **D-17:** `case:` and `validate:` remain **string-only** semantics (a boolean
  is parsed via accept/reject; a choice is constrained by `options`).
- **D-18:** `--defaults` (existing `self.app.pargs.defaults`) uses each
  variable's `default` without prompting, across all types (bool default → the
  bool; choice default → the option value).
- **D-19:** Migration scope is in-repo only: rewrite the 5 cli generate
  templates' `.generate.yml` that use features, the `demo/generate-features/`
  template + README, and the ~22 `tests/data/templates/generate/test*/`
  fixtures (+ `tests/ext/test_ext_generate.py`) to the new schema. 100%
  coverage must hold.

### Claude's Discretion
- Exact internal refactor of `_process_features` / `_generate` (the two will
  likely collapse into one ordered variable-resolution pass) — planner/executor
  decide structure, as long as the released `variables:` behavior is preserved
  and the new schema (D-04..D-15) is honored.
- Precise regex/anchoring rules for `extend.when` string matching (likely
  `re.match`, mirroring `validate:` at `ext_generate.py:322`).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `cement.utils.shell.Prompt` — already used for both the boolean feature
  prompt (`ext_generate.py:185`) and variable prompts (`:304`). The unified
  prompt engine builds on this.
- Lazy recursive `_resolve()` (`:118-193`) — already provides order-independent
  resolution and "don't nag for a feature whose `requires` is false" behavior.
  Reuse for D-11 `requires` resolution.
- `or []` coalescing idiom for `variables`/`exclude`/`ignore` (`:227-229`,
  `:252-254`) — keep for the merged blocks.

### Established Patterns
- Variables already land top-level in `data` (`:325`) — D-05 just routes the
  former features through the same sink instead of `data['features']`.
- `validate:` uses `assert re.match(...)` and aborts (no re-prompt) — D-14's
  boolean accept/reject invalid-input behavior mirrors this exactly.
- YAML loaded via `yaml.full_load` (`:245`) → YAML 1.1 boolean key coercion is
  why D-13 avoids bare `yes`/`no`/`true`/`false` keys.

### Integration Points
- `self.app.template.copy(source, dest, data, ...)` (`:328`) — the render
  boundary; `data` is the Jinja/template context. Top-level exposure (D-05)
  means `data[name]` for every variable/feature.

</code_context>

<specifics>
## Specific Ideas

- Tom's verbatim examples (kept as concrete UX targets, not literal schema):
  - desired prompt feel: `Enable automate [(Y)es/(N)o] [Y]:` (vars-style)
  - the broken case: `{% if feature_nicegui %}` must work for a boolean
  - the working contrast: `{% if feature_automate == "Y" %}` (a string var)
- Author-facing target shape (illustrative, from discussion):
  ```yaml
  variables:
    - name: web_framework
      type: choice
      options: [none, flask, fastapi]
      default: none
      extend:
        - when: [flask, fastapi]
          variables:
            - name: bind_host
              prompt: "Bind host"
              default: "0.0.0.0"
    - name: enable_nicegui
      type: boolean
      default: true
      prompt:
        text: "Use nicegui? [(Y)ay/(N)ay]"
        accept: [y, yay]
        reject: [n, nay]
  ```

</specifics>
