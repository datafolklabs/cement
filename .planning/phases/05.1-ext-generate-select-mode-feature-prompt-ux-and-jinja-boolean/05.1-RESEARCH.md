# Phase 05.1: ext.generate select-mode feature prompt UX and Jinja boolean - Research

**Researched:** 2026-05-29
**Domain:** Cement `ext.generate` template engine — variable/feature resolution, `shell.Prompt`, YAML 1.1 schema design, jinja2/mustache rendering
**Confidence:** HIGH (all claims verified against in-repo source + empirical Python runs in the project venv)

## Summary

This phase collapses the *unreleased* `features:` mechanism in `cement/ext/ext_generate.py`
into the *released* `variables:` schema. The whole feature engine (`_process_features`,
~lines 29–229) is removed and merged into a single, ordered, depth-first variable
resolution pass that replaces both `_process_features` and the `_generate` variables loop
(~lines 274–325). All the reusable machinery already exists in-repo: `shell.Prompt`
(numbered/options/default/case-insensitive/retry), the lazy recursive `_resolve()` pattern
for order-independent dependency resolution, and the top-level `data[name]` render sink.
There are **no new runtime dependencies** and **no external research targets** — this is a
self-contained refactor of existing code against existing libraries.

The two highest-risk areas are (1) the **single-pass control-flow redesign** that must
interleave declaration-order prompting, depth-first nested `extend.variables`, and
order-independent `requires` resolution while preserving byte-identical behavior for plain
released variables; and (2) the **YAML 1.1 boolean-coercion landmine** — empirically
confirmed that `yes`/`no`/`on`/`off`/`true`/`false` coerce to Python bools *both* as map
keys *and* as unquoted list values under `yaml.full_load`. This is exactly why D-13 names
the token keys `accept:`/`reject:`, but it also means the *values* inside those lists
(`accept: [y, yes]`) silently coerce `yes → True` and must be quoted or normalized.

**Primary recommendation:** Build one recursive resolver `_resolve_variable(name)` keyed by
declaration-order traversal, reusing the existing lazy-`_resolve` memoization for `requires`,
emitting `data[name]` (typed bool/str) for every variable, and accumulating
`extend`-block `variables`/`exclude`/`ignore` as rules fire. Keep the `assert`-and-abort
failure model (matching `validate:`) for the boolean object-form junk-input case. Cover all
new interactive prompt branches with `# pragma: nocover  # defensive: unreachable` per the
existing convention; exercise every non-interactive path via `--defaults` and `prompt: false`.

### Key finding — re-prompt vs assert-abort
`shell.Prompt` **does** have a built-in retry loop, but it only re-prompts when input fails an
`options` membership/numbered check. For a **bare string** prompt (no options), any non-empty
input is accepted as-is. Therefore:

- **`type: boolean` string-form** (`prompt: "Use X?"`): build a Prompt with **no** options and
  parse the returned token externally (`y/yes`→True, `n/no`→False, empty→default). There is no
  framework-level whitelist, so **per D-14 the model is assert-abort, NOT re-prompt** — consistent
  with how `validate:` aborts (L321-323). Decide: either (a) treat unknown input as invalid and
  `assert`+abort (recommended, mirrors D-14/validate), or (b) drive it through `options=['y','n',
  'yes','no']` to get free retry. **Recommendation: (a) assert-abort** — D-14 explicitly mirrors
  `validate:` abort semantics, and a re-prompt loop would diverge from the established UX.
- **`type: boolean` object-form** (`{text, accept, reject}`): same — read `.prompt()`, lowercase,
  membership-test against `accept`/`reject` token lists; match neither → `assert`+abort with an
  `Invalid Response`-style message (D-14).
- **`type: choice`**: use `options` + `numbered=True` and let the built-in retry/whitelist handle
  invalid numbers (this is the existing select-mode behavior at L165-176, currently pragma'd).

## Architecture Patterns

### Recommended control-flow: one ordered, recursive resolver

Replace `_process_features` + the `_generate` variables loop with a single function that walks
the top-level `variables` list **in declaration order**, and for each variable:

```
resolve_and_emit(var):
    if var.name already in data: return            # memoized (requires re-entry)
    if var has `requires`: for each req, resolve_and_emit(req) first  # order-independent
        if any requires-gate fails (truthy/equality/in-list mismatch): SKIP this var entirely
    value = prompt_or_default(var)                  # typed by var.type
    apply case (string-only) + validate (string-only)
    data[var.name] = value                          # top-level, typed
    for rule in var.extend or []:                   # rules compose
        if when_matches(value, rule.when):
            exclude_list += rule.exclude or []
            ignore_list  += rule.ignore  or []
            for nested in rule.variables or []:     # depth-first, in place
                resolve_and_emit(nested)            # full recursion
```

Notes:
- **Declaration order** (D-06) is the outer loop order; **depth-first nesting** (D-08) is the
  recursion inside a fired rule.
- **`requires` order-independence** (D-11): resolve the required variable on demand (memoized via
  `data` membership or a separate `resolved` set), exactly like the existing lazy `_resolve`
  (L118-193). A required var declared later still resolves before its dependent.
- **Skip vs disable:** a variable whose `requires` gate fails is simply not prompted and not
  emitted (or emitted as its falsy/default — decide explicitly; the current feature code sets
  `state=False` and merges the `disabled` block — but there is no `disabled` block anymore, so a
  failed-requires variable should likely be **omitted from `data` or set to its `default`**.
  **OPEN QUESTION Q1** below.)
- **Nesting ≡ requires** (D-10): a nested `extend.variables` child only resolves when the parent
  rule fired, which is structurally equivalent to `requires: {parent: <matched value>}`. Both must
  produce the same gating outcome.

### `extend.when` matching semantics (verified design)

`when` is matched against the **resolved value** of the carrying variable:

| `when` form | Match rule | Notes |
|-------------|-----------|-------|
| scalar (`when: flask`) | equality after str-coercion of both sides | choice values are str; bool values compare against Python bool — see coercion note. |
| list (`when: [flask, fastapi]`) | `value in list` (str-coerce members) | in-list membership. |
| regex (string type only) | `re.match(pattern, value)` | Mirror `validate:` at L322 — `re.match` (anchored at start, not end). D-16 discretion notes `re.match`. Only valid for `type: string`. |

**str-coercion rule:** the current select code coerces both sides with `str()` (`str(opt['value'])
== state`, L219) precisely because YAML may decode `value: 1` as int. Reuse the same str-coerce-
both-sides discipline for `when` scalar/list compares. For **boolean** carrying variables, decide
whether `when: true` (YAML→Python `True`) compares against the Python bool value or str — recommend
comparing booleans **by Python equality** (the resolved value IS a real bool per D-05) and reserving
str-coercion for choice/string. **OPEN QUESTION Q2.**

### Template-context exposure — verified for BOTH engines

`[VERIFIED: empirical run in project venv, 2026-05-29]`

```
jinja2:  {% if x %} with x=True  → "Y";  x=False → "N";  {{ x }} with True → "True"
pystache:{{#x}}/{{^x}} with True → "Y";  False → "N";  {{ x }} with True → "True"
```

- Writing `data[name] = True/False` (real Python bool) makes `{% if feature_x %}` work in
  **both** jinja2 and mustache (pystache). This is the direct fix for #782 pt 4.
- **Gotcha:** `{{ x }}` for a bool renders the capitalized Python repr `"True"`/`"False"` in both
  engines — NOT `"true"`/`"false"` and NOT `"yes"`/`"no"`. Template authors who interpolate a bool
  as text (rare) get `True`/`False`. Document in the demo. Conditionals (`{% if %}`, `{{#x}}`) are
  the intended use and behave correctly.
- The render boundary is `self.app.template.copy(source, dest, data, ...)` at L328. `data` is the
  full context; every variable lands at `data[name]` top-level. The `data['features']` namespace
  (L199-204) is **removed entirely** (root of bug #4).

### Anti-Patterns to Avoid
- **Don't keep a two-pass "resolve all states, then merge" split.** The current code resolves all
  feature states first (L195-196) then merges blocks (L201-229). The new design must interleave so
  nested `extend.variables` prompt *in place* in declaration order (D-08). A separate merge pass
  would prompt nested vars out of order.
- **Don't namespace under `data['features']`.** Removed — top-level only (D-05).
- **Don't silently coerce unknown boolean input to False.** D-14: assert+abort, no silent coercion.
- **Don't use bare `yes`/`no`/`on`/`off`/`true`/`false` as YAML keys OR unquoted list values** in
  any new schema or fixture (see landmine below).

## Common Pitfalls

### Pitfall 1: YAML 1.1 boolean coercion — keys AND values
**What goes wrong:** Under `yaml.full_load` (YAML 1.1), the tokens
`yes/no/on/off/true/false/y/n`(some) coerce to Python bools — **both as map keys and as unquoted
list values.**
**Verified empirically (project venv, 2026-05-29):**
```
prompt: {text, yes: [...], no: [...], true: [...], on: [...]}  → keys: ['text', True, False]
   (yes AND on both collapse to True → key collision, 5 keys become 3)
accept: [y, yes]   → ['y', True]      ← the bare `yes` VALUE coerces to Python True
accept: [y, "yes"] → ['y', 'yes']     ← quoting prevents it
default: true      → Python True (correct, intended for bool defaults)
```
**Why it happens:** PyYAML's YAML 1.1 boolean resolver. The released schema dodged it because
`name/prompt/validate/case/default` keys aren't boolean tokens.
**How to avoid:**
- D-13 already names the *keys* `accept:`/`reject:` (safe — confirmed not coerced).
- **NEW finding for the planner:** the *token lists themselves* (`accept: [y, yes]`) will coerce
  `yes`/`no`/etc. to `True`/`False`. Either (a) require authors to quote bool-like tokens, (b)
  str-coerce every accept/reject member at load (`str(tok).lower()` — but `str(True)` → `"True"`,
  not `"yes"`, so this is lossy), or (c) document that bool-like tokens must be quoted. **Recommend
  (c) + validation:** if an accept/reject member loaded as a Python bool, raise `ValueError` telling
  the author to quote it. This keeps the failure fast and explicit.
**Warning signs:** an `accept: [y, yes]` that mysteriously never matches the literal string `"yes"`.

### Pitfall 2: Breaking the released `variables:` byte-for-byte
**What goes wrong:** Refactoring the variable loop changes behavior for a plain
`{name, prompt, default}` variable.
**Why it happens:** The `var_defaults` dict (L266-272) and the silent/`--defaults`/prompt branch
ladder (L281-310) encode exact released semantics, including `prompt: false` (L282-293), the
`str(default)` coercion (L293, locked by test29), and `assert var['default'] is not None` for
silent (L290-292, locked by test27).
**How to avoid:** Treat `type: string` with no new keys as the identity path. Keep the
`var_defaults` merge and the silent-variable short-circuit verbatim. Run test1–test5, test22,
test27, test29 unchanged as the compat oracle.
**Warning signs:** any diff in test1/test22/test27/test29 output.

### Pitfall 3: `--defaults` must cover every type
**What goes wrong:** `--defaults` works for strings but not bool/choice.
**Why it happens:** `--defaults` (D-18) must dispatch each type to its default without prompting:
bool default → the Python bool, choice default → the option value, string default → the string.
**How to avoid:** Branch on `type` in the `--defaults` path; all current feature fixtures run with
`--defaults` (the only way they're testable non-interactively).
**Warning signs:** a `--defaults` run that hangs on stdin (`OSError: reading from stdin while
output is captured` — see test_prompt) means a type wasn't handled in the defaults path.

### Pitfall 4: Coverage on new interactive branches
**What goes wrong:** New prompt code drops coverage below 100%.
**Why it happens:** Interactive `Prompt().prompt()` branches can't run under captured stdin.
**How to avoid:** Follow the existing convention — the 22 existing `# pragma: nocover` markers
(L141-192) all sit on interactive Prompt branches, labeled `# pragma: nocover` (select) or
`# pragma: nocover  # defensive: unreachable` (variable prompt L301, L310). Use
`# defensive: unreachable` for the new bool/choice interactive branches. Where a branch *can* be
driven (e.g. `test_no_default` patches `shell.Prompt.prompt`), prefer a real test over a pragma.
**Warning signs:** `make test` coverage < 100%.

## 100% Coverage Strategy

`[VERIFIED: existing test patterns, ext_generate.py + test_ext_generate.py]`

The existing suite achieves 100% on interactive code via two mechanisms — both reusable:

1. **`--defaults` drives every non-interactive path.** Every feature test passes `--defaults`
   (e.g. `['generate', 'test16', tmp.dir, '--defaults']`). This exercises the
   resolve/merge/validate logic without touching `Prompt().prompt()`. The new resolver's
   default-dispatch branches (one per `type`) are covered this way.
2. **Interactive Prompt branches are `# pragma: nocover  # defensive: unreachable`.** All 22
   existing pragmas (select prompt L141-176, bool-feature prompt L182-192, variable prompt L301/
   L310) mark code that can't run under captured stdin. New interactive branches (bool string-form
   parse-after-prompt, bool object-form accept/reject-after-prompt, choice numbered prompt) get the
   same treatment.
3. **Where a branch CAN be tested, patch `shell.Prompt.prompt`.** `test_no_default` does exactly
   this: `patch.object(shell.Prompt, 'prompt', return_value='Bogus')`. **Recommend** using this to
   actually cover the bool accept/reject *parsing* logic (which is NOT inside Prompt and therefore
   IS coverable): patch `prompt()` to return `"yes"`, `"n"`, junk → assert True/False/abort. This
   keeps the y/n→bool mapping and the D-14 assert-abort under real test coverage rather than pragma.

**Net:** the *prompt rendering/input* stays pragma'd; the *value-mapping and validation* logic
(the part that actually has bugs) should be unit-tested via patched `prompt()`. This is both higher
coverage quality and required to hit 100% on the new mapping branches.

## Open Questions (RESOLVED)

> Resolved during planning (Phase 05.1 plans 02/03). Decisions locked below.

1. **Failed-`requires` variable: omit from `data`, or set to default/falsy?** (Q1)
   - What we know: today a failed-requires feature is set `state=False` and its `disabled` block
     merges. There is no `disabled` block in the new schema.
   - What's unclear: does a `requires`-gated-out variable appear in `data` at all? If a template
     does `{% if optional_var %}` and `optional_var` was gated out, it must be falsy/absent.
   - **RESOLVED:** set a `requires`-gated-out variable to its `default` at `data[name]` (and do not
     fire its `extend` rules) so `{{ var }}`/`{% if var %}` never KeyErrors in the template.
     Implemented in Plan 03 Task 2.

2. **`extend.when` matching for boolean values: Python-equality or str-coerce?** (Q2)
   - What we know: choice/string values str-coerce both sides (existing L219 pattern). Booleans are
     real Python bools (D-05).
   - What's unclear: does `when: true` (YAML→Python `True`) compare to the bool value by `==` or
     by `str()`? `str(True)=='True'` would break a `when: true` against a real bool.
   - **RESOLVED:** compare booleans by Python `==`; str-coerce both sides only for choice/string
     (scalar equality lands in Plan 02 Task 2, list/regex in Plan 03 Task 2). A `type: boolean`
     variable's `extend: [{when: true, ...}]` matches when the value is Python `True`.

3. **Is `choice` `options:` empty/missing-default still a validated error?** (Q3)
   - test26/test28 currently ValueError on empty options / missing default in select mode.
   - **RESOLVED:** keep fail-fast `ValueError` validation for choice (empty options, default not in
     options, duplicate picker labels per D-16) — preserves the test26/28/30 spirit under the new
     schema. Implemented in Plan 02 Task 1/Task 2.
