## Pattern Assignments

### `.github/workflows/gates.yml` (reusable workflow, `workflow_call`)

**Analog:** `.github/workflows/build_and_test.yml` — the current serial gate chain
IS the gate content (D-13). Refactor, do not rewrite. Preserve the serial
`needs:` chain (comply → test → test-all → cli-smoke-test) per Phase 02 D-16
fail-fast; NEW OS jobs (win-core, mac/win native smoke) run in parallel with the
matrix (independent) per Research Open Question 3.

**Trigger + inputs pattern** (replace lines 3-7 of the analog; D-13/D-14):
```yaml
name: Gates
on:
  workflow_call:
    inputs:
      full-os-matrix:
        type: boolean
        default: true   # release always true; PR true per D-14
```

**Top-level env — copy verbatim** (analog lines 9-13, service host wiring for full `make test`):
```yaml
env:
  SMTP_HOST: localhost
  SMTP_PORT: 1025
  MEMCACHED_HOST: localhost
  REDIS_HOST: localhost
```

**Linux gate job shape — copy verbatim** (analog lines 36-58, the `test` job is the canonical Linux job: checkout → apt libmemcached-dev → setup-python → compose services → setup-pdm → `pdm install` → `make test`):
```yaml
  test:
    needs: comply
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7.0.0
      - uses: ConorMacBride/install-package@v1.1.0
        with:
          apt: libmemcached-dev
      - name: Set up Python
        uses: actions/setup-python@v6.3.0
        with:
          python-version: "3.x"
          architecture: "x64"
      - uses: hoverkraft-tech/compose-action@v3.0.0
        with:
          compose-file: "./docker/compose-services-only.yml"
      - name: Setup PDM
        uses: pdm-project/setup-pdm@v4.5
      - name: Install dependencies
        run: pdm install
      - name: Make Test
        run: make test
```

**Matrix pattern — copy verbatim** (analog lines 60-66; keep pypy per D discretion default):
```yaml
  test-all:
    needs: test
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest]
        python-version: ["3.10", "3.11", "3.12", "3.13", "3.14", "pypy3.10", "pypy3.11"]
```

**Linux docker smoke — copy verbatim** (analog lines 83-96, drives full `docker-compose.yml` + `./scripts/cli-smoke-test.sh`; stays the DEEP gate, Linux-only):
```yaml
  cli-smoke-test:
    needs: test-all
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7.0.0
      - uses: hoverkraft-tech/compose-action@v3.0.0
        with:
          compose-file: "./docker-compose.yml"
      - name: CLI Smoke Tests
        run: ./scripts/cli-smoke-test.sh
      - if: always()
        name: Review Output
        run: cat ./tmp/cli-smoke-test.out
```

**NEW jobs (no analog — see "No Analog Found"):** `test-core-windows`,
`cli-smoke-native`. Critical: use `pdm run pytest ...` / `pdm run ruff` / `pdm
run mypy` DIRECTLY on Windows/macOS — never `make` (Research Pitfall 5). The
Windows core-test equivalent of `make test-core` (`Makefile` lines 31-32) is
`pdm run pytest --cov=cement.core tests/core`.

### `.github/workflows/release.yml` (NEW release pipeline)

**Analog:** `build_and_test.yml` for job/step idioms (checkout, setup-python,
setup-pdm pins), `pdm.yml` for the `workflow_dispatch` trigger idiom. All
concrete job bodies are specified in RESEARCH.md "Code Examples" — reference
those directly. Below are the pattern anchors the planner must carry.

**Action pins — copy exactly from `build_and_test.yml`** (CI-05 exact-tag
discipline; already-in-repo pins are the source of truth):
```yaml
actions/checkout@v7.0.0
actions/setup-python@v6.3.0
pdm-project/setup-pdm@v4.5
ConorMacBride/install-package@v1.1.0
hoverkraft-tech/compose-action@v3.0.0
```
New pins (verified in RESEARCH Standard Stack, 2026-07-12):
`pypa/gh-action-pypi-publish@v1.14.0`, `docker/setup-qemu-action@v4.2.0`,
`docker/setup-buildx-action@v4.2.0`, `docker/login-action@v4.4.0`,
`docker/build-push-action@v7.3.0`, `actions/upload-artifact@v7.0.1`,
`actions/download-artifact@v8.0.1`, `peter-evans/create-pull-request@v8.1.1`,
`actions/create-github-app-token@v3.2.0` (conditional).

**Trigger pattern** (D-01/D-06/D-09; strict 3-part tag guard per Pitfall 7 —
avoids moving tags `3`/`3.0`):
```yaml
on:
  push:
    tags: ['[0-9]+.[0-9]+.[0-9]+']
  workflow_dispatch:
```

**Gates reuse** (D-13 — calls the SAME file PR CI calls):
```yaml
  gates:
    needs: preflight
    uses: ./.github/workflows/gates.yml
    with: { full-os-matrix: true }
```

**Single-approval env-scoped OIDC publish** (D-03/D-04 — THE load-bearing
pattern; `environment: release` sits on the ONE publish job and both pauses for
approval AND emits the OIDC env claim; see RESEARCH Pattern 2 + Pitfall 1):
```yaml
  publish-pypi:
    needs: [testpypi-smoke]
    if: github.event_name == 'push'   # dispatch dry-run never publishes (D-06)
    environment: release
    permissions: { id-token: write }
```
Anti-pattern (RESEARCH): do NOT put a no-op approval job before an ungated
publish, and do NOT put `environment: release` on multiple downstream jobs.
Keep the OIDC publish in THIS top-level file — never inside `gates.yml`.

**Downstream fan-out, preflight, testpypi stage, docker buildx, gh-release,
branch-sync, dev-bump, checklist issue:** all have complete copy-ready YAML in
RESEARCH.md "Code Examples" and "Architecture Patterns" — planner references
those line-for-line. Default `permissions: contents: read` at workflow level,
elevate per-job (`id-token: write`, `contents: write`, `pull-requests: write`).
`concurrency` with `cancel-in-progress: false` (RESEARCH anti-pattern).

### `scripts/testpypi-smoke.py` (NEW — install/import/round-trip, D-07/REL-02)

**Analog:** `scripts/audit-public-api.py` (dev-script structure) +
`scripts/devtools.py` (canonical `App` subclass usage).

**Header/shebang convention — copy from `audit-public-api.py` lines 1-2, 47-51**
(shebang `#!/usr/bin/env python3`, module docstring, explicit exit codes; these
`scripts/*.py` are NOT in the wheel — `pyproject.toml` `[tool.pdm.build] includes`
lists only `cement/` + templates + CONTRIBUTORS/CHANGELOG — so they are lint/type
clean but outside the coverage surface):
```python
#!/usr/bin/env python3
"""
testpypi-smoke.py — install cement==X.Y.Z from TestPyPI, import, run App().run().
...
Exit codes:
* 0 — smoke passed
* non-zero — install/import/run failure
"""
```

**App round-trip pattern — mirror `devtools.py` lines 153-164** (minimal `App`
subclass with nested `Meta`, `with App() as app: app.run()`). RESEARCH "Code
Examples" gives the exact smoke body (`exit_on_close = False`, version assert).

**Pitfall 2 (RESEARCH):** install PLAIN `cement==X.Y.Z` (core is zero-dep —
resolves from TestPyPI alone); only add `--extra-index-url
https://pypi.org/simple/` if smoking the `[cli]` extra (jinja2/pyyaml live on
real PyPI, not TestPyPI).

### `scripts/cli-smoke-native.py` (NEW — cross-platform smoke, D-14)

**Analog:** `scripts/cli-smoke-test.sh` (the assertion logic to port) +
`scripts/audit-public-api.py` (Python dev-script skeleton).

**Assertions to port from `cli-smoke-test.sh`** (re-express `docker exec ... |
grep` as `subprocess.run` + string asserts):
- `cement --version` → banner grep (lines 31-34). CRITICAL platform fix: the
  shell script greps `Platform Linux.*` (line 34) — the native version must
  accept `Linux`/`Darwin`/`Windows` (Pitfall 3). The banner source is
  `cement/utils/version.py:get_version_banner` (lines 76-85) — `Platform
  {platform.platform()}`.
- `cement --help` / `-h` → "Cement Framework Developer Tools", "generate" (lines 35-40)
- `cement generate -h` → project/todo-tutorial/plugin/script/extension (lines 41-46)
- `cement generate project -D $tmp/myapp` → `pip install .` → `myapp --version`
  banner (lines 54-59)

**Tiering (RESEARCH Pitfall 3 / Assumption A6):** native macOS/Windows smoke is
the LIGHTER CLI+generate+install-generated check. Do NOT port the heavy
"generated project `pdm install` + `make comply` + `make test`" depth (shell
lines 62-86) — that stays Linux-docker-only (`make`+services unavailable on
Windows). Tier explicitly.

**Subprocess idiom** (repo precedent for shelling out lives in
`cement/utils/version.py:get_git_changeset` lines 88-106 using
`subprocess.Popen`; `devtools.py` uses `cement.utils.shell.exec_cmd`). For a
standalone dev script prefer stdlib `subprocess.run(..., check=True,
capture_output=True, text=True)`.

### `scripts/bump_dev_version.py` (NEW — VERSION + changelog bump, D-12/REL-05)

**Analog:** `scripts/audit-public-api.py` (dev-script skeleton, `pathlib`, exit
codes) + `cement/core/backend.py` (the VERSION source of truth) + `CHANGELOG.md`
(the section format to prepend).

**VERSION source of truth** (`cement/core/backend.py` line 3 — the CONTEXT.md
shorthand "bump `cement/utils/version.py` VERSION" actually targets THIS tuple,
read via `get_version()`):
```python
VERSION = (3, 0, 15, 'final', 0)  # pragma: nocover  # version constant
```
Bump `3.0.16` → next dev is `(3, 0, 17, 'final', 0)`. Preserve the exact inline
`# pragma: nocover` comment when rewriting (regex-replace the tuple, keep the
trailing comment — do not disturb coverage pragmas).

**CHANGELOG prepend format** (`CHANGELOG.md` lines 1-3 — a fresh dev section goes
directly under the `# ChangeLog` H1; note the repo's live convention pairs
"DEVELOPMENT (will be released as stable/X.Y.Z)"):
```markdown
# ChangeLog

## 3.0.17 - DEVELOPMENT

## 3.0.15 - DEVELOPMENT (will be released as stable/3.0.16)
...
```
The D-02 preflight (RESEARCH Code Examples) validates the INVERSE — that the
releasing version's section is NOT `- DEVELOPMENT`; this script re-opens the next
one. Keep the two format regexes consistent (`^## +<ver>` anchor).

**Bucket structure to seed** (CLAUDE.md changelog policy): Bugs / Features /
Refactoring / Misc / Deprecations. Script writes `[[tool.pdm.version]] source =
"call"` means version flows from `backend.py` automatically — do NOT hand-edit
`pyproject.toml` version (`pyproject.toml` lines 224-226).

## Shared Patterns

### Action version pinning (CI-05)
**Source:** `.github/workflows/build_and_test.yml` (every `uses:` is `@vX.Y.Z`)
**Apply to:** `gates.yml`, `release.yml`
Exact-tag pins only, no floating refs. Static gate (RESEARCH Validation):
`grep -rE 'uses:.*@' .github/workflows/ | grep -vE '@v[0-9]'` returns empty.
Reuse in-repo pins verbatim; new pins from RESEARCH Standard Stack table.

### Serial fail-fast gate chain (Phase 02 D-16)
**Source:** `build_and_test.yml` `needs:` chain (comply→test→test-all→cli-smoke-test)
**Apply to:** `gates.yml`
Carry the serial `needs:` chain; only the NEW independent OS jobs may parallelize.

### Dev-script structure (shebang + docstring + explicit exit codes)
**Source:** `scripts/audit-public-api.py` lines 1-2, 47-51; `from __future__ import
annotations` line 52 ("script-internal; doesn't affect cement/")
**Apply to:** `testpypi-smoke.py`, `cli-smoke-native.py`, `bump_dev_version.py`
All three: executable shebang, module docstring with a `Phase 05.4` contract
anchor block (mirrors the D-01.. anchors in audit-public-api.py), documented exit
codes, `pathlib`, ruff+mypy clean, NOT part of the wheel/coverage surface.

### `App` subclass round-trip
**Source:** `scripts/devtools.py` lines 153-164 (`class ...App(App): class Meta:
... with App() as app: app.run()`)
**Apply to:** `testpypi-smoke.py`
Minimal `App` subclass with nested `Meta`; use `exit_on_close = False` for the
smoke (per RESEARCH Code Example) so the round-trip returns to the script.

### Cross-platform banner assertion
**Source:** `cement/utils/version.py:get_version_banner` lines 76-85
**Apply to:** `cli-smoke-native.py` (accept `Linux`/`Darwin`/`Windows`, NOT the
hardcoded `Platform Linux.*` from `cli-smoke-test.sh` line 34)
