# Phase 05.4: GitHub Actions Release Workflow - Context

**Gathered:** 2026-07-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Automate the manual release checklist in issue #791 as a GitHub Actions–driven
release workflow. Cutting a release becomes: push a release tag → gates run →
one human approval → PyPI publish, Docker Hub multi-arch images, `stable/3.0.x`
branch sync, docs builds, GitHub Release, and an emitted post-release human
checklist. Hard pre-requisite for Phase 6 — the 3.0.16 Release Cut runs this
workflow. Locked by ROADMAP success criteria (not re-decided here): gates
hard-stop the release on any failure; OIDC trusted publishing where available;
Docker `linux/amd64` + `linux/arm64`; GitHub Release published with changelog
body.

</domain>

<decisions>
## Implementation Decisions

### Trigger & approval flow
- **D-01:** Tag push is the canonical release trigger. Pushing tag `3.0.16`
  starts the run: gates → TestPyPI validation → approval → publish. The human
  act of tagging is the source of truth; a failed gate means delete/re-tag.
- **D-02:** The run starts with a fast pre-flight job that verifies (a) the
  tag exactly matches the package version (`cement.utils.version:get_version`
  / `pdm show`) and (b) `CHANGELOG.md` has a finalized section for the version
  (not `- DEVELOPMENT`). A malformed release dies in seconds, not after a
  30-minute matrix.
- **D-03:** The human approval gate is a GitHub Environment (e.g. `release`)
  with required reviewers on the publish side. GitHub natively pauses the run;
  PyPI OIDC trusted-publishing scoping attaches to this environment.
- **D-04:** Exactly ONE approval stop. After gates + TestPyPI smoke pass, a
  single approval unlocks the entire publish side: PyPI, Docker Hub, branch
  sync, GitHub Release, docs trigger, post-release outputs. Nothing publishes
  before approval; nothing needs a second approval after it.

### TestPyPI rehearsal path
- **D-05:** TestPyPI is an inline stage of every release run — gates → build
  sdist+wheel → upload to TestPyPI → install smoke → approval → PyPI. The
  approved artifacts are always index-install-proven (REL-02 by construction).
- **D-06:** `workflow_dispatch` is a first-class DRY-RUN mode, runnable from
  any branch before a tag exists: pre-flight + gates + build + TestPyPI +
  smoke, ending in a summary. It never reaches approval/publish. Tag push is
  the ONLY path to PyPI. This is how Phase 6 validates the workflow end-to-end
  before the real 3.0.16 tag (CI-04).
- **D-07:** TestPyPI install smoke is a matrix over all 5 CPythons
  (3.10–3.14): `pip install cement==X.Y.Z` from TestPyPI + import + a minimal
  `App().run()` round-trip script. Exactly REL-02's wording.
- **D-08:** Uploads use `--skip-existing` semantics: if the version already
  exists on TestPyPI (re-run after late failure, repeated dry-run), skip the
  upload but still run the install smoke against the existing artifact.
  Re-runs are idempotent. Accepted caveat: a re-run smokes whatever was first
  uploaded for that version.

### Naming & version conventions
- **D-09:** Tags stay unprefixed (`3.0.16`, matching `3.0.2`…`3.0.8` history).
  Trigger pattern on `3.*`-style tags with a strict semver guard in pre-flight.
- **D-10:** The workflow fast-forwards the existing `stable/3.0.x` branch to
  the release tag. Issue #791's "stable/X.Y" is shorthand — the repo
  convention is `stable/X.Y.x` (see stable/2.8.x, stable/2.10.x). Create
  `stable/X.Y.x` if a future minor doesn't have one.
- **D-11:** Docker Hub push set: `datafolklabs/cement:3`, `:3.0`, `:3.0.16`,
  AND `:latest` (preserves current Makefile behavior for `:latest` users).
  All multi-arch (amd64 + arm64) via buildx.
- **D-12:** After publish, the workflow opens an automated PR against `main`
  bumping `cement/utils/version.py` VERSION to the next dev version (3.0.17)
  and adding a fresh `## 3.0.17 - DEVELOPMENT` changelog section. No direct
  push to main; the user merges the PR (REL-05 with branch discipline held).

### Gate architecture & OS coverage
- **D-13:** The gate jobs (comply, test, Python matrix, cli-smoke-test) are
  refactored into a reusable `workflow_call` workflow. Both PR CI
  (`build_and_test.yml`) and the release workflow invoke the same definition —
  one source of truth, gates cannot drift between PR and release.
- **D-14:** The NEW OS jobs — Windows core tests (`make test-core`-shaped;
  full `make test` needs Linux docker services) and macOS/Windows CLI smoke
  tests — join the shared gates and run on every PR too, not just at release.
  Public repo = free runner minutes; release day stays boring.
- **D-15:** Notifications (mailing list, Slack) stay HUMAN steps this phase —
  no webhook secrets in the release path. They appear on the emitted
  post-release checklist. Automation can be revisited after the workflow has
  a release or two behind it.
- **D-16:** The post-release human checklist (GitBook docs updates, mailing
  list, Slack, merge the dev-bump PR) is emitted as an auto-created GitHub
  issue ("3.0.16 post-release checklist") with checkboxes, assigned to the
  maintainer — mirrors the #791 tracker style — and echoed in the job summary.

### Claude's Discretion
- Exact job graph/parallelization inside the release workflow, action choices
  (pypa/gh-action-pypi-publish, docker/build-push-action, etc.) and version
  pinning per CI-05 discipline.
- Mechanics of the RTD docs trigger (RTD auto-builds on tag/branch push via
  its webhook; verify `.readthedocs.yaml` version activation covers it).
- How the cli-smoke-test script is adapted to run natively on macOS/Windows
  runners (current `scripts/cli-smoke-test.sh` drives docker compose).
- Concurrency guards, GITHUB_TOKEN permissions scoping, and GitHub Release
  body extraction from CHANGELOG.md.
- Whether pypy stays in the shared matrix for release gates (it is in PR CI
  today; keeping it is the default).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `build_and_test.yml` job chain: comply → test → test-all (3.10–3.14 +
  pypy3.10/3.11, Linux) → cli-smoke-test. This IS the gate content; D-13
  refactors it to `workflow_call` rather than rewriting it.
- `docker/compose-services-only.yml` powers redis/memcached/mailpit services
  for full `make test` — Linux-only, which is why Windows runs `test-core`.
- `make dist` = `pdm build` (pdm-backend, PEP 517-isolated since Phase 01.1 /
  05.3) — sdist+wheel build is already one command.
- Phase 05.3's cli-smoke-test already runs generated-project `make comply` +
  `make test` inside matrix containers — release gates inherit this for free
  via reuse.

### Established Patterns
- Phase 02 D-16: PR CI keeps a SERIAL job graph (fail-fast preference) —
  carry into the reusable gates rather than parallelizing everything.
- Action version pinning is exact (`actions/checkout@v7.0.0`,
  `setup-python@v6.3.0`) per CI-05 — new actions follow the same discipline.
- AUDIT POINT comment convention (Phase 1 D-08 lineage) for any config whose
  drift matters — applies to workflow-level version/format guards.
- Conventional Commits + no direct commits to main (CLAUDE.md) — the D-12
  dev-bump lands as a PR, consistent with repo policy.

### Integration Points
- Tag push on `3.*` → new `release.yml`; `workflow_dispatch` → same file in
  dry-run mode (D-06).
- GitHub Environment `release` — user must configure required reviewers +
  PyPI/TestPyPI trusted publishers (OIDC) before first run; Docker Hub needs
  a classic token secret (no OIDC support).
- `stable/3.0.x` exists locally and on origin — fast-forward target (D-10).
- Existing tags are unprefixed; the trigger pattern must match bare `3.0.16`.

</code_context>
