# Phase 05.4 Plan 04: release.yml Publish Side + Fan-out Summary

Added the publish half of `.github/workflows/release.yml`: a single
env-scoped OIDC PyPI publish job that is the one human approval stop, plus the
post-approval fan-out (Docker Hub multi-arch, stable/3.0.x fast-forward,
GitHub Release with changelog body, 3.0.17 dev-bump PR, and the emitted
post-release checklist issue).

## What Was Built

**Task 1 — `publish-pypi` + `docker` (commit d3fb7dee):**
- `publish-pypi`: `needs: [testpypi-smoke]`, `if: github.event_name == 'push'`
  (dry-run dispatch never reaches it, D-06), `environment: release` (the ONE
  approval AND the OIDC env claim, D-03/D-04/Pitfall 1), `id-token: write` only
  here. Downloads the `dist` artifact and publishes to REAL PyPI via
  `pypa/gh-action-pypi-publish@v1.14.0` with NO `repository-url` and NO
  `skip-existing` (a duplicate version fails loud).
- `docker`: `needs: publish-pypi`, buildx `linux/amd64,linux/arm64`, literal
  D-11 tag set `datafolklabs/cement:{ref_name,3.0,3,latest}`, Docker Hub token
  login (`secrets.DOCKERHUB_USERNAME`/`DOCKERHUB_TOKEN` — Docker Hub has no
  OIDC).

**Task 2 — four fan-out jobs, each `needs: publish-pypi` (commit 810311a2):**
- `branch-sync` (`contents: write`): `git push origin
  <sha>:refs/heads/stable/3.0.x` — fast-forward only; a non-descendant fails by
  design (D-10/Pitfall 6).
- `gh-release` (`contents: write`): awk-extracts the `## <tag>` CHANGELOG
  section into `RELEASE_BODY.md`, then `gh release create ... --notes-file`
  (built-in gh, zero third-party — REL-03/DOCS-03).
- `dev-bump` (`contents: write` + `pull-requests: write`): checks out `main`,
  runs `scripts/bump_dev_version.py 3.0.17`, opens a PR via
  `peter-evans/create-pull-request@v8.1.1` on branch `chore/dev-bump-3.0.17`
  (D-12/REL-05).
- `post-release-checklist` (`issues: write`): `gh issue create` (assignee
  derks) with checkbox items for GitBook / mailing-list / Slack / merge the
  dev-bump PR, echoed to `$GITHUB_STEP_SUMMARY` (D-15/D-16).

Every new action is exact-tag pinned (CI-05). Least-privilege upheld:
`id-token: write` stays only on `publish-pypi`; each fan-out job elevates only
its own scope.

## Deviations from Plan

None affecting behavior. One cosmetic adjustment: initial job comments used
the literal phrase `environment: release`, inflating the acceptance grep-count
to 3. Comments were reworded to reference "the `release` environment" so the
literal `grep -c 'environment: release'` returns exactly 1 (the D-04 gate)
while preserving the exact YAML declaration on `publish-pypi`. This is an
acceptance-criterion compliance edit, not a design change.

Note on `dev-bump`: the RESEARCH snippet showed only checkout + bump-script +
create-pull-request. The script runs via `pdm run`, so `setup-python`,
`setup-pdm`, and `pdm install` steps were added (matching the `preflight`/
`build` jobs) so the step can actually execute — required-for-correctness
(Rule 2), documented here.
