# 05.4 — Release Workflow Provisioning Runbook

> **Purpose.** `.github/workflows/release.yml` and `.github/workflows/gates.yml`
> are inert until the external run-time dependencies that live in **provider UIs**
> (GitHub Environments, PyPI/TestPyPI trusted publishers, Docker Hub secrets, Read
> the Docs automation) exist. Those cannot be created from the repo. This runbook
> is the ordered, exact-value setup guide the maintainer follows **before the
> first real tag push**, plus a pre-first-tag checklist and the accepted
> tradeoffs the maintainer signs off on.
>
> **Scope of the D-06 dry-run vs. the real publish.** A `workflow_dispatch`
> dry-run (CI-04) needs only the **TestPyPI trusted publisher** (items 3 + the
> `testpypi` environment) to go green through `testpypi-smoke`. The **real
> publish** side additionally needs the `release` environment + reviewers (item
> 1), the **PyPI** trusted publisher (item 2), the **Docker Hub** secrets (item
> 4), the **RTD** integration + tag-rule allowance (item 5), and the `stable/3.0.x` fast-forward
> allowance (item 6).

## Exact match values (single source of truth)

These are the literal strings the trusted publishers, environments, and secrets
must match. A mismatch on any of them produces an `invalid-publisher` /
`trusted publisher` OIDC rejection (Pitfall 1) or a silently-ungated publish.

| Field | Exact value | Where it comes from |
|-------|-------------|---------------------|
| GitHub repo slug (owner/name) | `datafolklabs/cement` | `origin` remote |
| Workflow filename | `release.yml` | the publish workflow; trusted publishers match on filename, NOT path |
| Real-publish environment name | `release` | `release.yml` job `publish-pypi` → `environment: release` |
| TestPyPI environment name | `testpypi` | `release.yml` job `testpypi-publish` → `environment: testpypi` |
| PyPI project name | `cement` | the distribution being published |
| TestPyPI project name | `cement` | same distribution, TestPyPI index |
| Docker Hub image repo | `datafolklabs/cement` | `docker` job `tags:` |
| Docker Hub username secret | `DOCKERHUB_USERNAME` | `docker` job `username: ${{ secrets.DOCKERHUB_USERNAME }}` |
| Docker Hub token secret | `DOCKERHUB_TOKEN` | `docker` job `password: ${{ secrets.DOCKERHUB_TOKEN }}` |
| Stable branch (fast-forward target) | `stable/3.0.x` | `branch-sync` job push target |

> **Note — `testpypi` is NOT optional in the current workflow.** The
> `testpypi-publish` job hard-declares `environment: testpypi`, so the OIDC token
> it presents to TestPyPI **carries the `testpypi` environment claim**. The
> TestPyPI trusted publisher must therefore be configured **either** with
> environment `testpypi` (exact match — recommended) **or** with no environment
> constraint at all. Do NOT configure it with a *different* environment name, or
> the OIDC exchange fails (Pitfall 1).

## 1. GitHub Environment `release` (+ `testpypi`) with required reviewers

**Location:** GitHub → `datafolklabs/cement` → **Settings → Environments**.

**Why:** The `release` environment is the single human approval gate (D-04) AND
the OIDC environment-claim anchor (D-03) that the PyPI trusted publisher verifies.
Without it, the real `publish-pypi` job would fan out un-approved (or the OIDC
claim would not match a env-scoped publisher).

**Steps:**
1. **New environment** → name it exactly `release`.
   - Under **Deployment protection rules**, enable **Required reviewers** and add
     the maintainer(s) who must approve every real publish. This is the one and
     only approval stop in the pipeline (every publish-side job `needs:
     publish-pypi`, so nothing runs before approval).
2. **New environment** → name it exactly `testpypi`.
   - No reviewers are required here (the dry-run must run unattended). Its only
     job is to make the `testpypi` **environment claim** available to the OIDC
     token so the TestPyPI trusted publisher can be env-scoped (see the note
     above). GitHub auto-creates a referenced environment with no protection
     rules, but creating it explicitly keeps the trusted-publisher scoping
     unambiguous.
   - **Restrict deployment branches** (recommended hardening): limit the
     `testpypi` environment to `main` + release tags once dry-run testing from
     feature branches is done. Without it, any write-access user can
     `workflow_dispatch` from any branch and OIDC-publish that branch's bytes
     to TestPyPI.
3. **Repo → Settings → Actions → General:** enable **"Allow GitHub Actions to
   create and approve pull requests"** — the `dev-bump` job's
   `create-pull-request` action cannot open the post-release PR without it.

> The D-06 dry-run reaches `testpypi-publish` but is event-gated OUT of
> `publish-pypi` (`if: github.event_name == 'push'`), so it never prompts for the
> `release` approval — the dry-run works even before reviewers are configured.

## 2. PyPI trusted publisher for `cement` (real publish)

**Location:** PyPI → **Manage → `cement` → Publishing** (Trusted Publishers).
(For a brand-new project name, use PyPI → **Publishing → Add a pending
publisher**.)

**Why:** OIDC trusted publishing means **no long-lived PyPI API token** lives in
the repo (roadmap-locked). The `publish-pypi` job authenticates via `id-token:
write` + the `release` environment claim.

**Configure a GitHub Actions trusted publisher with these EXACT values:**
- **Owner / repository:** `datafolklabs/cement`
- **Workflow filename:** `release.yml`  ← filename only, not a path
- **Environment name:** `release`

Any drift on repo, workflow filename `release.yml`, or environment `release`
causes an `invalid-publisher` rejection at upload (Pitfall 1). The publish must
stay in `release.yml` itself (not a reusable/called workflow), or the effective
workflow reference changes and the claim mismatches.

## 3. TestPyPI trusted publisher (REQUIRED for the D-06 dry-run)

**Location:** TestPyPI → **Manage → `cement` → Publishing** (or **Add a pending
publisher** for a new name) at `https://test.pypi.org`.

**Why:** The dry-run's `testpypi-publish` job uploads to TestPyPI via OIDC. This
is the ONE external dependency the CI-04 dry-run strictly needs.

**Configure a GitHub Actions trusted publisher with these EXACT values:**
- **Owner / repository:** `datafolklabs/cement`
- **Workflow filename:** `release.yml`
- **Environment name:** `testpypi`  ← must match `release.yml`'s
  `testpypi-publish` job, OR leave the environment field blank (both accepted;
  a *different* env name fails — Pitfall 1)

`skip-existing: true` on the job makes re-running the dry-run for an
already-uploaded version idempotent (no hard failure on a duplicate).

> **Stale-upload caveat:** within a dev cycle, only the *first* dry-run uploads
> fresh bytes for a given version number (`skip-existing` + immutable
> filenames); later dry-runs smoke-test that original upload, not the current
> commit. A true re-validation requires a version bump or waiting for the next
> dev cycle.

## 4. Docker Hub access token → repo secrets

**Location (store secrets):** GitHub → `datafolklabs/cement` → **Settings →
Secrets and variables → Actions → New repository secret**.

**Why:** Docker Hub has **no OIDC** (D-11), so the multi-arch `docker` job logs
in with an access token — **never the account password** (T-05.4-13/18).

**Token model (verified against Docker docs, 2026-07):** Docker Hub has two
token types, and only one supports repository scoping:

- **Personal Access Tokens (PATs)** — Docker Home → avatar → **Account
  settings → Personal access tokens**. Permission levels are Read / Write /
  Delete, but they apply to **every repository the account can access** — PATs
  CANNOT be scoped to a single repository.
- **Organization Access Tokens (OATs)** — Docker Home → select the
  `datafolklabs` organization → **Identity & auth → Access tokens → Generate
  access token**. OATs CAN be scoped to specific repositories (e.g. Image Push
  on `datafolklabs/cement` only), but require a **Docker Team or Business
  subscription** plus org-owner permissions. When logging in with an OAT, the
  username is the **organization name** (`datafolklabs`), not a personal
  username.

**Steps (pick ONE token path):**
1. **Preferred — OAT (requires Team/Business plan):** mint an org access token
   scoped to `datafolklabs/cement` with Image Push permission. Set
   `DOCKERHUB_USERNAME` = `datafolklabs`.
2. **Free-plan alternative — dedicated bot account:** create a machine account
   (e.g. `datafolklabsci`), add it to the org with write access to ONLY
   `datafolklabs/cement` (team permissions), then mint a Read & Write PAT under
   that account. Repo scoping is achieved via the account's permissions, not
   the token. Set `DOCKERHUB_USERNAME` = the bot username.
3. **Accepted-tradeoff fallback:** a Read & Write PAT under a maintainer
   account with push rights. The token reaches every repo that account can
   write to — revocable and better than a password, but NOT repo-scoped.
4. Add two **repository** secrets with these EXACT names:
   - `DOCKERHUB_USERNAME` — per the token path chosen above.
   - `DOCKERHUB_TOKEN` — the access token (NOT the password).

The token is exercised only by the real-publish `docker` job — it is **not**
needed for the D-06 dry-run and may be deferred until just before the first real
tag push.

## 5. Read the Docs — moving-tag docs model (`3.0`)

**The model:** RTD publishes docs **per minor line, not per patch release**.
The live version is `3.0` (serving `/en/3.0/`), built from the **moving git
tag `3.0`**. Historically each release manually re-pointed the moving tags
`3` and `3.0`; the automated pipeline now does it via the `tag-sync` job in
`release.yml`, which force-pushes both tags to the released commit after the
publish approval. The `3.0` tag-push event reaches RTD through its GitHub
integration and triggers a rebuild of `/en/3.0/` — RTD builds only from
`3.0`; `3` is maintained as repo convention, not a docs source. (Docker's
`3`/`3.0` are *image* tags, a separate namespace.)

**Provisioning requirements:**

1. **GitHub integration connected** — RTD project → Admin → Integrations (or
   the Read the Docs GitHub App on `datafolklabs/cement`). If the webhook path
   is dead, the rebuild never fires. (Fallback: an explicit RTD API build
   trigger with an RTD token — A3; not wired by default.)
2. **Tag rules must allow the Actions bot to force-update `3` and `3.0`** — the
   `tag-sync` job force-pushes both existing tags. If a GitHub tag
   ruleset/protection covers them, grant the bot bypass or the job fails.
3. **Do NOT add a per-patch activation rule.** An automation rule matching
   three-part tags (`3.0.16`, `3.0.17`, …) would activate a new public doc
   version per patch release — contrary to the major.minor scheme. No
   automation rule is needed for the `3.0` line at all: the version already
   exists and is active; rebuilds ride on the tag push.
4. **Optional future-proofing (new minor line):** when the `3.2` line someday
   starts, its `3.2` moving tag must be activated on RTD once — either
   manually (Versions page) or via an automation rule: Version type **Tag**,
   Custom match `^\d+\.\d+$`, Action **Activate version**. The rule only fires
   on newly-created refs, so legacy tags (`2.99`, `3`) are unaffected.

**Verify:** this path cannot be exercised by the D-06 dry-run (no tag push).
On the first real release, confirm RTD kicks off a `3.0` build within a minute
of the `tag-sync` job completing.

## 6. `stable/3.0.x` fast-forward (one-time reconcile + protection allowance)

**What the workflow does:** the `branch-sync` job runs
`git push origin "<release-sha>:refs/heads/stable/3.0.x"` as
`github-actions[bot]` (the job's `GITHUB_TOKEN`, scoped `contents: write`).
This is a **plain push, not a force push** — git rejects it as
non-fast-forward unless the released commit **descends from the current tip
of `stable/3.0.x`**. That rejection is intentional (Pitfall 6): the branch
only ever moves forward along released history.

**6a. Branch protection — verified 2026-07-12, nothing to do today:**

- Classic branch protection on `stable/3.0.x`: **none** (API returns "Branch
  not protected").
- Repo rulesets: one ruleset ("Default") exists and is **disabled**.

So the bot can push today. This item only becomes work if protection is added
later: a ruleset covering `stable/*` with push restrictions, required PRs, or
required status checks would break `branch-sync`. In that case add the
**GitHub Actions** app to the ruleset's **bypass list** (Settings → Rules →
Rulesets → edit rule → Bypass list), or exclude `stable/*` from the rule.

**6b. ⚠ One-time reconcile REQUIRED before the first release —
the fast-forward WILL fail today:**

Verified 2026-07-12: `stable/3.0.x` (tip `0679bf51`) is **not an ancestor of
`main`** — the branch carries three content commits `main` never took
(`997b5b64` "Fix RTD Build", `88094753` "Fix RTD: Attempting to Force Build",
`70548f8d` "Resolve Merge Conflict"), all touching only
`.readthedocs.yaml`/`readthedocs.yml` in ways long superseded on `main`.
Until reconciled, `branch-sync` fails on every release.

**Recommended fix — record ancestry with an `-s ours` merge on `main`
(zero content change, no force push, preserves stable's meaning):**

```bash
git fetch origin main stable/3.0.x
git switch main && git pull --ff-only origin main
git merge -s ours origin/stable/3.0.x \
  -m "chore: record stable/3.0.x ancestry for release fast-forward"
# verify NOTHING changed content-wise (must print no output):
git diff HEAD^ HEAD
git push origin main
# verify the precondition now holds:
git merge-base --is-ancestor origin/stable/3.0.x main && echo FF-OK
```

`-s ours` creates a merge commit whose tree is identical to `main` — it takes
no content from `stable/3.0.x`, it only records the ancestry link so every
future release commit descends from stable's tip. `stable/3.0.x` itself is
not touched and keeps pointing at the 3.0.14-era tip until the first
automated release fast-forwards it.

**Alternative (mirror semantics, loses the 3 obsolete commits):** on release
day, force-reset the branch to the released commit once —
`git push --force origin <release-sha>:refs/heads/stable/3.0.x` — after
which every subsequent release fast-forwards cleanly. Do NOT do this before
release day: it would point "stable" at unreleased dev code.

**6c. Ongoing invariant:** never commit directly to `stable/3.0.x` again —
any commit landed there (and not on `main`) re-creates the divergence and
breaks the next release's fast-forward. Pre-tag verification (also useful in
Phase 6):

```bash
git merge-base --is-ancestor origin/stable/3.0.x <release-commit> && echo FF-OK
```

## Pre-first-tag checklist

Complete every box before pushing the first real `MAJOR.MINOR.PATCH` tag. Boxes
marked **(dry-run)** are the only ones the CI-04 `workflow_dispatch` dry-run
strictly needs.

- [ ] **(dry-run)** GitHub Environment `testpypi` exists (Settings → Environments)
- [ ] **(dry-run)** TestPyPI trusted publisher exists — repo `datafolklabs/cement`, workflow `release.yml`, env `testpypi` (or blank)
- [ ] GitHub Environment `release` exists **with required reviewers** (Settings → Environments)
- [ ] PyPI trusted publisher exists — repo `datafolklabs/cement`, workflow `release.yml`, env `release`
- [ ] Docker Hub access token minted with push rights to `datafolklabs/cement` (OAT / bot-account PAT / maintainer PAT — see section 4)
- [ ] Repo secret `DOCKERHUB_USERNAME` set (Settings → Secrets and variables → Actions)
- [ ] Repo secret `DOCKERHUB_TOKEN` set (access token, not the account password)
- [ ] Read the Docs GitHub integration connected; tag rules allow the Actions bot to force-update tags `3` and `3.0` (see section 5 — no per-patch activation rule)
- [ ] One-time `-s ours` ancestry merge landed on `main` (section 6b — `stable/3.0.x` is diverged today and `branch-sync` WILL fail without it)
- [ ] No ruleset/protection blocks the Actions bot pushing `stable/3.0.x` (none exists as of 2026-07-12 — re-check if rules were added; section 6a)
- [ ] Pre-tag check passes: `git merge-base --is-ancestor origin/stable/3.0.x <release-commit>` (A4; section 6c)
- [ ] Repo setting **"Allow GitHub Actions to create and approve pull requests"** enabled (Settings → Actions → General — required by the `dev-bump` job)
- [ ] `testpypi` environment deployment branches restricted to `main` + tags (section 1 hardening — after feature-branch dry-run testing is done)
- [ ] `CHANGELOG.md` has a finalized `## <version>` section (NOT `- DEVELOPMENT`) — preflight (D-02) hard-fails otherwise (dry-runs tolerate the DEVELOPMENT marker; only real tag pushes require finalization)
- [ ] Package version (`cement.core.backend.VERSION`) equals the tag being pushed — preflight hard-fails otherwise

## Known accepted tradeoffs

The maintainer signs off on these knowingly before the first release.

1. **Dev-bump PR runs no CI (Pitfall 4, accepted for the first release).** The
   post-release `dev-bump` job opens the "bump to 3.0.17" PR authored by
   `GITHUB_TOKEN`. GitHub's anti-recursion rule means a `GITHUB_TOKEN`-authored PR
   does **not** trigger `on: pull_request` CI, so that PR shows zero checks. This
   is accepted: the bump is trivial and the maintainer merges it (main CI runs on
   merge). If CI-on-bump-PR becomes required later, author the PR with a GitHub
   App token (`actions/create-github-app-token`) + `APP_ID`/`APP_PRIVATE_KEY`
   secrets (A5).

2. **Exact-tag action pins, not full-commit-SHA pins (supply-chain posture).**
   Every action is pinned to an exact release tag (e.g. `actions/checkout@v7.0.0`,
   `pypa/gh-action-pypi-publish@v1.14.0`), matching the repo's existing CI-05
   AUDIT-POINT convention. The strictly harder posture is full-commit-SHA pinning
   + Dependabot (recommended by the `gh-action-pypi-publish` maintainers). We stay
   on exact tags to match repo convention, but the maintainer **may elect to
   tighten the community-authored actions to full-SHA pins** later. The
   community (non-`actions/`/`pypa/`/`docker/`/`pdm-project/`) actions in use are:
   `peter-evans/create-pull-request@v8.1.1`, `hoverkraft-tech/compose-action@v3.0.0`,
   and `ConorMacBride/install-package@v1.1.0`.
