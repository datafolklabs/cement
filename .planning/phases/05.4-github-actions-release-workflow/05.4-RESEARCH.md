# Phase 05.4: GitHub Actions Release Workflow - Research

**Researched:** 2026-07-12
**Domain:** CI/CD ŌĆö GitHub Actions release automation (PyPI OIDC publishing, Docker multi-arch, reusable workflows, GitHub Environments/approvals)
**Confidence:** HIGH

## Summary

This phase automates issue #791's manual release checklist as a tag-triggered
GitHub Actions workflow. The technical domain is well-trodden and served almost
entirely by first-party, blessed actions ŌĆö `pypa/gh-action-pypi-publish` for
OIDC trusted publishing, the `docker/*` action family for multi-arch buildx
images, GitHub Environments for the single human approval, and `workflow_call`
reusable workflows so PR CI and release CI share one gate definition (D-13).
There is very little to hand-roll; the work is composition, correct permission
scoping, and getting the OIDC + Environments + reusable-workflow interaction
exactly right.

The CONTEXT.md decisions (D-01..D-16) lock nearly every architectural choice.
Research therefore focuses on the *how* of the locked decisions and on the
handful of items left to Claude's discretion (job graph, action pins, RTD
trigger mechanics, native-OS smoke-test adaptation, concurrency, token
scoping, GitHub Release body extraction). The dominant risks are all
integration-shaped: OIDC environment-claim scoping requires the publish job
itself to declare `environment: release`; a naive "separate approval job"
design produces the env claim in the wrong job or forces two approvals;
`GITHUB_TOKEN`-created PRs (the D-12 dev-bump) do not trigger `on: pull_request`
CI; and the existing docker-driven `cli-smoke-test.sh` cannot run on
macOS/Windows runners and must be re-expressed natively.

**Primary recommendation:** Build two files ŌĆö a reusable `gates.yml`
(`on: workflow_call`) containing every gate job (comply, test, matrix, Linux
docker smoke, plus the NEW Windows-core and macOS/Windows native smoke jobs),
and a `release.yml` (`on: push` tags + `workflow_dispatch`) that calls
`gates.yml`, then runs preflight ŌåÆ build (isolated job) ŌåÆ TestPyPI publish +
install-smoke matrix ŌåÆ a SINGLE `publish-pypi` job carrying
`environment: release` (the one approval stop, which also produces the OIDC
env claim) ŌåÆ fan-out to Docker Hub, `stable/3.0.x` fast-forward, GitHub Release,
RTD trigger, and the dev-bump PR. Refactor `build_and_test.yml` to a thin
`pull_request` caller of `gates.yml`. Pin every action to an exact tag per
CI-05.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Tag push is the canonical release trigger. Pushing tag `3.0.16`
  starts the run: gates ŌåÆ TestPyPI validation ŌåÆ approval ŌåÆ publish. A failed
  gate means delete/re-tag.
- **D-02:** Fast pre-flight job verifies (a) tag exactly matches package version
  (`cement.utils.version:get_version` / `pdm show`) and (b) `CHANGELOG.md` has a
  finalized section for the version (not `- DEVELOPMENT`).
- **D-03:** Human approval gate is a GitHub Environment (`release`) with required
  reviewers on the publish side. PyPI OIDC trusted-publishing scoping attaches to
  this environment.
- **D-04:** Exactly ONE approval stop. After gates + TestPyPI smoke pass, a single
  approval unlocks the entire publish side: PyPI, Docker Hub, branch sync, GitHub
  Release, docs trigger, post-release outputs.
- **D-05:** TestPyPI is an inline stage of every release run ŌĆö gates ŌåÆ build
  sdist+wheel ŌåÆ upload to TestPyPI ŌåÆ install smoke ŌåÆ approval ŌåÆ PyPI.
- **D-06:** `workflow_dispatch` is a first-class DRY-RUN mode, runnable from any
  branch before a tag exists: pre-flight + gates + build + TestPyPI + smoke,
  ending in a summary. Never reaches approval/publish. Tag push is the ONLY path
  to PyPI. (This is how Phase 6 validates the workflow ŌĆö CI-04.)
- **D-07:** TestPyPI install smoke is a matrix over all 5 CPythons (3.10ŌĆō3.14):
  `pip install cement==X.Y.Z` from TestPyPI + import + minimal `App().run()`.
- **D-08:** Uploads use `--skip-existing` semantics: if the version already exists
  on TestPyPI, skip the upload but still run install smoke against the existing
  artifact. Re-runs are idempotent.
- **D-09:** Tags stay unprefixed (`3.0.16`). Trigger on `3.*`-style tags with a
  strict semver guard in pre-flight.
- **D-10:** Workflow fast-forwards the existing `stable/3.0.x` branch to the
  release tag. Repo convention is `stable/X.Y.x` (create if a future minor lacks
  one).
- **D-11:** Docker Hub push set: `datafolklabs/cement:3`, `:3.0`, `:3.0.16`, AND
  `:latest`. All multi-arch (amd64 + arm64) via buildx.
- **D-12:** After publish, open an automated PR against `main` bumping
  `cement/utils/version.py` VERSION to `3.0.17` and adding a fresh
  `## 3.0.17 - DEVELOPMENT` changelog section. No direct push to main; user merges.
- **D-13:** Gate jobs (comply, test, Python matrix, cli-smoke-test) refactored
  into a reusable `workflow_call` workflow. Both PR CI (`build_and_test.yml`) and
  release invoke the same definition.
- **D-14:** NEW OS jobs ŌĆö Windows core tests (`make test-core`-shaped) and
  macOS/Windows CLI smoke tests ŌĆö join the shared gates and run on every PR too.
- **D-15:** Notifications (mailing list, Slack) stay HUMAN steps this phase ŌĆö no
  webhook secrets in the release path.
- **D-16:** The post-release human checklist is emitted as an auto-created GitHub
  issue ("3.0.16 post-release checklist") with checkboxes, assigned to the
  maintainer, and echoed in the job summary.

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `softprops/action-gh-release` | `gh release create` (built-in CLI) | `gh` is zero-dependency and already available on runners; softprops gives declarative `body_path`/`files`. Either is fine ŌĆö `gh` avoids a third-party action (supply-chain), softprops is more ergonomic. **Recommend `gh` for zero third-party surface** unless attaching many assets. |
| `peter-evans/create-pull-request` | `gh pr create` after a scripted branch push | `gh pr create` avoids a third-party action but you hand-roll the branch/commit/push; peter-evans handles idempotent update-if-exists. For the trivial dev-bump, either works. |
| `docker/metadata-action` | Hand-write the 4 tag lines in `build-push-action` `tags:` | Tags are static and known (`:3`,`:3.0`,`:3.0.16`,`:latest`); a literal multiline `tags:` is simpler and fully transparent. **Recommend literal tags** ŌĆö metadata-action is overkill for a fixed set. |
| `pypa/gh-action-pypi-publish` env-scoped OIDC | Long-lived PyPI API token secret | OIDC is roadmap-locked ("no credentials stored as long-lived secrets where OIDC available"). Token is the fallback only if trusted publishing cannot be configured. |

**Packages flagged as suspicious [SUS]:** none.

**Supply-chain hardening note:** CI-05 requires exact-tag pins (e.g.
`checkout@v7.0.0`), matching the repo's existing convention. The strictly harder
posture is full-commit-SHA pinning + Dependabot (recommended by the
`gh-action-pypi-publish` maintainers). **Recommend staying with exact tags** to
match the established repo convention and AUDIT POINT discipline, but call it out
at plan time as a security tradeoff the maintainer may elect to tighten for the
two community actions if they are kept.

### Pattern 1: Reusable gates via `workflow_call` (D-13)

**What:** One workflow file defines every gate as jobs; both PR CI and release
CI call it with `uses:`.
**When to use:** Any time two triggers must run byte-identical checks.
**Example:**
```yaml
# .github/workflows/gates.yml

on:
  workflow_call:
    inputs:
      full-os-matrix:
        description: "Run Windows/macOS OS jobs (always true for release; true on PR per D-14)"
        type: boolean
        default: true
env:
  SMTP_HOST: localhost
  SMTP_PORT: 1025
  MEMCACHED_HOST: localhost
  REDIS_HOST: localhost
jobs:
  comply:      { runs-on: ubuntu-latest, steps: [ ... make comply ... ] }
  test:        { needs: comply, ... make test ... }
  test-all:    { needs: test, strategy: { matrix: { python-version: [...] } }, ... }
  test-core-windows:   # NEW (D-14) ŌĆö see Pitfall 5 for make-on-Windows
    runs-on: windows-latest

        with: { python-version: "3.x" }
      - uses: pdm-project/setup-pdm@v4.5
      - run: pdm install
      - run: pdm run pytest --cov=cement.core tests/core   # NOT `make test-core`
  cli-smoke-native:    # NEW (D-14)
    strategy: { matrix: { os: [macos-latest, windows-latest], python-version: ["3.10","3.14"] } }
    runs-on: ${{ matrix.os }}
```

### Pattern 2: Single-approval env-scoped OIDC publish (D-03/D-04)

**What:** The `publish-pypi` job itself declares `environment: release`. That
declaration (a) pauses for required-reviewer approval ŌĆö the ONE stop ŌĆö and (b)
puts `environment` into the OIDC token so PyPI's environment-scoped trusted
publisher matches.
**When to use:** Env-scoped trusted publishing with exactly one human gate.
**Example:**
```yaml
  publish-pypi:
    needs: [testpypi-smoke]
    if: github.event_name == 'push'          # dispatch never reaches here (D-06)
    runs-on: ubuntu-latest
    environment: release                     # ŌÅĖ approval + OIDC env claim (D-03/D-04)
    permissions:
      id-token: write                        # OIDC ŌĆö no username/password
    steps:
      - uses: actions/download-artifact@v8.0.1
        with: { name: dist, path: dist }
      - uses: pypa/gh-action-pypi-publish@v1.14.0
        # no repository-url ŌåÆ defaults to real PyPI; no skip-existing ŌåÆ fail loud
  docker:
    needs: publish-pypi                       # everything fans out AFTER the one approval
    ...
```
**Anti-pattern (do NOT do):** a separate no-op `approval` job on the environment
followed by an ungated `publish-pypi` ŌĆö the OIDC env claim would then be absent
from the publish job and env-scoped trusted publishing fails; OR putting
`environment: release` on *multiple* downstream jobs, which forces multiple
approvals (violates D-04).

### Anti-Patterns to Avoid

- **Publishing from inside a reusable workflow.** Keep OIDC publish jobs in the
  top-level `release.yml`. Trusted-publisher matching keys on the *entry-point*
  workflow filename; publishing from a called workflow complicates the claim
  and has known attestation edge cases (gh-action-pypi-publish #283). Only the
  *gates* are reusable ŌĆö never the publish.
- **`skip-existing: true` on the real PyPI upload.** Use it ONLY for TestPyPI.
  On PyPI, a duplicate version must fail loudly (re-tagging the same version is a
  release error you want surfaced).
- **`make` targets on Windows runners.** `make test-core`/`make comply` assume
  GNU make + a POSIX shell; invoke `pdm run pytest ...` / `pdm run ruff`/`mypy`
  directly (Pitfall 5).
- **Cancelling in-progress releases.** Set `cancel-in-progress: false` ŌĆö never
  interrupt a run mid-publish.

## Common Pitfalls

### Pitfall 1: OIDC environment claim missing / trusted-publisher mismatch
**What goes wrong:** PyPI rejects the upload with an OIDC/trusted-publisher error
even though `id-token: write` is set.
**Why it happens:** The trusted publisher on PyPI is configured with an
`environment` constraint, but the publish job does not declare
`environment: release`, so the OIDC token lacks the environment claim. OR the
trusted publisher's *workflow filename* does not match `release.yml`. OR publish
was attempted from a reusable (called) workflow, changing the effective workflow
reference.
**How to avoid:** Publish job declares `environment: release`; keep publish in
`release.yml` (not a reusable workflow); configure the PyPI/TestPyPI trusted
publishers with the exact repo, workflow filename `release.yml`, and env name.
**Warning signs:** `invalid-publisher` / `trusted publisher` errors on upload.

### Pitfall 2: TestPyPI install can't resolve `cement[cli]` dependencies
**What goes wrong:** `pip install cement==3.0.16` from TestPyPI fails resolving
`jinja2`/`pyYaml` (the `[cli]` extra deps), which live on real PyPI, not TestPyPI.
**Why it happens:** `-i https://test.pypi.org/simple/` makes TestPyPI the ONLY
index. cement core has zero deps, but the smoke's minimal `App().run()` may not
need the extra ŌĆö however if the smoke installs `cement[cli]` it needs jinja2/pyyaml.
**How to avoid:** For the D-07 smoke, install plain `cement==X.Y.Z` (zero deps ŌĆö
resolves cleanly from TestPyPI alone) for the core round-trip; if the `[cli]`
surface is smoked, add `--extra-index-url https://pypi.org/simple/`.
**Warning signs:** `Could not find a version that satisfies the requirement jinja2`.

### Pitfall 3: Native macOS/Windows smoke can't use the docker script
**What goes wrong:** Reusing `scripts/cli-smoke-test.sh` on macOS/Windows runners
fails ŌĆö macOS runners have no Docker daemon; Windows runners default to Windows
containers; and the script greps `Platform Linux.*`.
**Why it happens:** The existing script drives `docker compose` + `docker exec`
into `python:$pyver` Linux containers. That model is Linux-only.
**How to avoid:** Write a native, cross-platform smoke (recommend a single
`scripts/cli-smoke-native.py` driven by `subprocess`) that: builds/installs the
wheel into the runner's own Python, runs `cement --version`/`--help`/`generate
project`/installs the generated project/checks its `--version`, and asserts a
platform-agnostic banner (accept `Linux`/`Darwin`/`Windows`). Keep the heavy
"generated project `make comply && make test`" depth on the LINUX docker smoke
only; the macOS/Windows native smoke is a lighter CLI-invocation + generate +
install-generated check (macOS/Windows lack the make+services depth). Tier
explicitly.
**Warning signs:** `docker: command not found` / `Cannot connect to the Docker
daemon` on macOS; `Platform Linux` grep fails on Darwin/Windows.

### Pitfall 4: The D-12 dev-bump PR doesn't trigger CI
**What goes wrong:** The auto-created "bump to 3.0.17" PR shows no PR checks.
**Why it happens:** Events triggered by `GITHUB_TOKEN` (including PRs opened by
`create-pull-request` or `gh` using the default token) do NOT trigger further
workflow runs ŌĆö a deliberate GitHub anti-recursion rule.
**How to avoid:** Accept it (the bump is trivial; the maintainer merges it per
D-12 and `main` CI runs on merge) OR author the PR with a GitHub App token
(`actions/create-github-app-token@v3.2.0`) / PAT so `on: pull_request` fires.
**Recommendation:** Accept no-CI on the trivial bump PR for the first release;
revisit if CI-on-bump-PR is desired. Flag as a decision point.
**Warning signs:** New PR with zero checks.

### Pitfall 5: `make` targets fail on Windows runners
**What goes wrong:** `make test-core` / `make comply` error on `windows-latest`.
**Why it happens:** GNU make and the POSIX shell assumed by the Makefile are not
reliably present/consistent on Windows runners; recipes use bash-isms.
**How to avoid:** On Windows/macOS jobs, invoke the underlying tools directly:
`pdm run pytest --cov=cement.core tests/core`, `pdm run ruff check`, `pdm run
mypy`. Reserve `make` for Linux jobs.
**Warning signs:** `make: command not found` / recipe shell errors on Windows.

### Pitfall 6: `stable/3.0.x` fast-forward blocked by branch protection
**What goes wrong:** The `branch-sync` push to `stable/3.0.x` is rejected.
**Why it happens:** Branch protection on `stable/*`, or `GITHUB_TOKEN` lacking
`contents: write`, or the tagged commit is not a fast-forward descendant of
`stable/3.0.x`.
**How to avoid:** Grant the job `permissions: contents: write`; ensure the
release tag descends from `stable/3.0.x`'s tip (fast-forward only ŌĆö `git push
origin <sha>:refs/heads/stable/3.0.x` or `git merge --ff-only`); confirm branch
protection allows the Actions bot (or the maintainer approves). Document as a
provisioning checkpoint.
**Warning signs:** `! [remote rejected] ... (protected branch)` /
`non-fast-forward`.

### Pitfall 7: Accidental trigger from moving git tags `3` / `3.0`
**What goes wrong:** The repo has bare moving tags `3` and `3.0` (git tags, seen
in `git tag`); a loose trigger pattern (`3.*`) could fire the release on those.
**Why it happens:** `tags: ['3.*']` matches `3.0` and `3`ŌĆ” (well `3` has no dot).
**How to avoid:** Use a strict three-part numeric pattern
`tags: ['[0-9]+.[0-9]+.[0-9]+']` (matches `3.0.16`, not `3` or `3.0`), and keep
the semver guard in preflight (D-02/D-09) as defense-in-depth.
**Warning signs:** Release run kicked off by a non-release tag.

## Code Examples

### Preflight: tag==version + finalized changelog (D-02/D-09)
```yaml
  preflight:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7.0.0
      - uses: actions/setup-python@v6.3.0
        with: { python-version: "3.x" }
      - uses: pdm-project/setup-pdm@v4.5
      - name: Resolve version + guards
        run: |
          set -euo pipefail
          TAG="${{ github.ref_name }}"                       # e.g. 3.0.16 (empty on dispatch)
          # For dispatch dry-run, derive from the package instead of the tag:
          PKG=$(pdm run python -c "from cement.utils.version import get_version; print(get_version())")
          if [ "${{ github.event_name }}" = "push" ]; then
            [[ "$TAG" =~ ^[0-9]+\.[0-9]+\.[0-9]+$ ]] || { echo "::error::tag '$TAG' not strict semver"; exit 1; }
            [ "$TAG" = "$PKG" ] || { echo "::error::tag $TAG != package version $PKG"; exit 1; }
            VER="$TAG"
          else
            VER="$PKG"
          fi
          # CHANGELOG must have a finalized section, not "- DEVELOPMENT"
          grep -qE "^## +${VER}( |$)" CHANGELOG.md || { echo "::error::no '## $VER' in CHANGELOG"; exit 1; }
          if grep -qE "^## +${VER}.*DEVELOPMENT" CHANGELOG.md; then
            echo "::error::CHANGELOG $VER still marked DEVELOPMENT"; exit 1
          fi
          echo "version=$VER" >> "$GITHUB_OUTPUT"
```
Source: adapted from repo `cement/utils/version.py:get_version` + `pyproject.toml`
`[tool.pdm.version]` getter (VERIFIED in repo).

## Open Questions

1. **RTD trigger: webhook-automatic vs explicit API call?**
   - What we know: RTD auto-creates and builds new tags/versions on webhook when
     an automation rule activates PEP 440 versions > current stable.
   - What's unclear: whether cement's RTD project has such a rule enabled today.
   - Recommendation: verify the RTD project's automation rules; if a
     "activate + build on new semver tag" rule exists, the workflow does NOTHING
     for RTD (webhook handles it). If not, add either the rule (preferred, no
     secret) or an explicit RTD API build step (needs an RTD token secret).

2. **Keep the isolated build job, or build inside each publish job?**
   - What we know: separating build from publish is a supply-chain best practice
     (a compromised test/build step can't reach publish creds).
   - What's unclear: minor complexity of artifact upload/download vs benefit.
   - Recommendation: keep the isolated `build` job + upload/download-artifact ŌĆö
     it also lets TestPyPI and PyPI publish the identical bytes (D-05 "approved
     artifacts are index-install-proven").

3. **Serial vs parallel gate graph (Phase 02 D-16 carry-over).**
   - What we know: PR CI keeps a SERIAL fail-fast chain (complyŌåÆtestŌåÆmatrixŌåÆsmoke).
   - What's unclear: whether release wants the same seriality (slower but cheaper
     on failure) or parallel gates (faster feedback).
   - Recommendation: carry the serial chain into `gates.yml` per D-16; the NEW OS
     jobs can run in parallel with the matrix (they're independent) without
     violating the fail-fast spirit.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |

### Known Threat Patterns for GitHub Actions release pipelines

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Malicious/compromised third-party action | Tampering / Elevation | Exact-tag (or SHA) pins; prefer first-party `actions/`,`pypa/`,`docker/`; minimal community actions |
| Credential exfiltration from build step | Info Disclosure | Separate build job (no publish creds) from publish job; OIDC short-lived token; no secrets in build/test env |
| Unauthorized publish (no human review) | Elevation of Privilege | Environment `release` required reviewers ŌĆö the single approval (D-04) |
| Privilege escalation via reusable workflow | Elevation | Keep OIDC publish in top-level `release.yml`; gates-only are reusable; `secrets: inherit` only where needed |
| Accidental publish from wrong ref/tag | Tampering | Strict tag pattern (Pitfall 7) + preflight tag==version semver guard (D-02) |
| `pull_request_target`/untrusted PR code with secrets | Elevation | Release triggers on `push` tags + `workflow_dispatch` only ŌĆö never `pull_request_target`; gates run on `pull_request` (no secrets needed) |
| Dependency confusion on TestPyPI smoke | Tampering | Install plain `cement` (zero deps) from TestPyPI; only add `--extra-index-url` real PyPI when the `[cli]` extra is required (Pitfall 2) |
