**Researched:** 2026-04-30

## Summary

The CONTEXT.md decisions D-03 and D-04 are technically correct. The only
new file this phase ships is
`cement/cli/templates/generate/project/pyproject.toml` containing a single
`[build-system]` table with three pinned requires (`setuptools`, `wheel`,
`cement>={{ cement.version }}`) and `build-backend = "setuptools.build_meta"`.
Setuptools' build backend will see the `[build-system]` table for build-
isolation purposes, find no `[project]` table, and fall back to reading
project metadata from the existing `setup.py` — exactly the dual-source
shape D-04 mandates and that current setuptools officially supports.

`{{ cement.version }}` resolves at generate-time to the live cement
`get_version()` string (verified in `cement/ext/ext_generate.py` line 38).
With cement currently `(3, 0, 15, 'final', 0)`, that string is `"3.0.15"`
— a clean PEP 440 final release identifier. The dev-changeset form
(`.devNNNNNN`) only fires when `VERSION[3] == 'alpha'` AND `VERSION[4] == 0`
— not a current concern, and the floor-pin `>=` semantics handle it
correctly even if it ever fires (PEP 440 dev releases are considered
prereleases for matching, but a user who hits `pip install` on a
freshly-generated project will satisfy the constraint trivially against
any released cement on PyPI).

**Primary recommendation:** Land exactly the file shape D-03 specifies.
Add a 1-2 line comment header explaining the band-aid intent. Do not
deviate. Verify with `bash scripts/cli-smoke-test.sh` on Python 3.10–3.14.

### Pattern 1: Minimum-Surface PEP 517 Restoration

**What:** Add `[build-system]` to a legacy `setup.py`-only project to
restore install-via-pip behavior under PEP 517 build isolation, without
moving project metadata into `[project]`.

**When to use:** The exact situation this phase addresses — a setup.py
that imports a package not available in the default build-isolation venv.
Fixing requires either (a) declaring the missing package in
`build-system.requires`, or (b) refactoring setup.py to remove the
top-level import. Option (a) is D-03; option (b) is the deferred long-term
fix.

**Example (the entire file this phase will create):**

```toml
# Build-system contract for pip ≥21.3 PEP 517 isolation. cement is
# declared as a build-time dependency because setup.py imports it
# transitively via {{ label }}/core/version.py. See cement issue tracker
# for the long-term plan to remove cement from build-time deps.

[build-system]
requires = ["setuptools", "wheel", "cement>={{ cement.version }}"]
build-backend = "setuptools.build_meta"
```

`[CITED: https://setuptools.pypa.io/en/latest/build_meta.html]`
`[CITED: https://packaging.python.org/en/latest/specifications/pyproject-toml/]`

### Pattern 2: Setuptools Backend with setup.py-Sourced Metadata

**What:** When `pyproject.toml` declares `[build-system]` but no
`[project]` table, setuptools' build backend reads project metadata from
`setup.py` (or `setup.cfg`) as a fallback.

**When to use:** Transitional state — the project wants PEP 517 build
isolation but is not yet ready to migrate metadata to `[project]`.

**Verification:**
`[CITED: https://setuptools.pypa.io/en/latest/userguide/pyproject_config.html]`
"The setuptools build backend supports both the [project] table, and the
older format in setup.cfg or setup.py."

This is the exact dual-source pattern D-04 selects.

### Anti-Patterns to Avoid (this phase only)

- **Adding a `[project]` table "while we're in here"** — creates the
  dual-source-of-truth problem D-04 explicitly rejects. Both `setup.py`
  `name=`/`version=` and `[project] name=`/`version=` would specify
  metadata; setuptools' precedence rules become a source of bugs.
- **Switching to `build-backend = "setuptools.build_meta:__legacy__"`** —
  the `:__legacy__` variant exists for builds that have NO
  `pyproject.toml` at all (legacy fallback). Once we're shipping a
  `pyproject.toml`, the modern `setuptools.build_meta` is correct.
  `[CITED: https://setuptools.pypa.io/en/latest/build_meta.html]`
- **Pinning `cement` exactly (`==`) in `[build-system].requires`** —
  D-03 chooses `>=` deliberately. Exact pin would force re-rendering
  the template every cement release.
- **Modernizing the Makefile in the same commit** — D-09 / D-14 forbid.
  Even if `python setup.py develop` is deprecated, Phase 01.1 doesn't
  touch it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Build-time dependency declaration | Custom shell scripts that pre-install cement before `pip install .` | Standard `[build-system].requires` per PEP 517 | This is exactly what `[build-system].requires` exists for |
| Template variable substitution | New rendering logic | Existing `{{ cement.version }}` jinja2 placeholder (already used in `requirements.txt`) | `ext_generate.py` already populates this; reuse the established convention |
| PEP 517 build isolation override | `pip install --no-build-isolation` advice in README | Just declare the build dep correctly | `--no-build-isolation` is a workaround that breaks reproducibility — adding the dep to `[build-system].requires` is the real fix |

**Key insight:** The whole point of `[build-system].requires` is to declare
exactly what this phase needs to declare. The patch IS the canonical
mechanism, not a workaround for the absence of one.

## Common Pitfalls

### Pitfall 1: `wheel` listed in requires when it's no longer needed

**What goes wrong:** Listing `wheel` in `[build-system].requires` on a
modern setuptools (≥61) project is sometimes flagged as redundant —
modern setuptools is supposed to declare `wheel` as a transitive
build-time dep itself.

**Why it happens:** Old PEP 517 guidance (circa 2018-2020) instructed
projects to list `wheel` explicitly. Some style checkers still warn
about its absence; others now warn about its presence as redundant.

**How to avoid:** D-03 keeps `wheel` in the list. This is defensible
both ways and matches the "explicit is cheap insurance" reasoning in
the CONTEXT.md decision rationale. The generated Makefile's `dist`
target still calls `python setup.py bdist_wheel` (D-09 keeps it
unchanged), and that path benefits from `wheel` being explicitly
present.

**Warning signs:** A pip warning about deprecated direct invocation of
`setup.py bdist_wheel` may appear during `make dist` — that's
unrelated to the `[build-system].requires` choice and is owned by a
deferred phase.

`[CITED: https://setuptools.pypa.io/en/latest/userguide/pyproject_config.html]`
The setuptools own example shows `["setuptools", "setuptools-scm"]` —
no `wheel`. But pip's older recommendations and many real-world
projects still include it. Defensible either way; keep per D-03.
**Confidence: MEDIUM** (style preference, not a correctness issue).

### Pitfall 2: `{{ cement.version }}` resolves to a `.dev` form

**What goes wrong:** If cement is mid-release-cycle (`VERSION[3] ==
'alpha'`, `VERSION[4] == 0`) at the moment `cement generate project`
runs, `get_version()` returns `"3.0.16.dev20260430120000"`. Pinning
`cement>=3.0.16.dev20260430120000` then resolves at install time
against PyPI, where dev releases are excluded by default per PEP 440.

**Why it happens:** PEP 440 treats `.devN` as a developmental release.
By default, version specifiers exclude pre-releases and dev releases
unless `--pre` is passed or the constraint already targets a
pre-release.

**How to avoid:** This is **not** a current problem because
`cement/core/backend.py` says
`VERSION = (3, 0, 15, 'final', 0)` — `get_version()` returns `"3.0.15"`,
a clean final release identifier. The dev-changeset branch is unreachable
in the released cement state today.

**Edge case if it does fire:** If a downstream user runs `cement generate
project` against a dev-installed cement (e.g., editable install of an
in-flight 3.0.16-alpha), the rendered pyproject.toml would carry
`cement>=3.0.16.dev...`. When that user later does `pip install .`, pip's
build-isolation venv would attempt to satisfy `cement>=3.0.16.dev...`
against PyPI. Per PEP 440, the constraint already targets a dev release,
so dev releases are no longer excluded — it would resolve, but only if a
dev release of cement is published to PyPI. Otherwise it would resolve to
the next stable release that satisfies the floor, which is the right
behavior.

**Warning signs:** None for current users. Future maintainers should
note: if cement publishes alpha-0 to PyPI in flight (which it
historically does not), the floor pin produces a constraint that
explicitly opts in to pre-releases for the build env.

**Confidence: HIGH** (verified by reading `version.py:65-74`).

`[CITED: https://peps.python.org/pep-0440/]` —
"Pre-releases of any kind, including developmental releases, are
implicitly excluded from all version specifiers, unless they are
already present on the system, explicitly requested by the user, or if
the only available version that satisfies the version specifier is a
pre-release."

### Pitfall 3: Forgetting that the file is a template

**What goes wrong:** Writing literal cement version into the file
(e.g., `cement>=3.0.15`) instead of `cement>={{ cement.version }}`.

**Why it happens:** The file lives in
`cement/cli/templates/generate/project/` and looks like a normal
`pyproject.toml`. Easy to forget that `ext_generate.py` will jinja2-
substitute it before writing to the destination.

**How to avoid:** Every cement-version reference in the file MUST use
`{{ cement.version }}`. Match the existing convention in
`requirements.txt:1` which already does
`cement=={{ cement.version }}`.

**Warning signs:** A grep for `cement>=` should show ONLY the templated
form, no hardcoded version strings.

**Confidence: HIGH**.

### Pitfall 4: pip warning when only [build-system] is declared

**What goes wrong:** Concern that pip might emit a deprecation warning
when `pyproject.toml` exists with only `[build-system]` and no
`[project]` table.

**How to verify:** No such pip warning exists in current pip versions
(through 24.x). PEP 621 / PyPA pyproject.toml spec explicitly allows
this:
`[CITED: https://packaging.python.org/en/latest/specifications/pyproject-toml/]`
"The lack of a `[project]` table implicitly means the build backend will
dynamically provide all keys."

For setuptools' build backend, "dynamically provide" means "read from
setup.py / setup.cfg." This is supported, documented, and not deprecated.

**Confidence: HIGH** (verified against current spec and setuptools
docs).

### Known Threat Patterns for the Change Surface

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Supply-chain: `cement>={{ cement.version }}` allows future minors of cement, including a hypothetical compromised release, into a downstream user's build venv | Tampering | Floor pins are the standard PyPA practice. Tighter pins (e.g., `==`) shift the supply-chain risk to forced-stale builds, not eliminate it. Keep per D-03; this is not a bigger risk than already exists in `requirements.txt`'s `cement=={{ cement.version }}`. |
| Build venv pulls fresh setuptools / wheel from PyPI on every build | Tampering | Inherent to PEP 517. Mitigated upstream by PyPI's signing and pip's hash-checking modes (not used here). Outside this phase's scope. |
| Template injection via `{{ cement.version }}` placeholder | Tampering | The placeholder value is sourced from cement's own VERSION tuple via `get_version()` — controlled, not user-input. No user-supplied input flows into this template. Not exploitable. |

**Bottom line:** This phase introduces no new security surface beyond what
already exists in `requirements.txt`. The supply-chain shape of the
generated project is unchanged in kind — only widened by one entry
(adding cement to build-time deps in addition to runtime deps, where it
already is).
