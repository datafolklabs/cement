# Phase 01.1: Generated Project Template Build Modernization - Research

**Researched:** 2026-05-01
**Domain:** Python packaging — pdm-backend migration of a setuptools-era project
template; PEP 517 / PEP 621 / PEP 735 / PEP 639 alignment
**Confidence:** HIGH

## Summary

Phase 01.1 migrates `cement generate project`'s template from
setuptools+setup.py to pdm-backend with a complete `[project]`-table-driven
`pyproject.toml`. The strict-minimum `[build-system].requires` patch
(commit `8e7fba5e`) failed empirically because `setup.py` line 3 imports the
package being built (`from {{ label }}.core.version import get_version`),
which cannot resolve under PEP 517 build isolation regardless of build-time
deps. CONTEXT.md is locked at the full migration: pdm-backend, full
`[project]`, file-source version, Shape B `version.py`, delete legacy files,
PDM-aligned generated Makefile, smoke-test reverted to plain `pip install .`.

Research verified the load-bearing technical claims (regex pattern, package-
data auto-inclusion, PEP 735 pip support, PEP 639 deprecation posture, prior
commit shapes). Two scope-adjacent findings surfaced that CONTEXT.md does not
explicitly cover: the **template's `Dockerfile` and `README.md` reference
deleted files** (`requirements.txt`, `setup.py`). README is explicitly
deferred per CONTEXT.md `<deferred>`. Dockerfile is **not** explicitly
addressed — see Open Question 1.

**Primary recommendation:** Mirror Phase 01's atomic-per-concern commit
shape exactly. Follow the D-15 7-commit split as proposed; recommend the
"overwrite-in-place" path over revert+rewrite for the template's
pyproject.toml (single net-change, single audit point). Use the verified
default pdm-backend regex (`^(?:__version__|VERSION)\s*=\s*["'](.+?)["']\s*(?:#.*)?$`),
write Shape B with double quotes for safety, and rely on pdm-backend's
default package-data inclusion — no explicit `[tool.pdm.build]` block needed.

### Pattern 1: pdm-backend file-source version (verified)

**What:** Read version string from a Python file via regex, no exec.
**When to use:** When you want a single canonical version literal that's
readable both at build time (regex) and at import time (Python) without
running any code at build time.

**Example:**
```python
# myapp/core/version.py
__version__ = "0.0.1a0"

def get_version():
    return __version__
```

```toml
# pyproject.toml
[project]
dynamic = ["version"]

[tool.pdm.version]
source = "file"
path = "myapp/core/version.py"
```

**The actual default regex pdm-backend uses** [VERIFIED: pdm-backend
`src/pdm/backend/hooks/version/__init__.py`, line ~92, fetched from
GitHub main as of 2026-05-01]:

```python
r"^(?:__version__|VERSION)\s*=\s*[\"'](.+?)[\"']\s*(?:#.*)?$"
# applied with re.M flag (multiline)
```

Properties of this regex:
- Matches `__version__` OR `VERSION` (so the legacy tuple-named
  `VERSION = (0, 0, 1, ...)` would NOT match — it's not a quoted string).
- Single OR double quotes both match (`["']`).
- Trailing inline comment after `#` is allowed (`(?:#.*)?$`).
- Trailing whitespace before EOL is allowed (`\s*`).
- `re.M` makes `^...$` anchor each physical line.

**CONTEXT.md regex citation is slightly inaccurate.** CONTEXT.md says the
pattern is `^__version__\s*=\s*['"](.*?)['"]`. The actual default also
matches `VERSION =` and requires line-end discipline. For Shape B
(`__version__ = "0.0.1a0"`) both forms successfully match — this is a
documentation-precision nit, not a behavioral risk. Recommend the planner
not propagate the CONTEXT.md regex into commit bodies; quote the
upstream-verified one if a regex needs to be cited at all.

### Pattern 2: PEP 735 dev dependency group

**What:** Standardized location for non-runtime dependencies, distinct
from optional-extras (which are install-time-selectable per-user).
**When to use:** Tools that all developers of a project need (test runner,
build tools, publishers) — not optional consumer-facing extras.

**Example (D-03b):**
```toml
[dependency-groups]
dev = [
    "pytest",
    "pytest-cov",
    "coverage",
    "twine>=1.11.0",
]
```

**Install paths:**
- `pdm install` (default; installs runtime + default groups inc. dev)
- `pdm install -G dev` (explicit)
- `pip install --group dev` (pip ≥25.1, Apr 2025)
- `pip install --group ./pyproject.toml:dev` (cross-directory; pip ≥25.1)

**Backward-compat note:** Users on pip <25.1 cannot use `--group`; they
must use `pdm install` or hand-install the listed packages. This is
acceptable since the generated project's Makefile uses `pdm install` per
D-12, not pip directly. [VERIFIED: pip 25.1 release post, 2025-04-26]

### Pattern 3: Generated Makefile alignment with pdm-backend (D-12)

```makefile
.PHONY: clean virtualenv test docker dist dist-upload

clean:
	find . -name '*.py[co]' -delete

virtualenv:
	pdm venv create
	pdm install
	@echo
	@echo "VirtualENV Setup Complete. Now run: eval \$$(pdm venv activate)"
	@echo

test:
	pdm run pytest \
		-v \
		--cov={{ label }} \
		--cov-report=term \
		--cov-report=html:coverage-report \
		tests/

docker: clean
	docker build -t {{ label }}:latest .

dist: clean
	rm -rf dist/*
	pdm build

dist-upload:
	twine upload dist/*
```

Notes:
- `virtualenv` target preserves `pdm install` semantics (one command does
  what was previously two: `pip install -r requirements-dev.txt` +
  `python setup.py develop`).
- `test` target uses `pdm run pytest` to ensure the venv is active.
- `dist` target uses `pdm build` (single command produces both sdist and
  wheel; replaces `python setup.py sdist` + `python setup.py bdist_wheel`).
- `docker` target unchanged (still builds the Dockerfile — but see Open
  Question 1 about Dockerfile internal references).

### Anti-Patterns to Avoid

- **Re-introducing `cement` in `[build-system].requires`.** This was the
  failed strict-minimum approach. Build-time deps are for build *backends*
  and their helpers — not application runtime libs. pdm-backend has zero
  cement dependency.
- **Using `[tool.pdm.version] source = "call"`.** This re-introduces the
  build-time import bug (call source executes Python to get the version).
  CONTEXT.md correctly mandates file-source.
- **Adding `[tool.pdm.build] includes = [...]` for the templates dir.**
  Verified: pdm-backend default behavior auto-includes non-Python files
  inside detected package directories. Adding explicit includes risks
  *overriding* the defaults and silently breaking package-data inclusion
  (per pdm-backend docs: "specifying includes and excludes will override
  their default values"). See Pitfall 2.
- **Using `[project.optional-dependencies]` for dev tools.** That table
  is for end-user-selectable extras (e.g., `pip install myapp[redis]`),
  not for project developers. PEP 735 `[dependency-groups]` is the
  correct location.
- **Reverting `8e7fba5e` AND writing a new pyproject.toml in separate
  commits.** Net change is the same as overwriting in place — but the
  diff is cleaner with a single overwrite-in-place commit. CONTEXT.md
  D-15 step 1 explicitly notes this is "Optional — planner can also
  choose to overwrite the file in place; either is auditable." See
  Open Question 3 for recommendation.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Read version literal at build time | Custom AST walker, custom regex parser | pdm-backend file-source | Already implemented, well-tested, handles single/double quotes + trailing comments |
| Build wheel from `[project]` metadata | setup.py invocation, custom build script | pdm-backend `build_wheel()` | PEP 517 standard interface; pip handles isolation automatically |
| Generate sdist + wheel | `python setup.py sdist bdist_wheel` (deprecated) | `pdm build` | Single command; modern flow; setuptools deprecation warnings everywhere |
| Track dev tools alongside runtime deps | `requirements-dev.txt` with `-r requirements.txt` | PEP 735 `[dependency-groups]` | Standardized; tool-agnostic; install-time selection |
| License metadata | Custom `LICENSE` file references in classifiers | `[project].license` (table or SPDX string) | Standardized; PEP 639 evolution path |
| Package non-Python data files | `MANIFEST.in` + `include_package_data` | pdm-backend default packaging | Auto-included for files inside detected package dirs |

**Key insight:** The whole phase is "stop hand-rolling Python packaging
on top of setuptools+setup.py." Every line of the legacy template is a
custom solution to a problem that the modern PEP 621/735 stack solves
declaratively.

## Common Pitfalls

### Pitfall 1: Regex vs. Shape B literal mismatch
**What goes wrong:** `__version__` written with single quotes, missing
closing quote, or with trailing semicolon doesn't match the regex.
**Why it happens:** Hand-typing the version line in a hurry; copy-paste
loses a character.
**How to avoid:** Use double-quoted string `__version__ = "0.0.1a0"` per
D-05. Both single and double work, but double matches the codebase's
other `version.py` files. After writing, run a sanity grep:
```bash
grep -E '^(__version__|VERSION)\s*=\s*["'\''](.+?)["'\''][[:space:]]*(#.*)?$' \
  "cement/cli/templates/generate/project/{{ label }}/core/version.py"
```
**Warning signs:** `pdm build` (or `pip install .`) errors with
`Couldn't find version in file` from pdm-backend's `ConfigError`.

### Pitfall 2: package-data exclusion via accidental includes override
**What goes wrong:** Adding `[tool.pdm.build] includes = ["myapp/"]`
silently *removes* the `myapp/templates/*.jinja2` files from the wheel
because pdm-backend treats explicit includes as overrides of the
default-include-with-data behavior.
**Why it happens:** Engineers add explicit includes "to be safe" without
reading the docs.
**How to avoid:** **Do not add `[tool.pdm.build]` to the new
pyproject.toml.** pdm-backend's default behavior auto-includes non-Python
files inside detected packages. CONTEXT.md "Claude's Discretion" lists
this as a planner-verifies item — research has verified: NOT NEEDED.
**Warning signs:** Generated `myapp` runs `myapp command1 --foo bar` and
errors with `jinja2.exceptions.TemplateNotFound: command1.jinja2`. This
is a **runtime failure**, not a build failure — it would only surface in
a more thorough smoke test that exercises template rendering. **The
current smoke test does NOT exercise template rendering** (it only checks
`myapp --version`), so this would be a silent failure. **Mitigation:**
the cement smoke test happens to NOT trigger this code path, but if a
future smoke test adds rendering, the absence of `[tool.pdm.build]`
override is essential. [VERIFIED: pdm-backend `build_config/` docs]

### Pitfall 3: Missing `requires-python` in `[project]`
**What goes wrong:** Generated wheel has no Python version constraint;
pip happily installs on Python 3.8 where it'll fail at runtime when
cement (which requires 3.10+) is installed.
**Why it happens:** CONTEXT.md D-02 doesn't list `requires-python`.
**How to avoid:** Add `requires-python = ">=3.10"` to `[project]` —
matches cement's own `pyproject.toml` line 19 and matches what the
template's runtime supports (the smoke test runs 3.10–3.14). This is
a "Claude's Discretion" addition; research recommends including it.
**Warning signs:** None at build time. Surfaces when a downstream user
on Python 3.9 tries to install and gets a runtime ImportError.

### Pitfall 4: PEP 639 license-table deprecation
**What goes wrong:** pdm-backend (and downstream tools — twine, pip's
metadata validator) emit a `DeprecationWarning` for `license = {text =
"..."}`. setuptools v77+ has already deprecated it with removal targeted
Feb 2026.
**Why it happens:** PEP 639 prefers `license = "MIT"` (SPDX string) and
introduces `license-files = [...]` for license-file references.
**Current state in pdm-backend:** [VERIFIED: pdm-backend commit
`aa7aecb`] Backward compat is retained. The table form is accepted, not
errored. Deprecation warnings *may* fire on recent pdm-backend releases
(2.4.x).
**How to avoid this phase:** Use CONTEXT.md D-02's table form as
specified. **Do not** unilaterally upgrade to SPDX strings — the
template's `{{ license }}` substitution defaults to `"unlicensed"`
(per `.generate.yml` line 44) which is NOT a valid SPDX expression.
SPDX migration is a **deferred** follow-up; out of scope for 01.1.
**Warning signs:** `DeprecationWarning` lines in smoke-test stdout. As
long as they're warnings (not errors) the smoke test still passes.

### Pitfall 5: `cement.version` substitution producing dev-format string
**What goes wrong:** `{{ cement.version }}` resolves to a value that's
not a valid PEP 440 version specifier when used as a dependency pin.
**Why it happens:** `cement.utils.version.get_version()` can return
`3.0.15.dev<timestamp>` for alpha-0 releases (line 65-68 of
`cement/utils/version.py`). For `(3, 0, 15, 'alpha', 0)` with a git
checkset, the value would be `3.0.15.dev20260501163735`.
**Verification (current state):** VERSION = `(3, 0, 15, 'final', 0)`
[VERIFIED: `cement/core/backend.py:3`] — so the current dev-stem is
`3.0.15` (no `.dev` suffix because release is `final`, not `alpha`).
The substituted line `cement[jinja2,yaml,colorlog]==3.0.15` is a valid
PEP 440 specifier.
**How to avoid:** No action needed for cement 3.0.15. **Watch for
breakage during the 3.0.17-dev cycle** when VERSION may flip back to
`alpha-0` and the substituted version may pick up a `.dev` suffix.
That's a Phase-2-or-later concern, not a 01.1 risk. The smoke test
runs against the current `(3, 0, 15, 'final', 0)` build.
**Warning signs:** `pip install .` in the smoke test errors with
`InvalidRequirement` or `InvalidVersion`.

### Pitfall 6: README.md / Dockerfile reference deleted files
**What goes wrong:** After `requirements.txt`, `setup.py`, and
`requirements-dev.txt` are deleted, two artifacts in the template still
reference them:
- `Dockerfile` line 7-8: `RUN pip install --no-cache-dir -r
  requirements.txt && python setup.py install`
- `README.md` lines 6-8: install instructions reference
  `pip install -r requirements.txt` and `python setup.py install`
**Why it happens:** CONTEXT.md "Files MODIFIED" list is exhaustive
about pyproject.toml/version.py/Makefile/smoke-test but doesn't address
these two. README is **explicitly deferred** in CONTEXT.md `<deferred>`
section ("Modernize the generated project's CHANGELOG.md / README.md to
reflect the new toolchain"). Dockerfile is **NOT** addressed either way.
**How to avoid:** See Open Question 1 — recommend planner ask the user
whether Dockerfile is in or out of scope. The smoke test does NOT
docker-build the generated project (only `pip install .`), so leaving
the Dockerfile broken does not fail acceptance — but it ships a
broken Dockerfile to downstream users.
**Warning signs:** None at smoke-test time. Surfaces only when a
downstream user runs `make docker` on a freshly-generated project.

### Pitfall 7: Build-isolation environment variables leaking
**What goes wrong:** A residual env var (e.g., `PIP_FIND_LINKS=/src/dist`
left over from a prior commit, or in a developer's shell) silently
modifies pip's resolution path even after the smoke test stops setting
it explicitly.
**Why it happens:** D-13 reverts the explicit `-e PIP_FIND_LINKS=...`
from line 57, but if a developer has the env var set in their shell,
`docker exec` does NOT inherit host env (so this is fine for CI). For
local manual smoke-test debugging, ensure no sticky env vars.
**How to avoid:** D-13 reverts to *exactly* the original line 53 —
a plain `docker exec cement-cli-smoke-test /bin/bash -c "cd ... ; pip
install ."` with no `-e` flags.
**Warning signs:** Smoke test passes locally but fails in CI (or vice
versa) with `cement` not resolving from the expected source.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `setup.py` + `setup.cfg` + `MANIFEST.in` | `pyproject.toml` `[project]` table only | PEP 621 (2020), broadly adopted by 2023 | Single declarative source; no Python execution at config-read time |
| `python setup.py sdist bdist_wheel` | `pdm build` (or `python -m build`) | setup.py CLI deprecated since setuptools 58 (2021); fully removed in 80+ | One command; respects PEP 517 isolation |
| `requirements-dev.txt` with `-r requirements.txt` | `[dependency-groups].dev` (PEP 735) | PEP 735 accepted Oct 2024; pip 25.1 (Apr 2025) | Standardized; works with pip and PDM and uv |
| `license = {text = "..."}` | `license = "<SPDX>"` + `license-files = [...]` | PEP 639 accepted; setuptools v77 deprecated table form | **NOT migrated this phase** (deferred — `{{ license }}` defaults to non-SPDX `"unlicensed"`) |
| setuptools.build_meta | pdm-backend | Choice; matches cement's toolchain | Smaller build-env footprint; PEP 621/735 native |
| `cement` in `[build-system].requires` (the failed strict-minimum approach) | Cement is runtime-only; build-time uses pdm-backend regex | This phase | Decouples build env from runtime env; fixes PEP 517 import bug |

**Deprecated/outdated:**
- `python setup.py <command>` invocations: deprecated upstream by
  setuptools, removed entirely in setuptools 80+. The legacy template's
  Makefile uses these — D-12 fixes.
- `MANIFEST.in`: still works under setuptools but redundant under
  pdm-backend (default packaging covers what MANIFEST.in covered for
  this template).
- `setup.cfg`: this template's `setup.cfg` is **empty** (0 bytes —
  CONTEXT.md D-07's "1-line file" claim is inaccurate per `ls -l`).
  Deletion is trivial.
