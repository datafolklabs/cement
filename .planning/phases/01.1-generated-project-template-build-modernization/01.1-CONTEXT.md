# Phase 01.1: Generated Project Template Build Modernization - Context

**Gathered:** 2026-04-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the existing `cement generate project` template installable under
default PEP 517 build isolation on Python 3.10–3.14. The current template
fails because `setup.py` imports `{{ label }}.core.version`, which imports
`cement.utils.version` at module top-level — and the isolated build env
created by modern pip (default since 21.3) has no `cement`.

The fix is the minimum mechanical patch: add a `pyproject.toml` to the
template that declares `cement` as a build-time dependency in
`[build-system].requires`, so the isolated build env contains cement when
`setup.py` runs. **Nothing else in the template changes this phase.**

**In scope:** add one new file to the template. Verify the smoke test
goes green on the full matrix.

**Out of scope:**
- Migrating the template off setuptools
- Moving runtime deps from `requirements.txt` into `[project].dependencies`
- Dropping `version.py` / switching to `importlib.metadata`
- Modernizing the generated `Makefile` away from `python setup.py ...`
- Auditing or modifying sibling templates (`script`, `extension`, `plugin`)
- Removing cement as a build-time dependency long-term

All of those are real concerns, but they belong to a later phase or
milestone. This phase is the strict-minimum unblock for Phase 2's
CI-green goal — mirroring the D-13 strict-minimum boundary established
in Phase 1.

</domain>

<decisions>
## Implementation Decisions

### Scope discipline
- **D-01:** **Strict minimum patch.** Add exactly one new file
  (`cement/cli/templates/generate/project/pyproject.toml`). Touch no
  other template file. Mirrors Phase 1's D-13 pattern.
- **D-02:** Setuptools stays the build backend. The user's original
  framing ("migrate away from setuptools") is acknowledged as the
  long-term direction but explicitly **deferred** — a full backend
  migration plus version-source rework plus dev-workflow rework was
  scoped, then reverted on second look as scope creep for an inserted
  urgent phase. Revisit in a later milestone with full-phase ceremony.

### pyproject.toml shape (the only file this phase creates)
- **D-03:** `[build-system]` block:
  ```toml
  [build-system]
  requires = ["setuptools", "wheel", "cement>={{ cement.version }}"]
  build-backend = "setuptools.build_meta"
  ```
  - `cement>={{ cement.version }}` — floor pin matching the same
    template variable already used in `requirements.txt`. Loosest-still-
    correct; future cement minors can satisfy it without re-rendering
    the template.
  - `setuptools` and `wheel` declared explicitly even though they are
    the historical defaults — pip's default behavior is changing and
    being explicit is cheap insurance.
  - `build-backend = "setuptools.build_meta"` keeps setuptools the
    backend, preserving setup.py as the canonical config source.
- **D-04:** No `[project]` table in the new pyproject.toml. The new
  file declares ONLY `[build-system]`. All project metadata stays in
  `setup.py` to keep this patch surgical and avoid the dual-source
  problem that would arise from a partial migration.

### Files explicitly left untouched (audit-quality "no-go" list)
- **D-05:** `cement/cli/templates/generate/project/setup.py` — kept
  as-is. The `from {{ label }}.core.version import get_version` import
  on line 3 stays.
- **D-06:** `cement/cli/templates/generate/project/{{ label }}/core/version.py`
  — kept as-is. `from cement.utils.version import get_version as cement_get_version`
  stays. The `VERSION = (0, 0, 1, 'alpha', 0)` tuple convention stays —
  this mirrors cement's own version module shape and downstream users
  recognize it.
- **D-07:** `cement/cli/templates/generate/project/{{ label }}/controllers/base.py`
  — kept as-is. `from ..core.version import get_version` and
  `from cement.utils.version import get_version_banner` stay.
- **D-08:** `cement/cli/templates/generate/project/requirements.txt` and
  `requirements-dev.txt` — kept as-is. Runtime deps stay declared in
  requirements.txt with the existing `cement=={{ cement.version }}` plus
  the three extras lines.
- **D-09:** `cement/cli/templates/generate/project/Makefile` — kept
  as-is. The `virtualenv` target's `python setup.py develop` and the

<code_context>
## Existing Code Insights

### The actual mechanism of failure
- pip ≥21.3 enables PEP 517 build isolation by default. When `pip install .`
  runs on a generated project, pip creates an ephemeral build venv
  containing only what is declared in `[build-system].requires`. With no
  `pyproject.toml`, the implicit fallback is `setuptools` + `wheel` only.
- The generated `setup.py:3` does `from {{ label }}.core.version import get_version`.
- `{{ label }}/core/version.py:2` does `from cement.utils.version import get_version as cement_get_version`.
- The build env has no cement → `ModuleNotFoundError: No module named 'cement'`.
- Adding `cement` to `[build-system].requires` puts cement in the build
  venv. setup.py then runs successfully and the wheel builds.

### Why this never bit before the recent CI changes
- Pre-`020ec7b7` (last week's smoke-test commit), the smoke test used a
  legacy install path (`python setup.py install` or
  `pip install --no-build-isolation`), which runs setup.py against the
  container's main site-packages where cement is already installed.
  That path masks the bug entirely.
- Downstream users running `pip install .` on freshly-generated projects
  with modern pip have been silently affected for years. CI just never
  caught it because CI wasn't doing what real users do — until last week.

### Why the "minimum patch" is correct now
- The full modernization (drop setuptools, drop version.py imports,
  migrate to PDM, etc.) is a real project that deserves its own phase
  with proper research and planning. Inserting it as urgent work between
  Phase 1 and Phase 2 was scoped in this discussion and explicitly
  walked back as scope creep.
- The two-line `[build-system]` declaration restores the contract that
  pip ≤21.2 implicitly assumed: "the build env has whatever the
  surrounding system does." It's not elegant, but it's correct and
  isolated.

### Reusable assets (none new are needed)
- The smoke test (`scripts/cli-smoke-test.sh`) is the acceptance harness
  — already in place, already wired to `make cli-smoke-test`.
- The template directory structure (`cement/cli/templates/generate/project/`)
  uses Jinja-style `{{ }}` placeholders. The new pyproject.toml will
  use one such placeholder (`{{ cement.version }}`) consistent with how
  `requirements.txt` already does it.

### Integration points
- `cement generate project` (the CLI generator) uses the
  `cement/cli/templates/generate/project/` directory contents as the
  template source. Adding a new file there means newly-generated projects
  pick it up automatically.
- The `cement.cli.main` generator code does not need changes — it copies
  template files verbatim into the destination, so a new file in the
  template directory simply propagates.
- Existing generated projects in the wild are unaffected (they have
  their own pre-existing copies of these files).

</code_context>

<deferred>
## Deferred Ideas

These were considered, sketched, then explicitly walked back. They
remain real future-phase work — capture here so they aren't lost.

- **Migrate template off setuptools entirely** — pdm-backend (matches
  cement) or hatchling. Removes setuptools from build-system requires.
  Requires version-source rework (D-06 cannot remain as-is).
- **Migrate generated project's dev workflow to PDM** — `pdm install`,
  `pdm build`, `pdm run pytest` in the generated Makefile, replacing
  the legacy `python setup.py develop` / `python setup.py sdist bdist_wheel`
  patterns. Aligns generated projects with cement's own toolchain.
- **Move runtime deps from requirements.txt into [project].dependencies**
  — single source of truth in pyproject.toml. Drop requirements.txt.
  Smoke test line 52 (`pip install -r requirements.txt`) becomes
  removable at that point.
- **Drop `{{ label }}/core/version.py` entirely** — switch
  `controllers/base.py` to `importlib.metadata.version("{{ label }}")`.
  Static `version =` literal in pyproject.toml. Removes the cement
  import dependency from the package itself, not just the build env —
  which is the real long-term fix and the correct interpretation of
  the user's original framing.
- **Drop cement from [build-system].requires** — depends on the above
  three. Once version.py no longer imports cement (or version.py is
  gone entirely), the build env doesn't need cement either, and we
  finally satisfy the user's original principle.
- **Audit sibling templates** (`script/`, `extension/`, `plugin/`) for
  unrelated modernization debt. Not pip-installable so they don't have
  this specific bug, but they may have other staleness.
- **Modernize the generated Makefile's legacy targets** independently
  of build-backend choice. `python setup.py develop` is deprecated;
  `pip install -e .` is the modern equivalent. Even staying on
  setuptools, this could update.

When picking these up, the natural grouping is one phase covering items
1, 2, 3, and 4 (the cement-removal-from-build path), with item 5 (sibling
audit) as either a sub-task or its own follow-up phase.

</deferred>
