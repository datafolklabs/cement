<revision_history>
## Revision history

**Original scope (commit `a6b6cb4a`)** — strict-minimum patch: add a single
`pyproject.toml` declaring cement as a build-time dependency, leave setup.py
and everything else untouched.

**Why re-scoped:** the strict-minimum patch was empirically insufficient.
We hit two walls during smoke-test verification:

1. After adding `cement>={{ cement.version }}` to `[build-system].requires`,
   the dev-build cement (3.0.15) wasn't on PyPI — fixed via
   `PIP_FIND_LINKS=/src/dist` in the smoke test (commit `3d60764c`).

2. Even with cement available, `setup.py` line 3 (`from {{ label }}.core.version
   import get_version`) imports the package being built. Under PEP 517
   isolation the build env's `sys.path` does not include cwd, so
   `import myapp` fails with `ModuleNotFoundError`. **No amount of
   `[build-system].requires` configuration fixes this** — the issue is
   structural to setup.py-imports-self under isolation.

**Conclusion:** the deferred-modernization items captured in the original
`<deferred>` section weren't optional after all. This revision pulls them
forward into Phase 01.1 because the strict-minimum approach cannot
deliver the phase goal (`pip install .` succeeds on a generated project
under default PEP 517 isolation).

The `feat(cli): add pyproject.toml ...` commit (`8e7fba5e`) and the
smoke-test `PIP_FIND_LINKS` fix (`3d60764c`) will be REVERTED or
SUPERSEDED by the proper migration. Planner / executor must coordinate
this — see D-15 below.

</revision_history>

<domain>
## Phase Boundary

Migrate `cement generate project`'s template from setuptools+setup.py to
pdm-backend with a complete `[project]`-table-driven `pyproject.toml`. After
this phase, generated projects:

- Build cleanly under default PEP 517 isolation with no special pip flags
  (`pip install .` just works on Python 3.10–3.14 against a published cement)
- Use pdm-backend as their build system (matching cement's own toolchain)
- Declare runtime deps in `[project].dependencies` (single source of truth)
- Use a literal `__version__` string in `version.py` (no cement at build
  time, no package self-import at build time)
- Use PDM as the dev-workflow front-end (`pdm install`, `pdm build`,
  `pdm run pytest`)

**In scope:** the project template only. Acceptance is `make cli-smoke-test`
green on Python 3.10–3.14.

**Out of scope:**
- Sibling templates (`script`, `extension`, `plugin`) — they are not
  pip-installable as separate packages and don't have this bug
- Modernizing cement's own pyproject.toml — already on pdm-backend
- Any cement runtime code

</domain>

<decisions>
## Implementation Decisions

### Build system
- **D-01:** **Build backend: pdm-backend.** Matches cement's own choice.
  ```toml
  [build-system]
  requires = ["pdm-backend"]
  build-backend = "pdm.backend"
  ```
  Drops setuptools, drops wheel-as-explicit-build-dep.

### Project metadata
- **D-02:** **Full `[project]` table in pyproject.toml.** All metadata
  that previously lived in setup.py moves here:
  ```toml
  [project]
  name = "{{ label }}"
  description = "{{ description }}"
  authors = [{name = "{{ creator }}", email = "{{ creator_email }}"}]
  license = {text = "{{ license }}"}
  readme = "README.md"
  dynamic = ["version"]
  dependencies = [
      "cement[jinja2,yaml,colorlog]=={{ cement.version }}",
  ]

  [project.urls]
  Homepage = "{{ url }}"

  [project.scripts]
  {{ label }} = "{{ label }}.main:main"
  ```
- **D-03:** **Runtime deps consolidated.** Previous `requirements.txt`
  shape (`cement==X.Y` + three separate `cement[extra]` lines) collapses
  into a single line in `[project].dependencies` per D-02. requirements.txt
  is DELETED (D-07).
- **D-03b:** **Dev deps moved to `[dependency-groups].dev`** (PEP 735),
  consistent with how runtime deps move into `[project]`. The current
  `requirements-dev.txt` ships with: `pytest`, `pytest-cov`, `coverage`,
  `twine>=1.11.0`, `setuptools>=38.6.0`, `wheel>=0.31.0`. Migration:
  ```toml
  [dependency-groups]
  dev = [
      "pytest",
      "pytest-cov",
      "coverage",
      "twine>=1.11.0",
  ]
  ```
  - `setuptools` and `wheel` dropped from dev deps. They were dev deps
    only because the legacy `python setup.py sdist bdist_wheel` flow
    needed them; with pdm-backend the build system handles wheel
    creation directly via `pdm build`, and pdm-backend itself is the
    only build tool the project needs.
  - `twine` preserved — still useful for PyPI upload via `twine upload`.
    Users who prefer `pdm publish` can add it themselves.
  - `-r requirements.txt` line in the legacy file is dropped — under
    PEP 735, dev groups install alongside the project's runtime deps,
    so re-declaring runtime deps in the dev group is unnecessary.
  - Install via `pdm install -G dev` or `pip install --group dev .` (PEP
    735 syntax in pip ≥23.1) or `pdm install` (which installs default
    groups including dev).
  - `requirements-dev.txt` is DELETED (D-07b).

### Version source
- **D-04:** **pdm-backend file-source for version.** pdm-backend reads
  the version from a regex-matched line in version.py — no AST execution,
  no imports, no cement needed at build time:
  ```toml
  [tool.pdm.version]
  source = "file"
  path = "{{ label }}/core/version.py"
  ```
- **D-05:** **version.py adopts Shape B.** Drops the `VERSION = (0, 0, 1,
  'alpha', 0)` tuple convention. cement import removed. Final shape:
  ```python
  __version__ = "0.0.1a0"

  def get_version():
      return __version__
  ```
  - `__version__` is the canonical version string. pdm-backend reads it
    via the regex `^__version__\s*=\s*['"](.*?)['"]` (default pdm-backend
    file-source pattern).
  - `get_version()` is preserved as a programmatic-access wrapper so
    existing template consumers (`{{ label }}/controllers/base.py:4`)
    keep working without changes.
  - Loses the cement-style tuple convention. This is an intentional
    divergence — the tuple convention's value (formatter via cement) was
    the source of the build-time cement dependency we're removing.

<code_context>
## Existing Code Insights

### Why pdm-backend is the right backend choice
- cement itself uses pdm-backend (verified: project root `pyproject.toml`).
  Generated projects matching the parent project's toolchain reduces
  cognitive overhead for downstream users who interact with both.
- pdm-backend's file-source for version is *literally designed* for the
  Shape B pattern in D-05 — regex-based, no execution, no imports.
  Setuptools' equivalent (`[tool.setuptools.dynamic] version = {attr=...}`)
  uses AST inspection which works for our case but is more brittle and
  less idiomatic.
- Smaller `[build-system].requires` footprint (`["pdm-backend"]` vs
  `["setuptools", "wheel"]`).

### Why Shape B over Shape A
- Shape A (tuple + parallel literal) requires hand-syncing two values
  on every release. Downstream users WILL drift them. The footgun is
  not worth preserving the tuple convention.
- Shape B (literal + trivial wrapper) is self-consistent. `get_version()`
  exists for backward compat with `controllers/base.py`; it's a one-line
  passthrough.
- Cement's tuple-and-formatter shape exists because cement *provides* the
  formatter. Generated projects shouldn't replicate the shape if they're
  not using the formatter — that was the original premise of the deferred
  migration we're now executing.

### Why pip install . works after this migration
1. User runs `pip install .` on a freshly-generated myapp.
2. pip reads pyproject.toml. `[build-system]` says use pdm-backend.
3. pip creates an isolated build env, installs pdm-backend from PyPI
   (small dependency, no transitive cement).
4. pdm-backend reads `[project]` for metadata, reads version from
   version.py via regex (no execution, no cement needed at build time,
   no `import myapp`).
5. pdm-backend builds the wheel. No setup.py exists; no setup.py runs.
6. pip installs the wheel and resolves runtime deps from
   `[project].dependencies` against the user's environment / PyPI.
7. Smoke-test container has cement preinstalled (line 22) so the runtime
   resolution finds cement immediately and is satisfied.

### Why both prior 01.1 commits get superseded
- `8e7fba5e` (`feat(cli): add pyproject.toml...`) — wrote a `[build-system]`-
  only stub keeping cement as a build-time dep. Under the new design we
  don't need cement at build time at all. The new pyproject.toml is a
  complete rewrite, not an extension of the stub. Either revert + re-add
  or overwrite in place; either is auditable. Planner picks.
- `3d60764c` (`fix(ci): point smoke-test build env at /src/dist...`) —
  the `PIP_FIND_LINKS=/src/dist` env var was needed only because cement
  was in `[build-system].requires`. With pdm-backend the build env
  doesn't need cement, so this band-aid is no longer needed. Revert
  cleanly.

### What stays in place from prior commits
- `cement/cli/templates/generate/project/Dockerfile` — already
  `python:3.10-alpine` from Phase 1's D-05; not a 01.1 concern.
- `make cli-smoke-test` Makefile target — added in quick task
  `260430-i7q`; remains the entry point. No change here.
- The smoke test's `pdm build` step (line 89) — produces the cement sdist
