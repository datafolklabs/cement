<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `cement generate project` template-render boundary | Template variable values flow from `cement.utils.version.get_version()` (controlled, in-process) into the rendered `pyproject.toml`. No user-supplied input. |
| pip PEP 517 build-isolation venv boundary | When a downstream user runs `pip install .` on a generated project, pip resolves and installs `setuptools`, `wheel`, and `cement>={{ cement.version }}` into an ephemeral venv. Network boundary to PyPI. |
| setuptools backend → setup.py execution | Inside the build venv, `setuptools.build_meta` invokes `setup.py`, which imports `{{ label }}.core.version` and (transitively) `cement.utils.version`. Code execution. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01.1-01 | Tampering | Floor pin `cement>={{ cement.version }}` allows any future cement minor (including a hypothetical compromised release) into a downstream user's build venv | accept | Floor pins are PyPA-standard practice. Tighter pins (`==`) shift risk to forced-stale builds, not eliminate it. The same `cement` dependency already exists in the template's `requirements.txt` at runtime; build-time exposure is a strict subset of an exposure that is already accepted. No new mitigation possible at this phase's scope. Defer to a future phase that drops cement from build-time deps entirely (per CONTEXT.md `<deferred>` item 5). |
| T-01.1-02 | Tampering | Build venv pulls fresh `setuptools` and `wheel` from PyPI on every `pip install .` | accept | Inherent to PEP 517. Mitigated upstream by PyPI infrastructure (signing, transparency log) and pip's hash-checking modes (not enabled here). Out of scope for a 1-file template patch; phase boundary disallows. |
| T-01.1-03 | Tampering | Template-injection via `{{ cement.version }}` placeholder | mitigate | The placeholder value is sourced from cement's hardcoded `VERSION` tuple via `get_version()` — controlled, not user-input. No user-supplied input flows into this template. Verified at `cement/core/backend.py:3` and `cement/ext/ext_generate.py:38`. Not exploitable. |
| T-01.1-04 | Information Disclosure | New `pyproject.toml` content visible to anyone who installs the generated project | accept | The file's content is intentionally public packaging metadata — the standard pattern for every Python project on PyPI. No secrets. |

**Net assessment:** This phase introduces zero new attack surface beyond what
the existing `requirements.txt` already exposes. The supply-chain shape of the
generated project is unchanged in kind; only widened by one entry (cement now
declared at build time in addition to runtime, where it already is). Security
gate is informational here, not blocking.
</threat_model>
