#### `[project]` pattern (root pyproject.toml lines 1-20)

```toml
[project]
name = "cement"
description = "Application Framework for Python"
readme = "README.md"

    "python",
    "module",
    "cli framework"
]
authors=[{name = "Data Folk Labs, LLC", email= "derks@datafolklabs.com"}]

    "Programming Language :: Python :: 3",
    "License :: OSI Approved :: BSD License"
```

Notes for the template adaptation:
- **`license` form:** root uses `license = {file="LICENSE"}`. CONTEXT.md D-02
  mandates `license = {text = "{{ license }}"}` for the template (the template's
  `{{ license }}` substitution defaults to the literal string `"unlicensed"`,
  which is not a valid SPDX expression — see RESEARCH.md Pitfall 4). Stick with
  the table form for this phase.
- **`dynamic`:** root has `dynamic = ["version", "README"]`. The template only
  needs `dynamic = ["version"]` per CONTEXT.md (`<canonical_refs>` line 281:
  "Generated projects do not need the README dynamic — keep that simple").
- **`requires-python`:** root pins `>=3.10`. RESEARCH.md Pitfall 3 + Open
  Question 4 recommends matching this in the template. Claude's Discretion per
  CONTEXT.md but research's recommendation is yes.
- **`dependencies`:** root has `dependencies = []` (zero-runtime-dep core
  framework). Template per D-02 has the consolidated single-line dep
  `"cement[jinja2,yaml,colorlog]=={{ cement.version }}"`.

Final `[project]` block for the template (verbatim from RESEARCH.md "Code
Examples" §"Verified pyproject.toml shape"):

```toml
[project]

description = "{{ description }}"
```

#### `[tool.pdm.version]` pattern (root pyproject.toml lines 203-205)

```toml
[tool.pdm.version]
source = "call"
getter = "cement.utils.version:get_version"
```

**DIVERGENCE — do NOT copy verbatim.** cement uses `source = "call"` (which
executes Python at build time) because cement's tuple+formatter shape needs
`get_version()` to produce the PEP 440 string. The template **must** use
`source = "file"` per CONTEXT.md D-04, because `source = "call"` re-introduces
the import-during-build bug we are fixing (RESEARCH.md "Anti-Patterns to Avoid":
"Using `[tool.pdm.version] source = "call"` re-introduces the build-time
import bug"). Final block for the template:

```toml
[tool.pdm.version]
source = "file"
path = "{{ label }}/core/version.py"
```

**Adapt, don't copy.** Per CONTEXT.md D-03b the template's dev group is the
migrated `requirements-dev.txt` with `setuptools` and `wheel` dropped:

```toml
[dependency-groups]
dev = [
    "pytest",
    "pytest-cov",
    "coverage",
    "twine>=1.11.0",

```

Pin styles in the template are unpinned (`"pytest"` not `"pytest>=9.0.3"`)
because that's how `requirements-dev.txt` shipped them (lines 3-6 of
`cement/cli/templates/generate/project/requirements-dev.txt`).

#### `[tool.pdm.build]` pattern (root pyproject.toml lines 193-201)

```toml
[tool.pdm.build]
package-dir = "."
includes = [
    "cement/",
    "cement/cli/templates/generate/",
    "CONTRIBUTORS.md",
    "CHANGELOG.md"
]
excludes = ["tests/"]
```

**DO NOT copy this block into the template.** RESEARCH.md "Anti-Patterns to
Avoid" + Pitfall 2: explicit `includes` *override* pdm-backend's default
package-data behavior, which would silently exclude the template's
`{{ label }}/templates/*.jinja2` files from the generated project's wheel.
The default packaging behavior of pdm-backend auto-includes non-Python files
inside detected packages — exactly what the template needs. Omit
`[tool.pdm.build]` entirely.

### `cement/cli/templates/generate/project/{{ label }}/core/version.py` (REWRITE)

**Analog:** none in-tree. Shape B is fully spec'd in CONTEXT.md D-05; cement's
own `cement/utils/version.py` (read above) is the *opposite* shape (Shape A —
tuple-and-formatter) and explains why the template should NOT mirror it (see
"Why Shape B over Shape A" in CONTEXT.md `<code_context>`).

**Current content** (`cement/cli/templates/generate/project/{{ label }}/core/version.py`,
lines 1-7 — Shape A with cement import, the source of the import-self-during-
build bug):

```python

from cement.utils.version import get_version as cement_get_version

VERSION = (0, 0, 1, 'alpha', 0)

def get_version(version=VERSION):
    return cement_get_version(version)
```

**Target content** (Shape B per CONTEXT.md D-05 + RESEARCH.md "Verified
version.py Shape B" lines 676-682):

```python
__version__ = "0.0.1a0"


def get_version():
    return __version__
```

**Critical correctness conditions** (from RESEARCH.md Pattern 1 + Pitfall 1):
1. **Double-quoted literal.** pdm-backend's default regex
   `^(?:__version__|VERSION)\s*=\s*["'](.+?)["']\s*(?:#.*)?$` (re.M) accepts
   single OR double quotes, but double matches the project root's
   `cement/utils/version.py` codebase convention.
2. **Identifier `__version__`.** Must be exactly `__version__` (or `VERSION`,
   per the regex's first alternation, but a string literal not a tuple — the
   legacy template's `VERSION = (0, 0, 1, 'alpha', 0)` would NOT match because
   the right-hand side is a tuple, not a quoted string).
3. **Trailing inline comment OK.** The regex accepts `# comment` after the
   literal, but research recommends omitting it for clarity.
4. **Compatibility wrapper preserved.** `get_version()` must remain a
   function (not a string constant) because
   `cement/cli/templates/generate/project/{{ label }}/controllers/base.py:4`
   does `from ..core.version import get_version` and calls it on line 9
   (`%s\n%s\n""" % (get_version(), get_version_banner())`). D-14 mandates
   base.py is unchanged; therefore `get_version` must be callable.
5. **`0.0.1a0` is the PEP 440 equivalent** of the legacy
   `(0, 0, 1, 'alpha', 0)` tuple as rendered by cement's
   `get_version()` formatter (RESEARCH.md "Verified version.py Shape B" note).
6. **Sanity grep** the generated file matches the regex (RESEARCH.md
   Pitfall 1 mitigation):
   ```bash
   grep -E '^(__version__|VERSION)\s*=\s*["'\''](.+?)["'\''][[:space:]]*(#.*)?$' \
     "cement/cli/templates/generate/project/{{ label }}/core/version.py"
   ```

**For executor — verifying the `get_version()` import compatibility:**

`cement/cli/templates/generate/project/{{ label }}/controllers/base.py` line
4 — UNCHANGED per D-14 — currently reads:

```python
from ..core.version import get_version
```

This continues to resolve under Shape B because `get_version` is preserved
as a function returning `__version__`.

### `cement/cli/templates/generate/project/Makefile` (UPDATE)

**Analog:** `Makefile` (cement root) — `virtualenv`, `test`, and `dist`
targets all use PDM idioms that the template should mirror.

**Current content** (`cement/cli/templates/generate/project/Makefile`,
all 32 lines — uses legacy setup.py / pip-based workflow):

```makefile
.PHONY: clean virtualenv test docker dist dist-upload

clean:
	find . -name '*.py[co]' -delete

virtualenv:
	virtualenv --prompt '|> {{ label }} <| ' env
	env/bin/pip install -r requirements-dev.txt
	env/bin/python setup.py develop
	@echo
	@echo "VirtualENV Setup Complete. Now run: source env/bin/activate"
	@echo

test:
	python -m pytest \
		-v \
		--cov={{ label }} \
		--cov-report=term \
		--cov-report=html:coverage-report \
		tests/

docker: clean
	docker build -t {{ label }}:latest .

dist: clean
	rm -rf dist/*
	python setup.py sdist
	python setup.py bdist_wheel

dist-upload:
	twine upload dist/*
```

**`virtualenv` target pattern** (cement root `Makefile` lines 37-42):

```makefile
virtualenv:
	pdm venv create
	pdm install
	@echo
	@echo "VirtualENV Setup Complete. Now run: eval $(pdm venv activate)"
	@echo
```

Note the cement root `Makefile` uses a single un-escaped `$` in
`eval $(pdm venv activate)` because that line is inside a recipe's
`@echo` — the `$` is passed to the shell. **For the template, escape to
`\$$(pdm venv activate)`** per RESEARCH.md "Pattern 3: Generated Makefile
alignment" line 403 — Makefile recipes interpret `$` and re-emit `$$` as
literal `$` for the inner shell. Verified shape:

```makefile
virtualenv:
	pdm venv create
	pdm install
	@echo
	@echo "VirtualENV Setup Complete. Now run: eval \$$(pdm venv activate)"
	@echo
```

**`test` target pattern** (cement root `Makefile` lines 28-29):

```makefile
test: comply
	pdm run pytest --cov=cement tests
```

The cement root's `test` includes `comply` as a prerequisite (ruff + mypy);
the template doesn't have a `comply` target so we omit the prerequisite.
The substituted target keeps the existing pytest flags (`-v`,
`--cov-report=term`, `--cov-report=html:coverage-report`) but switches the
runner to `pdm run pytest`:

```makefile
test:
	pdm run pytest \
		-v \
		--cov={{ label }} \
		--cov-report=term \
		--cov-report=html:coverage-report \
		tests/
```

**`dist` target pattern** (cement root `Makefile` lines 79-80):

```makefile
dist:
	pdm build
```

The template's existing `dist` does `clean` + `rm -rf dist/*` + 2x setup.py
calls. Replace with `clean` + `rm -rf dist/*` + `pdm build` (single command
producing both sdist and wheel):

```makefile
dist: clean
	rm -rf dist/*
	pdm build
```

**Targets that DO NOT change:**
- `clean` — `find . -name '*.py[co]' -delete` is identical to cement root
  line 65 (cement root has additional cleanup but the template's minimal
  form is fine).
- `docker` — unchanged; still `docker build -t {{ label }}:latest .`.
- `dist-upload` — unchanged; still `twine upload dist/*`. (`twine` is
  preserved in `[dependency-groups].dev` per D-03b.)

## Shared Patterns

### Pattern: Template-substitution variables

**Source:** `cement/ext/ext_generate.py:38` — `data['cement']['version'] = get_version()`
**Apply to:** All template files containing `{{ ... }}`.

The cement-generate extension Jinja-substitutes these variables at generation
time (sourced from `.generate.yml` defaults plus runtime cement-version
introspection):

| Variable | Source | Example value |
|----------|--------|---------------|
| `{{ label }}` | user input / `.generate.yml` default `myapp` | `myapp` |
| `{{ description }}` | user input | `My Application` |
| `{{ creator }}` | user input | `John Doe` |
| `{{ creator_email }}` | user input | `john@example.com` |
| `{{ url }}` | user input | `http://example.com` |
| `{{ license }}` | user input / default `unlicensed` | `unlicensed` |
| `{{ cement.version }}` | runtime introspection via `cement.utils.version.get_version()` | `3.0.15` |

**Implication for executor:** these variables MUST remain literal `{{ var }}`
in the template files; do NOT pre-substitute. They are processed by
`cement/ext/ext_generate.py` at `cement generate project` time, not by
templating in this phase.

### Pattern: pdm-backend default packaging (no `[tool.pdm.build]` needed)

**Source:** RESEARCH.md "Anti-Patterns to Avoid" + Pitfall 2; verified
against `https://backend.pdm-project.org/build_config/`.
**Apply to:** template `pyproject.toml`.

pdm-backend's default behavior auto-includes non-Python files (e.g.,
`{{ label }}/templates/*.jinja2`) inside detected package directories. Adding
explicit `[tool.pdm.build] includes = [...]` *overrides* the defaults and
silently strips template package data. Omit the block.

### Pattern: Atomic-per-concern commit shape

**Source:** `.planning/phases/01-tooling-baseline-python-matrix/01-CONTEXT.md`
D-02; CONTEXT.md D-15.
**Apply to:** all phase commits.
