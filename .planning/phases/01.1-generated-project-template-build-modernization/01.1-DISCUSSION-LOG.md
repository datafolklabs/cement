## Build Backend Choice

| Option | Description | Selected |
|--------|-------------|----------|
| pdm-backend (Recommended) | Matches cement itself. Lean, supports dynamic version. MANIFEST.in goes away. | ✓ (initial) |
| hatchling | Most popular modern PEP 517 backend. Adds a tool cement doesn't use. | |
| setuptools (modern, declarative) | Stay on setuptools, drop setup.py, use [project] table. | |

**User's choice:** pdm-backend (initially)
**Notes:** Rationale was ecosystem consistency with cement.

---

## Mid-discussion Reversal

After the recap of the seven implied decisions (D-01 through D-09 of the
recap table), the user reverted the entire scope:

> "i want to revert - this is scope creep. Keep 'cement' as a build time
> dependency, keep the version-in-code stuff in line with cement (that is
> what users are use to). we can revisit this later"

This invalidated the four prior selections in their original
interpretation. The phase was rescoped to a strict-minimum patch.
