## Accomplishments

- **Phase acceptance gate green on full Python 3.10–3.14 matrix.** `make cli-smoke-test` exits 0; all 5 versions report `COMPLETED WITHOUT ERRORS`. Wheel built via pdm-backend at each version. `myapp --version` reports `Cement Framework 3.0.15` cleanly post-install.

**1. [Rule 1 - Bug] Test assertion for `setup.py` failed after deletion**

- **Found during:** Task 7 (acceptance gate), running `pdm run pytest --cov=cement tests`.
- **Issue:** `tests/cli/test_main.py::test_generate` asserted `os.path.exists(os.path.join(tmp.dir, 'setup.py'))`. Task 4 deleted setup.py from the template, so freshly-generated projects no longer contain that file. Test failed with `AssertionError: assert False`.
- **Fix:** Updated the assertion to check for `pyproject.toml` instead — that's the modern equivalent and the only metadata file remaining in the generated output. The test still verifies the same end-to-end behavior (template generator runs, scaffold is produced); the assertion now matches the new contract.
- **Files modified:** `tests/cli/test_main.py` (1 line change).
- **Verification:** Re-ran full test suite; 316 tests passed, 100% coverage maintained, ruff + mypy clean.
- **Committed in:** `555d0774` (`test(cli): assert pyproject.toml in generated project, not setup.py`).
- **Why Rule 1 (not Rule 4):** The regression was directly caused by an in-scope plan deletion. The fix is mechanical (one-line change matching the new file contract) — no architectural reshape. Falls cleanly within auto-fix scope.

**2. [Rule 3 - Blocking issue] Task 1 revert produced a smaller diff than D-13's net change required**

- **Found during:** Task 1 (PIP_FIND_LINKS revert), inspecting `git diff --staged` after `git revert --no-commit 3d60764c`.

It did NOT add the `docker exec ... pip install -r requirements.txt` line — that line existed pre-`3d60764c`. A pure revert of `3d60764c` would leave the requirements.txt line in place, but the plan's acceptance criteria require it gone (`! grep -q 'pip install -r requirements.txt'`) and the commit message body promises it's dropped (D-13).
- **Fix:** After staging the pure revert, manually edited the file to also remove the `pip install -r requirements.txt` line, then staged and committed both changes together with the manually-authored conventional-commit message. Net effect matches D-13 exactly.
- **Files modified:** `scripts/cli-smoke-test.sh` (additional line removal beyond pure revert).
- **Verification:** `bash -n scripts/cli-smoke-test.sh` exits 0; `! grep -q 'PIP_FIND_LINKS'` and `! grep -q 'pip install -r requirements.txt'` both satisfied; full smoke-test matrix green.
- **Committed in:** `63c65006` (`chore(ci): revert PIP_FIND_LINKS smoke-test workaround`).
- **Why Rule 3 (not Rule 4):** The plan's verbatim instruction in Task 1 step 3 explicitly listed line 52 (requirements.txt) as part of the expected revert diff, and the acceptance criteria mandated its removal. The discrepancy was between the plan's written description (which assumed `3d60764c` had added that line) and reality. ## Deferred Follow-ups

These were explicitly out-of-scope for Phase 01.1 per CONTEXT.md `<deferred>` and PLAN.md `<deferred_followups>`:

- **README.md modernization (DD-2):** the generated project's README still references `pip install -r requirements.txt` and `python setup.py install` in the Installation section. Lightweight follow-up; not an acceptance blocker.
- **Smoke-test package-data assertion (DD-3):** smoke test does not currently exercise template rendering, so a regression where `[tool.pdm.build]` accidentally gets added (silently stripping `*.jinja2` files) wouldn't be caught. Mitigation in 01.1 is "don't add `[tool.pdm.build]`" and a future quick task can add a smoke-test step that exercises e.g. `myapp command1 --foo bar`.
- **Sibling templates audit:** `script/`, `extension/`, `plugin/` templates not pip-installable as separate packages — likely no PEP 517 bug, but unrelated modernization debt may exist.
- **`todo-tutorial` template migration:** same setup.py-imports-self pattern likely lurks there. Verify and migrate in a follow-up phase.
- **Drop `cement` from `[project].dependencies`:** out of scope; cement IS the framework users signed up for when they ran `cement generate project`.
