## Summary

Phase 01.1 migrates `cement generate project`'s template from setuptools+setup.py
to pdm-backend. The diff is small, surgical, and matches the locked decisions
in CONTEXT.md / RESEARCH.md. All load-bearing technical claims verify
empirically:

- The literal `__version__ = "0.0.1a0"` line in `version.py` matches the
  pdm-backend default regex (verified by running the regex against the file
  — captured group `0.0.1a0`).
- `[project].readme = "README.md"` resolves — `README.md` exists in the
  template root.
- The `templates/` package-data directory exists at
  `{{ label }}/templates/` and pdm-backend's default packaging will include
  it without an explicit `[tool.pdm.build]` block (per RESEARCH.md
  Pitfall 2).
- `scripts/cli-smoke-test.sh` no longer contains `PIP_FIND_LINKS` or any
  `pip install -r requirements.txt` reference; the surviving install line
  `cd $tmp/myapp ; pip install .` is correct for the new toolchain.
- `Dockerfile` no longer references `requirements.txt` or `setup.py` — the
  collapsed `pip install --no-cache-dir .` is the only install step.

No critical or security issues found. One warning (license-form deprecation
posture, intentionally accepted per RESEARCH.md but worth flagging in the
artifact) and four info-level notes (consistency suggestions, none
blocking).

The changed assertion in `tests/cli/test_main.py` — `setup.py` →
`pyproject.toml` — is correct: the prior assertion was a smoke test that
the template generator produced a build-manifest file at the project root.
Under the new toolchain, `pyproject.toml` is the single source of truth and
the only manifest at the project root. The new assertion preserves the
original intent (existence check on the generator's primary artifact). It
does NOT regress coverage since the test was never asserting setup.py
content, only its existence.

## Warnings

### WR-01: `license = {text = "..."}` table form is PEP 639 deprecated

**File:** `cement/cli/templates/generate/project/pyproject.toml:12`
**Issue:** Per PEP 639 (and RESEARCH.md "Alternatives Considered" → license
row + Pitfall 4), the `license = {text = "..."}` table form is formally
deprecated in favor of an SPDX expression string (e.g.,
`license = "MIT"`). pdm-backend 2.4.8 still accepts the table form (with
backward-compat warnings, no errors) and setuptools v77+ has begun warning
on it. The phase context explicitly notes this is **intentionally
accepted for now** as a CONTEXT.md-locked decision, with PEP 639 cleanup
flagged for follow-up. This finding documents the known posture — not a
correctness defect — so downstream consumers of the template don't get
surprised when their build emits a deprecation warning.

**Fix:** No action required this phase. Future follow-up: when
pdm-backend or pip elevates this from warning to error (target Feb 2026
per PEP 639 timeline), migrate the template to:
```toml
license = "{{ license }}"
```
…with a corresponding template-engine constraint that `{{ license }}`
must be a valid SPDX identifier (currently it is a freeform string —
template would need validation upstream in `ext_generate.py`'s
`license` prompt).

## Info

### IN-01: `requires-python = ">=3.10"` is hard-coded, not Jinja-templated

**File:** `cement/cli/templates/generate/project/pyproject.toml:14`
**Issue:** The minimum Python floor is hard-coded to `>=3.10`. Cement's
own minimum just moved to 3.10 in this milestone, so the values match
today. However, when cement's floor next moves (e.g., dropping 3.10 when
upstream EOLs it per CLAUDE.md "Policy — Python support"), the generated
template will silently lag unless a maintainer remembers to bump this
literal too. Cement's own version is already templated as
`{{ cement.version }}` on line 17 — there is precedent for sourcing
runtime values from the generator context.

**Fix:** Consider exposing cement's own `requires-python` floor through
the template context (e.g., `{{ cement.requires_python }}`) so the
generated project's floor tracks cement's automatically. Out of scope for
this phase per CONTEXT.md `<deferred>`-style scoping; file as a
follow-up. If kept hard-coded, consider a comment noting it must be
hand-bumped alongside cement's own floor.

### IN-02: Trailing newline missing on `pyproject.toml`

**File:** `cement/cli/templates/generate/project/pyproject.toml:37`
**Issue:** Last line is `]` without a trailing newline (the file ends
exactly at the closing bracket of `[dependency-groups].dev`). POSIX text
files conventionally end with a newline. Many tools (git diff, cat, some
editors) flag this. Not a functional defect — pdm-backend parses the file
identically with or without the trailing newline.

**Fix:** Add a single trailing newline after line 37's `]`. Ensures
clean diffs and consistent rendering with the project's other files
(checked: `Makefile`, `version.py`, `Dockerfile` all end with a newline).

### IN-03: `Makefile` `test` target hard-codes `coverage-report` HTML path with no `clean` hook

**File:** `cement/cli/templates/generate/project/Makefile:13-19`
**Issue:** The `test` target writes HTML coverage to `coverage-report/`
but the `clean` target (line 3-4) only deletes `*.py[co]` files. Repeated
test runs will accumulate stale `coverage-report/` artifacts that are not
cleaned by `make clean`. This mirrors the pre-migration behavior (was
already this way under setuptools), so it is not a regression — but the
phase is touching the file anyway.

**Fix:** Optional consistency improvement (out of phase scope per
CONTEXT.md `<deferred>`):
```make
clean:
	find . -name '*.py[co]' -delete
	rm -rf coverage-report/ .coverage dist/ build/
```
Defer to follow-up; do not bundle into this phase.

### IN-04: `Dockerfile` base image `python:3.10-alpine` lags cement's matrix high-end

**File:** `cement/cli/templates/generate/project/Dockerfile:1`
**Issue:** Generated projects pin to `python:3.10-alpine` (the floor),
not the latest supported version (`3.14`). For a starter template, the
floor is a defensible choice (broadest compatibility), and this is
unchanged from the pre-migration template — so no regression. Calling it
out as an info note since the file is being touched in this phase and
the value is semantically adjacent to `requires-python` (IN-01).

**Fix:** No action required this phase. Possible follow-up: align
with `requires-python` policy — either both stay at the floor (current),
or both track the latest supported (mirror cement's own Dockerfile
posture, if any). Coordinate with IN-01 if templating
`requires-python`.
