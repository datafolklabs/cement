<objective>
Fix `scripts/cli-smoke-test.sh` to close two phase-1 gaps the CI job just
exposed:

1. The default `PYTHON_VERSIONS` still lists `3.9`, but `pyproject.toml`
   declares `requires-python = ">=3.10"`. pip refuses to install the
   cement sdist on 3.9, so the very first matrix entry blows up the job.
2. The generated-project install path uses `python setup.py install` plus
   an explicit `pip install setuptools` — both deprecated since
   setuptools 58 (2021), removed in setuptools 80 (Feb 2025), and broken
   on Python 3.12+ where stdlib `distutils` is gone. Even after fixing
   issue (1), this would fail on 3.12/3.13/3.14.

**Purpose:** Phase 1 dropped Python 3.9 from "every surface" — except this
script. The cli-smoke-test CI job (`.github/workflows/build_and_test.yml`
lines 85-98) is currently red on the Phase-1 PR branch. This is a
narrow, mechanical gap-closure: align the script with the matrix already
codified in `.github/workflows/build_and_test.yml:68` and replace the
deprecated install pattern with a PEP 517 invocation.

**Output:** A single conventional commit modifying only
`scripts/cli-smoke-test.sh`. No template changes, no CI workflow
changes, no broader cleanup.

Rationale (do NOT add this as a comment in the script — keep the script
free of new commentary; the rationale lives in the commit body and the
plan SUMMARY):
- `pip install .` triggers PEP 517 build via setuptools build-backend
  fallback for legacy `setup.py` projects.
- `pip install setuptools` is unnecessary: pip provisions setuptools
  in the isolated build environment automatically.
- `pip install -r requirements.txt` is also redundant — `pip install .`
  installs the project plus its declared deps. Dropping it removes a
  second cement-from-PyPI install of the just-built sdist's deps.
  (The generated `requirements.txt` lists `cement==X.Y` plus extras,
  which `setup.py`/`setup.cfg`-driven `pip install .` does NOT
  replicate today, but the smoke test only needs the entry-point
  binary `myapp --version` to import cement and report version. The
  cement install at line 22 already provides cement; the generated
  app's setup.py declares no `install_requires`. So dropping
  `requirements.txt` is intentional and matches the modern pattern.)
