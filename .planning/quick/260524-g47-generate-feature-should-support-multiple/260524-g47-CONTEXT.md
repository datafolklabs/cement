# Quick Task 260524-g47: Generate feature should support multiple valid Inputs to enable prompt - GH Issue 779 - Context

**Gathered:** 2026-05-24
**Status:** Ready for planning (post-pivot — see "Pivot" below)

<pivot>
## Pivot (2026-05-24, post code-review reflection)

First-attempt design hand-rolled regex validation, case normalization,
and a manual `assert val in valid_values` dispatch guard on top of a
bespoke `FeaturePrompt` subclass. A walk through `cement/utils/shell.py`
revealed `shell.Prompt` already provides exactly what this feature
needs out of the box:

- `Meta.options: list[str]` — whitelist input must match (line 514-531)
- `Meta.numbered: bool` — display `1: X / 2: Y / 3: Z` and return the
  matched option **string** (not the number) via `options[int(input)-1]`
  at line 517
- `Meta.case_insensitive: bool` — covers the manual `case:` field
- `Meta.max_attempts: int` + `max_attempts_exception: bool` — built-in
  retry/abort semantics, raises `FrameworkError` on exhaustion
- `Meta.default: str` — fallback when user hits Enter

**Decision:** Build the new select-mode on top of `shell.Prompt` rather
than parallel to it. Drop every concept that duplicates existing
framework machinery (regex `validate`, `case` field on features, manual
unmatched-input assertion). Rename `selected:` → `options:` to match the
`shell.Prompt.Meta.options` vocabulary. Add an optional per-option
`prompt:` label so the displayed text can be more descriptive than the
runtime `value:` (e.g. `value: flask` displays as `Flask Web Framework`).

Artifacts from the first attempt — RESEARCH.md, REVIEW.md, SUMMARY.md,
VERIFICATION.md — describe the path-not-taken design and remain on
disk as historical record. PLAN.md gets rewritten against the simpler
shape. CONTEXT.md (this file) is rewritten in place to capture the
new locked decisions.

The behavior the user sees is unchanged in spirit (multi-valued feature
prompts that dispatch into N config branches); the SURFACE AREA of
this feature inside `cement/ext/ext_generate.py` shrinks by ~60%.
</pivot>

<domain>
## Task Boundary

Extend `cement/ext/ext_generate.py` (the `.generate.yml` template
engine, introduced by PR #768) so a single feature can support more
than two answers. Today the feature prompt is boolean-like (`Enable
Feature: name [Y/n]`) and dispatches to one of two static blocks:
`enabled` or `disabled`. GH #779 adds a multi-valued mode where the
feature prompt presents the user with a numbered list of choices and
dispatches into one of N `options` branches keyed by the chosen value.

**In scope:**

- New optional schema keys on a feature: `prompt_mode` (defaults to
  `'boolean'`, explicit value `'select'` opts in), `default` (the value
  dispatched in `--defaults` mode AND the Prompt default), `options`
  (the per-branch list).
- An `options` branch may declare any of: `ignore`, `exclude`,
  `variables` (same vocabulary as today's `enabled`/`disabled` blocks)
  plus an optional `prompt:` label.
- Silent (no-prompt) variables under `options` branches via `prompt:
  false` sentinel on a variable definition (`prompt: false` also works
  on top-level variables).
- Fail-fast config validation for the new schema (raise `ValueError`
  at config-validation time on every invalid shape).
- Tests covering the new dispatch path AND regression coverage that
  the existing boolean path is byte-identical.
- Extend `demo/generate-features/` additively with one new
  `prompt_mode: select` feature + README commentary walking through
  the new schema.

**Out of scope:**

- GitBook / RST documentation refresh — captured in pending todo
  `2026-05-09-document-optional-features-in-gitbook-post-3-0-16`
  (blocked on 3.0.16 release cut).
- CHANGELOG entry beyond the standard `[ext.generate]` Features line
  in the active `## 3.0.15 - DEVELOPMENT` section.
- Restructuring or renaming any existing schema vocabulary
  (`enabled`/`disabled`/`requires`/`variables`/`exclude`/`ignore` keep
  their exact current semantics).
- Reinventing what `shell.Prompt` already does. If `shell.Prompt`
  supports a needed behavior, USE IT — do not reimplement.

</domain>

<decisions>
## Implementation Decisions (POST-PIVOT — these supersede the first attempt)

### A. Schema detection & integration

- **Decision:** Multi-valued mode is gated by an explicit `prompt_mode:
  select` key. The default is `prompt_mode: 'boolean'` (legacy
  semantics). Absence of the key is treated as `'boolean'` — the
  explicit form is offered for clarity but never required for
  backward-compat.
- **Validator:** Whitelist check — `prompt_mode` must be one of
  `{'boolean', 'select'}`. Anything else raises `ValueError` up front.
- **Boolean path:** When `mode == 'boolean'`, the new validator
  short-circuits with `continue`; the existing `_resolve`
  `else`-branch handles boolean features exactly as it does today
  (byte-identical behavior; the existing `BoolFeaturePrompt` /
  `bp`/`bval` symbols remain unchanged).
- **Select path:** When `mode == 'select'`, the new validator runs
  the option-set checks below, then `_resolve` dispatches via
  `shell.Prompt`.

### B. `options` data shape

- **Decision:** `options:` is a YAML **list of dicts** each with a
  required `value:` key. The `value:` is what flows downstream as the
  feature's resolved state AND is what `default:` must match.

  ```yaml
  options:
    - value: "None"
      prompt: "None — no web framework"   # optional display label
      ignore: [...]
    - value: "flask"
      prompt: "Flask Web Framework"
      variables: [...]
    - value: "fastapi"
      prompt: "FastAPI Web Framework"
      variables: [...]
  ```

- **Rationale:** Mirrors the `variables:` and `features:` shapes —
  list-of-dicts with a leading identifier key — so the YAML reads
  consistent with the rest of the file. Ordered iteration via the
  list structure (the numbered Prompt display preserves YAML order).
  `str()` coercion on every `value:` at validation time so YAML
  decoded `1` (int) and `"1"` (str) compare uniformly.
- **Renamed from `selected:` → `options:`** — vocabulary matches
  `shell.Prompt.Meta.options` and reads as "the list of available
  options" rather than "the result of selection."

### C. Per-option label via optional `prompt:` key

- **Decision:** Each option may declare an optional `prompt:` field
  whose value is the human-readable label shown in the numbered list.
  If omitted, the label falls back to `str(value)`.

  ```yaml
  - value: "flask"
    prompt: "Flask Web Framework"    # → numbered list shows: "2: Flask Web Framework"
  ```

- **Rationale:** The runtime `value:` is often short and
  machine-friendly (`flask`, `fastapi`, `none`). The user-facing
  picker label is often longer and human-friendly. Keeping them
  separable lets template authors pick simple `{{ web_framework }}`
  substitution targets without compromising prompt UX.
- **Implementation:** Build `display_labels = [opt.get('prompt') or
  str(opt['value']) for opt in options]`, pass `options=display_labels`
  +  `numbered=True` to `shell.Prompt`, then map the returned label
  back to the original `value:` via positional zip.
- **Validator:** Display labels must be unique within a feature
  (`ValueError` if two options collide on the same effective label).
  Otherwise the numbered list is ambiguous.

### D. `shell.Prompt` integration (no parallel infrastructure)

- **Decision:** The select-mode prompt is a thin `shell.Prompt`
  subclass that sets `Meta.options`, `Meta.numbered=True`,
  `Meta.case_insensitive=True`, `Meta.default=str(default)`,
  `Meta.text=feature.prompt`. The `shell.Prompt.prompt()` return
  value is the resolved label string, which we then map back to the
  underlying `value:` via positional lookup.

  ```python
  class SelectPrompt(shell.Prompt):
      class Meta:
          text = feature.get('prompt') or f"Select Feature: {name}"
          options = display_labels
          numbered = True
          case_insensitive = True
          default = str(feature['default'])
  ```

- **What we drop vs the first attempt:**
  - `validate:` regex field on features — `shell.Prompt.options`
    enforces the whitelist; no regex needed.
  - `case:` field on features — `shell.Prompt.case_insensitive=True`
    covers it. (Variables keep their own `case:` field; that's
    unrelated.)
  - Manual `assert val in valid_values` after the prompt returns —
    `shell.Prompt` never returns a value outside its `options` set
    (it re-prompts or raises `FrameworkError`).
  - The manual `p.prompt() or str(default)` empty-input dance —
    `shell.Prompt.Meta.default` handles it at the framework layer.
  - Custom `FeaturePrompt` class with bespoke `default_hint` text
    rendering — Prompt handles all of that.
- **Validator changes:** Drops the "validate is a string" check
  (no `validate` field). Drops the case-vocabulary check on features
  (`case:` field gone from features). Adds the
  unique-display-labels check.

### E. Silent (no-prompt) variables under `options` branches

- **Decision:** UNCHANGED from first attempt — `prompt: false`
  sentinel on a variable definition makes it silent (value lands
  from `default` without rendering an interactive prompt). The
  variable's `case` and `validate` (if declared) still apply to
  `default`. `default` is coerced to `str()` because the prompted
  path always returns `str` from `p.prompt()` — the silent path
  must match the invariant (else `False.upper()` etc. crashes).
- **Top-level silent variables** also supported (`prompt: false` on
  a variable in the file-level `variables:` block) — same semantics.
- **Use case shift:** With per-option `prompt:` labels, the
  `framework_name` silent-variable pattern from the first-attempt
  demo is no longer needed — the feature `value:` itself
  (`web_framework=flask`) is what templates substitute. Silent
  variables remain useful for independent per-branch metadata
  (e.g. `framework_version=0.115` under the `fastapi` branch).

### F. Strictness & fallback rules

- **Schema collision:** `prompt_mode: select` AND
  `enabled:`/`disabled:` blocks on the same feature →
  `ValueError` at validation time. (`enabled`/`disabled` is the
  legacy mode-`boolean` vocabulary; mixing is a template-author
  bug.)
- **Missing default:** `prompt_mode: select` requires `default:`
  AND `default:` must equal one of the `options[].value` entries
  (after `str()` coerce). Both are `ValueError` at validation.
- **Unmatched user input:** Not possible at runtime —
  `shell.Prompt` enforces the whitelist. No internal assert
  needed in our code.
- **`--defaults` mode:** Skip the prompt entirely; dispatch via
  `feature_states[name] = str(default)`. Same as today's
  `--defaults` behavior for boolean features.
- **Rationale:** Project-level D-13 strict-minimum favors fail-fast
  on misconfiguration and reuse-don't-reinvent on framework
  primitives. The existing `requires` validator (lines 43-48)
  establishes the `raise ValueError` precedent for cross-config
  consistency errors; we mirror that.

### Claude's Discretion

- Test layout: follow the existing `tests/ext/test_ext_generate.py`
  structure and naming conventions established by PR #768.
- Validation timing: cross-config validation runs **up front** in
  `_process_features` before any prompt is rendered, matching the
  existing pattern at lines 35-48.
- Error message style: match the existing concise format from
  `_process_features` — `"Feature 'X' ... reason"`. No deep stack
  traces; no f-string templating beyond what's already present.
- Backward compatibility: the existing `webapp` demo template and
  every `tests/data/templates/generate/test1..test15` fixture
  remain valid without modification. Only files added in this task
  exercise the new `prompt_mode: select` schema.

</decisions>

<specifics>
## Specific Ideas

### Concrete schema shape (canonical example)

```yaml
features:
  # legacy boolean (unchanged) — explicit form
  - name: docker
    prompt_mode: boolean
    default: true
    enabled:
      variables: [...]
    disabled:
      ignore: [...]

  # new select mode
  - name: web_framework
    prompt_mode: select
    prompt: "Web Framework"
    default: "none"
    options:
      - value: "none"
        prompt: "None — no web framework"
        ignore:
          - '.*requirements\.txt.*'
          - '.*wsgi\.py.*'
      - value: "flask"
        prompt: "Flask Web Framework"
        variables:
          - name: framework_version
            prompt: false       # silent — independent per-branch metadata
            default: "3.0"
        ignore:
          - '.*fastapi.*'
      - value: "fastapi"
        prompt: "FastAPI Web Framework"
        variables:
          - name: framework_version
            prompt: false
            default: "0.115"
        ignore:
          - '.*flask.*'
```

### Rendered prompt UX

```
Web Framework

1: None — no web framework
2: Flask Web Framework
3: FastAPI Web Framework

Enter the number for your selection: 2
```

User picks `2`, `shell.Prompt` returns `"Flask Web Framework"`, our
code maps that label back to `value: "flask"` via positional zip,
sets `feature_states['web_framework'] = 'flask'`, the merge step
walks the `selected[1]` branch and applies its `variables` /
`ignore`.
