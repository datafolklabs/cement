# Quick Task 260524-g47: Summary

**Description:** Extend `cement/ext/ext_generate.py` to support
multi-valued feature prompts (`prompt_mode: select`) per GH Issue
#779.

**Final shape:** 4 commits on `feat/generate-feature-prompt-multiple-values`.

## What landed

### Schema (single `feat:` commit)

- New optional feature keys: `prompt_mode` (defaults to `'boolean'`),
  `default` (required for select-mode), `options` (the per-branch
  list).
- `prompt_mode: 'boolean'` is the explicit default name; omission
  remains valid and exactly equivalent (backward-compat absolute).
- `prompt_mode: 'select'` opts into the multi-valued numbered-list
  flow built directly on `cement.utils.shell.Prompt` — no parallel
  prompt/validation infrastructure.
- `options:` is a list of dicts with required `value:` and optional
  `prompt:` label (falls back to `str(value)`). Labels are
  displayed in `shell.Prompt`'s numbered list; `value:` is what
  flows downstream as the resolved feature state.
- Each `options` branch may declare `ignore` / `exclude` /
  `variables` (same vocabulary as the legacy `enabled:` / `disabled:`
  blocks).
- Silent (no-prompt) variables under `options` branches via
  `prompt: false` sentinel on a variable definition. The variable's
  `case` and `validate` still apply to `default`. `str()` coerce so
  YAML-decoded non-string defaults (bool, int) don't crash the
  downstream case/validate operations. Silent variables work at the
  top level too.
- Fail-fast validation up front: `ValueError` on schema collision
  (`prompt_mode: select` + `enabled`/`disabled`), unknown
  `prompt_mode`, missing `value:` in option, empty `options`,
  missing `default`, default not in option values, duplicate option
  labels.
- The `or []` null-safety idiom (per commit 6895aa52) is
  generalized to per-block keys on both the legacy and select-mode
  merge paths.

### Demo (single `docs(demo):` commit)

`demo/generate-features/` extended additively with a new
`web_framework` select-mode feature (None / Flask / FastAPI).
Existing `docker` / `docker_compose` features stay byte-identical
aside from gaining an explicit `prompt_mode: boolean`. Two new
template files (`requirements.txt`, `wsgi.py`) added so the
`options` branches' `ignore` patterns have real files to suppress.
README extended with a "Multi-Valued Feature (prompt_mode: select)"
section walking through the schema, the case-insensitive numbered
picker, silent variables, the `--defaults` headless path, and the
`requires:` cascade interaction.

## Pivot from first attempt

The first attempt hand-rolled regex `validate:`, a feature-level
`case:` field, a bespoke `FeaturePrompt(auto=False)` subclass,
manual `p.prompt() or str(default)` empty-input handling, and an
`assert val in valid_values` runtime guard. A code-review pivot
discovered `cement.utils.shell.Prompt` already provides all of these
behaviors via `Meta.options` + `Meta.numbered=True` +
`Meta.case_insensitive=True` + `Meta.default` + the built-in
`max_attempts` retry loop. Rebuilding on top of `shell.Prompt`
dropped the `validate`/`case` feature fields entirely and reduced
`ext_generate.py` coverage stmt count from 205 (first attempt) to
202 (this rebuild). See `260524-g47-CONTEXT.md` §Pivot for the full
reasoning. RESEARCH.md / REVIEW.md / VERIFICATION.md from the first
attempt remain on disk as historical record of the path-not-taken
design.

## Adjudications honored

1. **Pitfall A (null-safety scope)** — IN-SCOPE; generalized
   `block.get(...) or []` to per-block keys on BOTH legacy and
   select-mode merge paths. Tests `test23` (legacy) and `test24`
   (select) exercise the null-coalesce on each path.
2. **Demo `.dockerignore` gap** — OUT-OF-SCOPE; pre-existing demo
   inconsistency, not touched.
