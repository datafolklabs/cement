## Warnings

### WR-01: Select-mode feature with no `default:` + `--defaults` silently dispatches to NO branch

**File:** `cement/ext/ext_generate.py:114-116, 186-193`
**Severity:** WARNING (correctness gap, not a crash)

**Issue:** Schema validation at lines 53-86 does NOT require a select-mode
feature to declare `default:`. The validation only enforces "if default
is set, it must be in selected values" (line 81). Then in the `_resolve`
closure:

```python
default = feature.get('default')   # None if not set
if self.app.pargs.defaults:
    feature_states[name] = str(default)   # → literal string "None"
```

The state becomes the literal string `"None"`. The merge step at lines
189-193 then walks `selected` looking for `str(branch['value']) == "None"`,
finds nothing, leaves `block = {}`, and silently produces no
`variables`/`exclude`/`ignore` extension. No error, no warning. The user
gets a partial, broken generation — only the non-select-feature outputs
land.

Verified by inspection: a fixture with `prompt_mode: select`, no
`default:`, and `selected: [{value: "1", ignore: ['.*x.*']}]` would pass
all current validation, run under `--defaults`, and skip the `ignore`
pattern entirely.

This contradicts the CONTEXT.md "fail-fast" discipline and Decision D
("the feature's `default` is dispatched directly without prompting,
through the same `selected` lookup as interactive mode") — Decision D
assumes a default exists.

**Fix:** Add a validation step in the new third loop after line 86:

```python
if feature.get('default') is None:
    raise ValueError(
        f"Feature '{feature['name']}' uses prompt_mode: select "
        f"but has no 'default' (required for --defaults dispatch "
        f"and as the prompt default)")
```

OR, less strictly, raise at runtime in `_resolve` when `--defaults` is
set but `default` is None. Either gets fail-fast back. The validation
step is preferable (matches the existing discipline). One new test
fixture + test required.

### WR-02: Interactive select prompt with `default=None` falls back to literal string `"None"`

**File:** `cement/ext/ext_generate.py:129`
**Severity:** WARNING (UX/robustness — pragma-skipped from coverage)

**Issue:** `val: str = p.prompt() or str(default)` — when `default` is
None (per WR-01, allowed today) and the user hits Enter without typing
anything, `p.prompt()` returns `''` (or None), `or str(default)` yields
`str(None) == "None"`. The downstream `assert val in valid_values` then
fails with `"must be one of: [...]" — 'None' not present`. The user gets
an `AssertionError` message that mentions "None" without explaining the
template-author misconfiguration (missing `default:`).

This line is in the `# pragma: nocover` block on the interactive prompt
path, so coverage doesn't catch it. But the symptom — `"None"` as a
fallback value — is a code smell.

**Fix:** Tighten WR-01 (require `default:` at config validation), which
eliminates this code path's reachability with a meaningful failure mode.
Alternatively, change the fallback to a clearer assertion before the
prompt is rendered. Lower priority than WR-01; fixing WR-01 makes this
unreachable.

### WR-03: `case` normalization on silent variables crashes if YAML default is non-string

**File:** `cement/ext/ext_generate.py:282-283`
**Severity:** WARNING (latent crash on edge-case template-author input)

**Issue:** With the new `prompt: false` short-circuit (line 263), `val =
var['default']` adopts the YAML-decoded type of `default` — which may be
`bool`, `int`, `float`, `list`, etc., not necessarily `str`. The
subsequent `if var['case'] in ['lower', 'upper', 'title']: val =
getattr(val, var['case'])()` then attempts `False.upper()`,
`42.lower()`, etc. — all raise `AttributeError`.

Example fixture that crashes (no test covers it):

```yaml
variables:
  - name: enabled
    prompt: false
    default: false      # YAML → Python bool False
    case: upper
```

For prompted variables (legacy path), `val = p.prompt()` always returned
`str`, so `case` was safe. The new silent-variable path widens the
input domain without widening the `case` handling. CONTEXT.md does state
"The variable's `case` and `validate` still apply to the default value"
— but that's contingent on the default being a string.

**Fix:** Either (a) coerce `val = str(var['default'])` after the silent-
variable assignment at line 263, or (b) only apply `case`/`validate`
when `isinstance(val, str)`, or (c) reject non-string `default` when
`case` or `validate` is set on a `prompt: false` variable at config-
validation time. Option (a) is the smallest, most surprise-free change
— a silent variable's default rendered into Jinja2 is going to be
stringified by Jinja anyway. Add one test covering `default: false` +
`case: upper`.

### WR-04: `requires`-cascade-with-select silently drops the select feature's `selected` branches

**File:** `cement/ext/ext_generate.py:103-109, 181-185`
**Severity:** WARNING (surprising semantics — codified by test but not documented)

**Issue:** When a `prompt_mode: select` feature has `requires: [X]` and
`X` resolves to False, the cascade sets `feature_states[name] = False`
(bool, line 108). Then the merge step takes the `isinstance(state, bool)`
branch, computes `block_key = 'disabled'`, and reads `feature.get(
'disabled') or {}` — but select-mode features are **forbidden** from
declaring `disabled:` (validated at line 61-65). Result: `block = {}`,
no variables/exclude/ignore extended.

This is the behavior test25 confirms. It is, however:

1. Undocumented in the demo README's "Multi-Valued Feature" section.
2. Undocumented in the CONTEXT.md "Strictness & fallback rules" section.
3. Surprising — a template author writes `selected: [{value: "1",
   ignore: ['.*x.*']}]` under a select feature, adds `requires:
   [bool_prereq]`, and silently their `ignore` pattern stops applying
   when `bool_prereq` is false.

The select feature's intended branches are still in the YAML; the
cascade silently discards them via the bool-path-with-no-disabled. This
is documented in test25's inline comment but not surfaced to users.

**Fix (lowest cost):** Add a one-paragraph note to the demo README's
"Multi-Valued Feature (prompt_mode: select)" section explaining the
requires-cascade semantics: "If a select-mode feature's `requires:`
prerequisites resolve to disabled, the feature is treated as disabled
and none of its `selected:` branches are dispatched."

**Fix (higher cost, optional):** In the `_resolve` cascade, when
returning `False` for a select-mode feature, emit a more specific
`self.app.log.warning(...)` than the generic `"Feature '{name}'
disabled (requires: {req})"` — e.g., note that no selected branch will
be applied. The current warning already fires but doesn't differentiate
bool vs select semantics.
